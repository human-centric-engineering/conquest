# F15.1 — Experiences foundation

> **Everything still open across P15 is consolidated in [`f15-followups.md`](./f15-followups.md).**

> ConQuest treats a questionnaire as the unit of everything: one authored version, one respondent
> session, one report. That is the right primitive, but it caps what the product can offer. Two
> journeys need **composition** rather than a bigger questionnaire — routing a respondent to the
> right follow-up based on what they said, and running one short questionnaire across a whole room
> at once. **Experiences** is the layer that composes existing questionnaires into those journeys.

The questionnaire stays the first-class citizen. Experiences sequence questionnaires, carry context
between them, and synthesise across them; nothing about authoring, the turn orchestrator, answer
slots, or respondent reports changes.

## The two kinds

**`agentic_switcher`** — an opening questionnaire, then an AI decision: conclude with a report, or
continue into a follow-up selected from a candidate set based on what was learnt.

The routing mechanism is deliberately **general-purpose**. Lead qualification is one application of
it, but nothing in the model, the naming, or the UI assumes a sales or marketing context. Equally
valid: triage into a specialist assessment, escalating depth on a topic the respondent cares about,
branching by role or domain, or a diagnostic that selects its own follow-up. This is why the step
kind is `branch` rather than `upsell`, the terminal decision is `conclude` rather than
`offer_report`, and the admin copy talks about "follow-ups" rather than "conversion".

**`facilitated_meeting`** — the same short questionnaire run by many participants simultaneously,
synthesised per breakout so a facilitator can see where a team agrees and where it does not.
(Runtime lands in F15.5; F15.1 ships the authoring shape.)

## Confirmed decisions

- **Experience sits ABOVE Rounds; it does not replace them.** A round remains the time-boxed
  delivery of ONE questionnaire to a cohort. An Experience may reuse a cohort as its roster
  (`AppExperience.cohortId`) and a round's access window per step (`AppExperienceStep.roundId`).
  No existing round or cohort model changed, and `AppQuestionnaireRoundItem` is untouched.

- **Code lives under `lib/app/questionnaire/experiences/**`, not a new `lib/app/experiences/**`
  root.** Precedent: `AppCohort` and `AppQuestionnaireRound` are also conceptually above
  questionnaires and live at `lib/app/questionnaire/rounds/`. A separate root would import ~25
  modules from `lib/app/questionnaire/**` and export nothing back — a boundary nobody enforces,
  with real cycle risk once the report enqueue path needs to ask "is this session part of an
  experience?". Schema does get its own file (`prisma/schema/app-experience.prisma`) because
  `app-questionnaire.prisma` is already ~1780 lines.

- **Continuity is configurable, and `stitched` is a presentation flag only.**
  `EXPERIENCE_CONTINUITY_MODES = ['linked', 'stitched', 'merged']`. `linked` and `stitched` share
  an **identical** persistence shape — one `AppQuestionnaireSession` per leg either way — so an
  experience can be switched between them at any time without migrating a row. That invariant is
  what keeps the one-continuous-chat feature cheap (F15.3). If `stitched` ever appears to need its
  own tables, the requirement is wrong, not the design. `merged` (F15.6) is the only mode that
  genuinely changes persistence, which is why it is isolated last and may never ship.

- **Step target pointers are unmodelled (UG-1).** `questionnaireId`, `versionId`, `roundId` and
  `cohortId` are plain Strings with no `@relation`. A real FK would let archiving or deleting a
  questionnaire cascade away an experience's authored structure and, through it, its run history.
  The consequence is that a pointer can dangle, so the read seam resolves titles in a batch and
  renders a missing target as "may have been deleted" rather than throwing. Asserted by
  `tests/unit/prisma/app-experience-schema.test.ts`.

- **`demoClientId` and `experienceId` ARE modelled relations, both cascading.** They are
  app-internal config edges (an experience is meaningless without its client; a step is meaningless
  without its experience), not identity↔answer edges — the same split `AppCohort` already makes.

- **Settings are a lazily-defaulted Json blob, not columns.** The facilitated-meeting kind is
  expected to accumulate many per-experience variants (facilitator visibility, insight cadence,
  synthesis instructions, respondent exposure). Each should be a settings key rather than a
  migration. Read through `narrowExperienceSettings`, never destructured raw.

- **`conclude` is the default routing fallback.** When the selector cannot be trusted, finishing
  with what was gathered is honest; routing someone into a long follow-up on a coin-flip is not.

- **Readiness is advisory, not enforced.** `experienceBlockers()` returns author-facing sentences
  and the editor shows them continuously, but nothing is blocked at the schema or the API. An
  author reorders and retypes mid-edit; a constraint that fires halfway through authoring is an
  obstacle, not a guardrail. (The one exception: deleting a non-draft experience is refused with a
  409, because the cascade would reach respondent history.)

- **Reorder takes the COMPLETE ordered id list, not a delta.** The handler rejects any list that is
  not exactly the experience's current step set, so a stale page 409s instead of writing an order
  derived from a set that no longer exists. A delta would let two concurrent drags interleave into
  an order neither author chose.

- **Step keys are scoped per experience and de-duplicated by suffix.** Two steps legitimately share
  a title in a long journey, so `deriveStepKey` appends `-2`, `-3`, … rather than rejecting. The
  derivation is racy by design: `@@unique([experienceId, key])` is the real arbiter and the route
  maps its P2002 to a 409.

## What shipped

**Schema** (`prisma/schema/app-experience.prisma`, new file; migration
`20260719222812_add_app_experience`):

- `AppExperience` — identity, `kind`, `status`, `continuityMode`, routing policy
  (`routingFallback`, `minRoutingConfidence`, `routingInstructions`), run-level `costBudgetUsd`,
  `accessMode` (reusing the `ACCESS_MODES` tuple), `publicRef`, `cohortId`, `settings` Json.
- `AppExperienceStep` — `key` (`@@unique([experienceId, key])`), `kind`, unmodelled target
  pointers, `title` / `purpose` / `selectionCriteria`, `ordinal`.
- `AppDemoClient.experiences` back-relation.

**Extended closed enums** (`lib/app/questionnaire/ai-run/types.ts`): `APP_AI_RUN_SUBJECTS` +=
`experience_run`; `APP_AI_RUN_KINDS` += `experience_routing`, `experience_handoff`,
`experience_report`. All three kinds added now rather than one per phase — extending a closed tuple
is a code-only change, and doing it once avoids three rounds of churn.

**Lib** (`lib/app/questionnaire/experiences/`): `types.ts` (every vocabulary tuple, label maps,
field bounds, `ExperienceSettingsShape`, `RoutingDecision`, `slugifyStepKey`), `settings.ts`
(`narrowExperienceSettings`), `schemas.ts` (Zod for every route), `views.ts` (Prisma-free view
models + `routableSteps` / `entryStep` / `experienceBlockers`), `workspace-nav.ts` (tab registry
built on the questionnaire workspace's `WorkspaceTab` interface — imported, not redeclared).

**API** (`app/api/v1/app/experiences/`, all `withAdminAuth`, all audited):
`GET|POST /`, `GET|PATCH|DELETE /:id`, `GET|POST /:id/steps`, `PATCH|DELETE /:id/steps/:stepId`,
`PATCH /:id/steps/reorder`. Read seam in `_lib/read.ts` (batched target resolution — no N+1),
key/ordinal helpers in `_lib/steps.ts`.

**Admin UI**: list page with stat tiles, search and kind/status filters; create dialog; workspace
shell (header + sub-nav) with Overview / Steps / Settings tabs; journey editor with add / edit /
reorder / delete; settings panel that PATCHes only dirty fields. Nav item registered in
`lib/app/admin-nav.ts` directly beneath Questionnaires.

**Tests**: 54 unit tests across settings narrowing (including the NaN/Infinity fallback and the
k-anonymity floor guard), view mapping and blockers, vocabulary exhaustiveness, slug edge cases,
workspace nav, and the schema shape (UG-1 FK posture, cascade DDL, index presence, and a guard that
the phantom pgvector `DROP INDEX` statements were stripped from the migration).

## Design notes

**The migration needed hand-stripping.** Prisma's diff engine cannot see the raw-SQL pgvector and
tsvector indexes, so the autogenerated migration proposed dropping all five plus a generated
default on `ai_knowledge_chunk`. Generated with `--create-only`, stripped, applied with
`migrate deploy`, and the five indexes verified present afterwards. The schema test now guards this
for every future Experience migration.

**`z.coerce.number()` breaks `zodResolver`.** Coercion widens a schema's _input_ type to `unknown`,
which no longer matches react-hook-form's form-values generic. The settings panel uses plain
`z.number()` with `register(..., { valueAsNumber: true })` instead.

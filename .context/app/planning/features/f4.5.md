# F4.5 — Completion logic

> Committable tracker for **F4.5**, the close primitive of P4 (Conversational engine,
> non-streaming). Decides when the agent may **offer to submit** (against the version's
> completion config + a required-questions gate), phrases that offer, and resolves the
> respondent's **accept / hold** — driving the F4.3 contradiction **completion-sweep** at
> the moment of offer and transitioning the session `active → completed` on a clean
> submit. Gated by `APP_QUESTIONNAIRES_ENABLED`; the offer phrasing has a per-feature
> sub-flag (but the deterministic assessment is always available).

## Intent

F4.1–F4.4 shipped which-question / what-answer / do-they-conflict / update-an-answer.
F4.5 is the wrap-up: recognise "done enough", offer submission in natural language,
run a final contradiction sweep, and submit (or hold for reconciliation). The first
P4 feature to transition session lifecycle state.

## Decisions (confirmed with the user at planning)

- **Two layers: deterministic gate + LLM phrasing.** Eligibility (`assessCompletion`) is
  pure deterministic math — _whether_ to offer. The capability only _phrases_ the offer
  (the "agent contract") — _how_ to say it. The LLM never decides eligibility, so the
  gate (incl. required questions) stays authoritative. (Per the user: "add a completion
  capability" — full F4.2/F4.3 capability shape for the wording, F4.1-style pure core
  for the decision.)
- **Required gate blocks the offer.** An unanswered `required` slot →
  `blocked_on_required`, even when weighted coverage already meets the threshold. This is
  the gate selection's `terminalDecision` lacks (it checks only weighted coverage).
- **Sweep conflict → hold for review.** On accept, if the completion-sweep finds
  contradictions (mode flag/probe), do NOT auto-submit: return findings, leave the
  session `active`, reconcile via F4.4, re-offer. Consistent with F4.3 never
  auto-overwriting.
- **Two routes.** `completion-status` (read-only assess + optional composed offer, no
  persistence) and `complete` (accept/hold action; persists status; runs the sweep).
- **No new config / no `sweep_only` mode.** Maps onto the committed flat config fields
  (`minQuestionsAnswered`, `coverageThreshold`, `maxQuestionsPerSession`) and the existing
  `shouldRunDetection(mode, windowN, 'completion-sweep')`. The plan's `completionConfig` /
  `sweep_only` terms don't exist in code.
- **Reuse, not reinvention.** `assessCompletion` reuses the F4.1 coverage helpers
  (`coverageRatio`, `answeredCount`, `unansweredQuestions`) — their param was narrowed to
  a structural `CoverageContext` (backward-compatible; `SelectionContext` still satisfies
  it). The routes reuse `buildSelectionContext` (no new context builder) and F4.3's
  `app_detect_contradictions` capability + `buildContradictionContext` for the sweep.
- **No new Prisma model.** Builds on F4.4's `AppQuestionnaireSession` / `AppAnswerSlot`.
  Adds `markSessionCompleted` to the F4.4 `_lib/answer-slots.ts` seam. No
  `AppQuestionnaireSessionEvent` yet (F4.6) — the `status` column is the only audit
  surface. **One hardening migration** (`20260605141500`): a raw-SQL partial unique index
  (`idx_app_questionnaire_session_preview_per_version`, WHERE `isPreview` = true) so the
  preview-session get-or-create is race-safe (catch P2002 → re-read); drift-probed in
  `lib/app/db-drift.ts`.
- **Completion sub-flag does NOT 404 its route.** Unlike F4.2–F4.4, a disabled
  `APP_QUESTIONNAIRES_COMPLETION_ENABLED` returns the (free) deterministic assessment
  without a composed offer — only the paid LLM phrasing is gated.
- **Fail-soft everywhere.** A failed offer composition → assessment + `diagnostic`, no
  offer. A failed/disabled completion-sweep → treated as clean (submit proceeds), never a
  5xx.

## Operator note

Completion logic is on under `APP_QUESTIONNAIRES_ENABLED` for the deterministic
assessment. To get a **composed offer**, also enable `APP_QUESTIONNAIRES_COMPLETION_ENABLED`
and `db:seed` (so the completion agent/capability exist) — with it off, `completion-status`
returns the assessment without an `offer` (not a 404). The **completion-sweep** runs only
when `APP_QUESTIONNAIRES_CONTRADICTION_DETECTION_ENABLED` is also on and the version's mode
is flag/probe; otherwise it's skipped (treated clean). The `complete` route **persists**: it
writes to a per-version preview session (`isPreview`, excluded from P8) and, on a clean
submit, sets the session `completed`.

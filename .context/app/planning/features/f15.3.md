# F15.3 — Stitched continuity

> **Everything still open across P15 is consolidated in [`f15-followups.md`](./f15-followups.md).**

> F15.2 made a run advance. F15.3 makes a respondent able to travel it.

Ships `stitched` presentation — one continuous conversation across legs — and, necessarily, the
wiring that F15.2 left open.

## The finding that reshaped the phase

**F15.2's respondent path was never mounted.** `HandoffCard` was fully written and imported by
nothing; a repo-wide grep returned only its own definition. `API.APP.EXPERIENCES.runStatus` had
exactly one caller: that orphaned card. The submit route returned `runId` and
`use-session-lifecycle.ts` never read it.

A respondent who finished a leg hit `SessionComplete` — a dead end — while the entire run backend
sat behind it, working and unreachable.

Two further gaps surfaced from there:

- The no-login surface had **no way to address leg B**. `/q/<versionId>` boots a NEW session and
  cannot open an existing one, and the poll returned a bare `sessionId` the anonymous client held
  no credential for.
- There was **no respondent entry point for an experience at all** — no start route, no share
  link in admin. The feature was unreachable end to end.

Closing those was not scope creep bolted onto `stitched`; without them `stitched` has nothing to
present.

## Confirmed decisions

- **Run membership rides the SESSION STATUS VIEW, not the submit response.** The submit response
  is seen once. A respondent who reloads a completed leg — or returns to the tab an hour later —
  would otherwise land on the terminal completion screen and never learn the journey continues.
  One indexed lookup on the leg table's `@unique` `sessionId`, null-short-circuited for the
  standalone majority, and fail-soft so an experience read cannot take down lifecycle status.

- **The continuity mode is read LIVE from the experience, never frozen onto the run.** An author
  switching modes mid-flight changes what in-flight respondents see. That is the point of the two
  modes sharing a persistence shape, and freezing it would quietly break the invariant.

- **`/x/<publicRef>` is the no-login address, with an httpOnly run cookie as the credential.**
  Chosen over the cheaper `/q/s/<sessionId>?t=<token>` **because of what the credential guards**:
  experience transcripts can contain raw safeguarding disclosures (F15.2 carries sensitivity state
  as summaries precisely because the raw text stays in the source transcript), and the stitched
  transcript endpoint replays exactly those legs. A URL-borne credential to that data lands in
  browser history, `Referer` headers, and any accidental paste. `httpOnly` also puts it beyond
  injected script. It additionally fulfils what the schema already promised `publicRef` would be.

- **Run-scoped, not session-scoped.** The thing authorised is a journey. A per-session credential
  has to be re-minted at every hop — which is exactly why the poll grew a minting side-effect.

- **The ref addresses; it never authorises.** `publicRef` is an eight-character human-quotable
  support code. `resolveRunSurface` refuses a bare ref outright, and there is no admin bypass:
  reading a respondent's conversation belongs on the audited session viewer.

- **The run token's HMAC is domain-separated.** A session token and a run token are structurally
  near-identical JSON signed with the same secret; without a domain prefix one could be replayed
  against the other's verifier if the field names ever converged. Cheap now, impossible to
  retrofit once tokens are in the wild. Pinned by a test.

- **Cookie NAMES are untrusted input.** `canReadRun` scans every `cq_run_*` cookie and trusts only
  the signed payload, so `cq_run_ANYTHING` carrying another run's token fails. Namespacing per run
  also stops a second journey evicting the first.

- **`canReadRun` is shared by the poll and transcript routes.** Two routes enforcing "prove you are
  inside this run" separately is precisely the shape that drifts, and the transcript route — which
  returns whole conversations rather than a one-word state — must never be the laxer one.

- **Stitched history is a pure READ, rendered as a separate prop.** `QuestionnaireTurn` was
  deliberately NOT widened with a `seam` role: the turn array feeds the reveal cursor, the
  inspector drawer, and `report/craft.ts`, which maps `turn.role` straight onto an `LlmMessage`. A
  synthetic role would have reached a real LLM call as a role the provider does not accept.

- **Auto-continue in `stitched`, never in `linked`.** The objection to moving someone without
  their agreement is why `linked` refuses to; it does not apply when the author chose one
  continuous conversation and the respondent's experience is simply that the interviewer carried
  on. Endings (`conclude` / `failed`) never auto-advance — an ending deserves to be read.

- **The seam marker defaults to `divider`.** A respondent moving into a materially more probing
  follow-up should be able to see the subject changed rather than discover it mid-answer. Hiding
  the seam stays an explicit author choice.

## What shipped

**Settings**: `stitchedSeamMarker` (`divider` | `none`) — types, strict narrowing, Zod, and a
Settings control revealed only under `stitched`. A key plus a default, no migration.

**Lib**: `SessionExperienceContext` + `StitchedHistory`/`StitchedSegment` (`run/types.ts`);
`useRunHandoff` and `useStitchedHistory` hooks; `boot-fetchers.ts` — the four token-authed session
reads extracted from `AnonymousSessionBoot` so the run surface shares them rather than copying.

**API seam**: `run-access.ts` (shared `canReadRun`), `run-access-token.ts` (mint/verify/cookie),
`run-surface.ts` (`/x/` resolution), `experienceContextForSession` and `loadStitchedHistory`
(`run-read.ts`).

**Routes**: `GET /runs/:runId/transcript` (prior legs, respondent-only, no admin bypass); run
cookie set on `POST /:id/runs`; leg-B token minting on the poll for token- and cookie-authed
callers.

**Pages**: `/x/[publicRef]` (stable journey address, not indexable) and `/x/new/[experienceId]`
(the shareable start link, `router.replace` so Back does not mint a second run).

**Components**: `StitchedContinuation`, `SeamDivider`, `RunSessionBoot`, `RunStartBoot`,
`ExperienceRespondentLink`; `HandoffCard` slimmed to presentation and made callback-driven;
`QuestionnaireChat` gained `stitchedHistory` / `stitchedSeamLabel`; `SessionWorkspace` mounts the
handoff from its completed arm.

**Tests**: 44 new unit tests across the token, the access rule, the `/x/` resolver, the history
loader, the seam-marker narrowing, and the continuation component.

## Design notes

**`router.push` is a no-op on a stable address.** The URL for leg B is the URL already in the
address bar, so an href-shaped API would have made the stitched handoff silently do nothing on
`/x/`. Caught while designing the credential, not by a test — which is why both components take an
`onContinue` callback and the workspace decides push vs refresh.

**Extracting the poll loop before writing the second presentation.** `linked` and `stitched` watch
the identical server event. Keeping two copies of the timeout, the hidden-tab back-off and the
stop condition would have drifted on exactly the parts that are hard to notice when wrong.

**No `as` on the wire.** The extracted transcript fetcher validates `reasoning` in full rather
than accepting `unknown` and casting — a partial schema plus an assertion is the thing the
boundary exists to prevent.

## Deferred

- **Cross-device resume.** The credential deliberately cannot travel with a copied link, so
  another device gets an explanatory notice. A run-level resume-by-code — the equivalent of the
  questionnaire surface's `ResumeByRefEntry` — is the natural follow-up.
- **The authenticated surface still changes URL between legs.** `/questionnaires/<id>` addresses a
  session; converging both surfaces on `/x/` is its own phase.
- **The run-level report** the `conclude` path promises — F15.4. Until then the completion screen's
  last-leg respondent report stands in.
- `merged` continuity — F15.6, and may never ship.

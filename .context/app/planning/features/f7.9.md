# F7.9 — Inline answer correction ("fix this answer", Variant B)

> Committable tracker for **F7.9**. A respondent who is misheard, or whose answer the extractor
> misread, could previously only correct it by sending a fresh chat turn ("no, I meant disagree") —
> heavier than the mistake warrants, and worse: the contradiction detector compares the new statement
> against the same slot's prior value and often fires a **false contradiction warning**. This adds a
> lightweight inline correction gesture that fixes the just-captured answer in place.

## Intent

Let the respondent correct what the latest turn captured through a small inline editor — beneath the
most-recent message in the chat **and** on the answer-panel rows — instead of re-explaining in a new
turn. The contradiction-warning problem dissolves for free: corrections route through the form-edit
path, which never touches the turn pipeline.

## Decisions

- **Reuse the form-edit path, don't add a write endpoint.** `PUT …/answers` already validates
  per-type values, records a `manual` refinement-history entry, flips the slot to `refined` +
  `respondentEdited`, and (data-slot mode) reconciles the reading. The fix reuses it verbatim via a
  thin `useInlineCorrection` hook. Because it bypasses the turn pipeline, **no extraction or
  contradiction detection runs on a correction** — the core win.
- **One shared editor, two surfaces.** `InlineAnswerEditor` always edits _question_ slots, reusing
  `QuestionField` for the per-type control. It mounts in the chat (`CorrectionStrip`, beneath the
  most-recent turn, shown once the reply settles) and on the panel rows (`AnswerSlotItem` "Edit
  answer"; `DataSlotRow` "Edit").
- **Data-slot mode edits the mapped questions.** A data-slot fix carries the slot's mapped questions
  (from `coverage.questions`, enriched with `key`/`type`/`typeConfig`/`value`); the PUT's
  reconciliation recomputes the reading. A data slot with **no** mapped questions shows no gesture.
  Accepted tradeoff: the reconciled reading becomes a deterministic restatement (`direct`), exactly
  as form-view edits already behave — keeps the structured answers (what reports rely on) correct.
- **Admin config toggle, default on, no platform flag.** `inlineCorrectionEnabled` is a per-version
  config (Settings tab). Respondent-facing UX with sensible default, so unlike voice/attachments/
  reasoning it carries no platform dark-launch flag. The read-only admin viewer never shows it; a
  blocked (non-active) session hides it (the PUT rejects it anyway).
- **Question-mode just-filled tracking.** The chat strip needs the keys the latest turn filled.
  `diffNewlyFilled` was data-slot-only; added `diffNewlyFilledQuestions` for question mode. The
  panel's own scroll/stepper stays data-slot-only (unchanged behaviour).

## Tests

- Unit: `diffNewlyFilledQuestions`, `buildCorrectionTargets` (both modes + hide-when-no-questions),
  `updateConfigSchema` toggle, `resolveInlineCorrectionForVersion`, enriched coverage fixtures.
- Component: `InlineAnswerEditor` (seed/save/clear/cancel/error, token header), `CorrectionStrip`
  (lists targets, opens editor, empty → renders nothing).
- Reused unchanged: `PUT …/answers` recording + reconciliation (existing `answers-route` /
  `form-answers` tests).

## Not here

- No direct `AppDataSlotFill` edit endpoint (data slots without mapped questions stay uncorrectable
  inline). No agent acknowledgement turn after a correction (the panel updates silently). No
  auto-resolution of a pending contradiction probe.

# F6.4 ‚Äî Demo session reset (DEMO-ONLY)

> Committable tracker for **F6.4**. Adds `POST /api/v1/app/demo-clients/:id/reset-sessions`
> ‚Äî the between-demos "clean slate" that hard-deletes the session graph (sessions + answer
> slots + turns + events) for every version of every questionnaire attributed to a demo
> client, with an optional `?resetInvitations=true` invitation sweep. Destructive,
> admin-only, audited, and stripped by a real client engagement. Gated by the master
> `APP_QUESTIONNAIRES_ENABLED` flag only (no sub-flag).

## Intent

A ConQuest sales demo runs a prospect's questionnaire against a demo client. Once a
prospect has driven a conversation, that client's questionnaires carry a respondent
session graph. Before the next prospect, the operator needs a clean slate. F6.4 is the
last piece of the distributed P2.5 demo-tenancy work that needed the session graph to
exist (F4.6 + F6.1) before it could be built ‚Äî it deletes run data only, leaving the
client, questionnaires, versions, and config intact.

## Decisions (confirmed with the user)

- **No dedicated feature sub-flag.** Gate on the master `withQuestionnairesEnabled` only,
  consistent with the create/update/delete demo-client siblings ‚Äî unlike the F6.x
  respondent sub-flags (live-sessions, voice, cost-cap), this is a free, synchronous,
  admin-only DB op with no LLM cost and no respondent surface. It's already triple-gated by
  admin auth + the typed confirmation + the anonymousMode refusal; a flag would add no
  safety and a forgotten-disabled flag would just 404 a legitimate admin action.
- **409 (anonymousMode) before 400 (confirmSlug).** The anonymous-mode refusal is a hard
  structural block, so a correct slug must still be refused ‚Äî surfacing the true blocker
  first. Matches the existing DELETE handler's guard-before-work posture.
- **"403 on ownership" is the admin-role guard.** `AppDemoClient` has no per-user owner
  column ‚Äî it's a global admin fixture, so `withAdminAuth` is the whole ownership story. No
  ownership check was added (nothing to check against).
- **No Profile table.** The indicative spec named "profiles"; profile data is
  `AppQuestionnaireConfig.profileFields` (config, not run data), not a session-scoped model.
  `deletedCounts` covers `sessions`, `answerSlots`, `turns`, `events`, `invitations` ‚Äî the
  tracker records the discrepancy so nobody hunts for a missing table.
- **Preview sessions included.** The delete filter is `versionId`-scoped and does not
  exclude `isPreview` ‚Äî a clean slate clears admin preview exercises too.
- **Empty graph still audits.** A client with zero questionnaires returns all-zero counts +
  200 + an audit row (operator intent recorded; idempotent reruns visible).
- **No schema change, no migration.** Reuses the existing cascade FKs; the delete is pure
  query orchestration.

## Build shape ‚Äî 1 PR

- **Schemas.** `resetSessionsSchema` (`{ confirmSlug }`, reuses the kebab-case `slugField`)
  - `resetSessionsQuerySchema` (`?resetInvitations=` ‚Üí boolean) in
    `lib/app/questionnaire/demo-clients/schemas.ts` (exported via the module barrel).
- **Vocabulary.** `RESET_PRESERVED_INVITATION_STATUSES = ['started','completed','revoked']`
  in `lib/app/questionnaire/invitations/types.ts`.
- **DB seam.** `app/api/v1/app/demo-clients/_lib/reset.ts` ‚Äî `loadResetTargets(demoClientId)`
  (collect version ids + the `anyAnonymous` signal in one query) and
  `performReset(versionIds, { resetInvitations })` (one `$transaction`; children before the
  parent session for accurate `deleteMany().count`; opt-in invitation sweep with the
  preserve filter; empty-graph short-circuit, no transaction).
- **Route.** `app/api/v1/app/demo-clients/[id]/reset-sessions/route.ts` ‚Äî
  `withQuestionnairesEnabled(withAdminAuth(...))`. Gate order: flag (404) ‚Üí auth (401/403)
  ‚Üí unknown id (404) ‚Üí body validation (400) ‚Üí anonymousMode (409) ‚Üí confirmSlug mismatch
  (400) ‚Üí `performReset` ‚Üí `logAdminAction('app_demo_client.reset_sessions')` ‚Üí 200 with
  `{ id, deletedCounts, resetInvitations }`.

## Tests

- **Unit (schemas)** ‚Äî `resetSessionsSchema` accepts kebab-case `confirmSlug`, rejects
  empty/uppercase/missing; `resetSessionsQuerySchema` `'true'`‚Üítrue, `'false'`/absent‚Üífalse,
  other‚Üíreject.
- **Unit (`_lib/reset`)** ‚Äî `loadResetTargets` maps version ids and computes `anyAnonymous`
  (true on any config.anonymousMode; null config ‚Üí false; empty client ‚Üí `[]`/false).
  `performReset` deletes answerSlots‚Üíturns‚Üíevents‚Üísessions in order with accurate counts,
  calls invitation `deleteMany` with `status: { notIn: [...] }` only when `resetInvitations`,
  and short-circuits an empty graph without opening a transaction.
- **Integration (route matrix)** ‚Äî flag off ‚Üí 404 before auth/DB; unauthenticated ‚Üí 401;
  non-admin ‚Üí 403; unknown id ‚Üí 404; malformed body ‚Üí 400; **anonymousMode on ‚Üí 409 even
  with a correct slug** (locks the precedence); slug mismatch ‚Üí 400 CONFIRM_SLUG_MISMATCH;
  happy path ‚Üí 200 + counts + `app_demo_client.reset_sessions` audit; `?resetInvitations=true`
  threads through to `performReset`; empty-graph client ‚Üí 200 all-zero + still audited.

## Not in F6.4

- An admin UI button / confirmation modal for the reset (API-only this phase; an
  admin-shell trigger can land alongside the P7 respondent surfaces).
- Clone-for-client (the other remaining distributed P2.5 item, P3+).
- A soft/restorable reset or export-before-wipe ‚Äî the reset is a hard delete by design.

# F4.4 — Answer refinement

> Committable tracker for **F4.4**, the answer-update primitive of P4 (Conversational
> engine, non-streaming). Given a respondent's already-captured answers + new context
> (a clarifying message and/or a flagged contradiction), decide per slot whether to
> **refine** (the value evolved → provenance `refined`), **overwrite** (a mistaken
> capture → provenance preserved), or **leave** it — then preserve a
> `refinementHistory` and **write** the change. Serves both contradiction resolution
> (consuming F4.3's `suggestedProbe` handoff) and the general "user clarified earlier"
> flow. Gated by `APP_QUESTIONNAIRES_ENABLED` + a per-feature sub-flag (404 when off).

## Intent

F4.2 shipped "what did they answer?"; F4.3 shipped "do the answers conflict?"; F4.4 is
the update — change an earlier answer in light of later context so the final dataset is
coherent. It's the consumer of F4.3's `suggestedProbe` (the detection→refinement
handoff) and the producer of the reserved `refined` provenance label.

## Decisions (at planning / build — confirmed)

- **Persistence is in scope — a deliberate divergence from F4.1–F4.3.** Those were
  no-persistence preview features (the answer tables were slated for F4.6). F4.4
  introduces the **persistence foundation**: a minimal `AppQuestionnaireSession` + the
  `AppAnswerSlot` model (with `refinementHistory`) + a migration, and a **real write
  path**. F4.6 builds its live per-turn loop on top of these same tables. (Recorded so
  F4.6 knows they exist.)
- **`overwrite` keeps the original provenance; only `refine` → `refined`.** `refined`
  means "the value evolved across turns" — a precise audit signal. A typo/mis-capture
  fix (`overwrite`) is not an evolution, so it preserves the original label but still
  appends a history entry (`source: 'correction'`) so the prior value isn't lost.
- **Own agent + capability** (the LLM decides refine/overwrite/leave), a true peer of
  the extractor (006) and detector (009) with its own budget. The _decision_ is
  LLM-driven; the _apply_ step (`applyRefinement`) is pure deterministic logic.
- **Core-local action/source tuples.** `REFINEMENT_ACTIONS = [refine, overwrite, leave]`
  and `REFINEMENT_SOURCES = [contradiction, clarification, correction]` live in the
  core (not the shared `types.ts`), the analogue of `CONTRADICTION_SEVERITIES` — the
  Zod enum and the normaliser derive from them.
- **No clock in the pure core.** `RefinementHistoryEntry` carries no timestamp; the
  persistence seam stamps `createdAt` at write time. Keeps the core deterministic.
- **Reuse F4.2 value validation.** The normaliser validates a new value with
  `validateAnswerValue` (which reads F2.1's `typeConfigSchemaFor`) — zero duplication;
  a refined value obeys the same per-type rules as a freshly extracted one.
- **Resilient schema, normaliser enforces semantics.** Zod checks structure only
  (`newValue` optional-unknown); the refine/overwrite-requires-a-value rule, the
  slot-must-be-answered rule, value-fails-type, and no-op drops are the normaliser's,
  which drops one odd decision rather than failing the pass (the F4.2 doctrine).
- **Own opt-in sub-flag** (`APP_QUESTIONNAIRES_ANSWER_REFINEMENT_ENABLED`, off by
  default) on top of the master flag — refinement spends an LLM call per pass.
- **Preview session for the write path.** No real respondent sessions exist yet, so
  the admin route exercises the write path against a per-version preview session
  (`isPreview`, reused idempotently, excluded from P8 analytics).
- **Fail-soft route**: a capability error → 200 with empty decisions + a `diagnostic`,
  never a 5xx (the idempotent seed upsert may already have persisted).

## Operator note

Answer refinement is off until **both** `APP_QUESTIONNAIRES_ENABLED` and
`APP_QUESTIONNAIRES_ANSWER_REFINEMENT_ENABLED` are on; run `db:seed` so the refiner
agent + capability exist. With the sub-flag off, the refine-answer route 404s. Unlike
the F4.2/F4.3 preview routes, this route **persists**: it creates a per-version preview
session (`isPreview: true`) and writes `AppAnswerSlot` rows. Preview sessions are
excluded from P8 analytics by the `isPreview` flag.

# F9.3 — Forking documentation

> Committable tracker for **F9.3**. The fork-readiness deliverables: a `forking.md` for a
> client-engagement team inheriting ConQuest, the external-fork upgrade guidance for tracking
> upstream Sunrise, and a consolidated record of what this build contributed back upstream.
> **Docs-only** — no code, schema, or route changes.

## Intent

F9.3 makes ConQuest hand-off-able. Two readers need a map. A **client-engagement team** that
inherits this repo to deliver a different vertical needs to know what the asset is (the
conversational-questionnaire engine), what to rename, and how to replace the demo tenancy with
a real client. Any **fork tracking upstream Sunrise** needs the merge discipline — upstream
remote, semantic-conflict rules, never editing Sunrise's migration SQL, and the loop for
contributing generic fixes back rather than carrying local patches forever. F9.3 writes both
maps and records the upstream contributions ConQuest's own build already made.

## Decisions

- **Two new files, not three; upgrade guidance is a section.** Deliverables 1 (re-skinning) and
  2 (external-fork upgrade guidance) land in one `forking.md` as Parts A and B — they share an
  audience boundary and one review pass. The contributions record is its own file because it's a
  distinct artefact (a ledger + sales evidence) with a different home (`planning/`, beside
  `upstream-gaps.md`).
- **Cross-reference platform docs, never duplicate or edit them.** `CUSTOMIZATION.md`,
  `VERSIONING.md`, and `.context/database/migrations.md` are **platform-owned** — they merge
  through from upstream, so editing them here is the fork-and-edit trap. All F9.3 output lives in
  ConQuest's app space and links to the platform docs as authoritative; the new value is
  ConQuest-specific detail (the rename surface, the demo-tenancy paths) and the _sequencing_ of
  the upstream-tracking mechanics into one guide.
- **`forking.md` is the canonical home for the three demo-tenancy replacement paths.**
  `demo-clients.md` § "Fork guidance" already links _to_ `forking.md` for them, so they live here
  (delete entirely · rename → `AppTenant` + RLS · keep as `AppBrand`), not duplicated there.
- **Rename counts are live, marked approximate.** The rename-surface table carries
  `grep`-measured orders of magnitude with an instruction to re-measure before starting — counts
  drift as the codebase grows.
- **No CHANGELOG entry.** App-surface docs, not Sunrise platform public surface.

## Build shape (branch `feat/F9.3-forking-docs`)

- **`.context/app/questionnaire/forking.md`** (new, keystone) — Part A (re-skinning): what to
  keep verbatim · the rename surface table (18 `App*` models, 18 `app_*` tables, 11
  `APP_QUESTIONNAIRES_*` flags, the `questionnaire`/`ConQuest` word counts) · ordered `sed`
  recipes with the `app_questionnaire`-before-`questionnaire` collision rule and the "don't touch
  the `lib/app/` bootstrap seams" guard · what to replace (extraction prompts, theme defaults,
  sample content) · the demo-tenancy replacement (grep `DEMO-ONLY`, three paths). Part B
  (upstream tracking): wire the `upstream` remote · the merge recipe · semantic-conflict
  discipline · the version-decision references · the contribute-back loop.
- **`.context/app/planning/contributions-upstream.md`** (new) — the resolved-counterpart ledger to
  `upstream-gaps.md`: a UG-1 / UG-2 table (Sunrise PR #286 / #285, both in v0.0.1) + the honest
  "smaller than planned, still a real sales artefact" framing + the how-an-entry-gets-here loop.
- **`.context/app/questionnaire/README.md`** — registered `forking.md` in the namespace index.
- **`.context/app/planning/development-plan.md`** — flipped the F9.3 status (summary line, P9 row,
  §F9.3 heading) from "not started" to done with the PR number.

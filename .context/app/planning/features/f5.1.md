# F5.1 — Judge agents

> Committable tracker for **F5.1**, the opening feature of P5 (Design-time evaluation).
> Stands up seven `kind='judge'` agents that score a questionnaire version's **structure**
> against its stated goal/audience across distinct dimensions (clarity, coverage,
> duplicates, type fit, ordering, audience match, goal match), each emitting **actionable
> findings** — the rows F5.3's review queue will consume. Ships the judges, a dispatch
> capability, and a **no-persistence preview route**; persistence (F5.2) and the review
> queue (F5.3) follow. Gated by `APP_QUESTIONNAIRES_ENABLED` + a sub-flag.

## Intent

P5 lets an admin evaluate a questionnaire's design before launch. F5.1 is the panel: seven
judges read the authored structure (goal, audience, sections, questions) and return a
per-dimension score plus concrete proposed edits. Unlike P4 there is no respondent — the
judges grade an artefact that already exists. With the preview route, an admin can run the
panel by hand and tune it; F5.2 builds persistence on the same dispatch seam.

## Decisions (confirmed with the user at planning)

- **App-native dispatch, NOT Sunrise's `AiEvaluationRun` worker.** The development-plan's
  F5.2 sketch named `AiEvaluationRun`, but that worker grades a _subject's generated output
  over a dataset of cases_ and scores 0–1; P5 grades an existing artefact and needs
  _suggestions_. So the judges are dispatched like F4.2–F4.5 — one structured
  `runStructuredCompletion` call per dimension via a `BaseCapability` — and emit
  score **+ findings**. Recorded as a reframe in the dev-plan decisions log.
- **Still seeded as `kind='judge'`.** The judges appear in the platform Judges surface and
  reuse the agent resolver / cost / admin-edit machinery — but their **rubric lives in code**
  (`evaluation/judge-prompt.ts`), not the agent row, the F4.5 completion-agent split. The
  seeded `systemInstructions` are a self-describing mirror; tuning is a git-diffable code
  change.
- **`isSystem: false` (app component).** Unlike the platform judges (`016-evaluation-judges`,
  `isSystem: true`, seed overwrites rubric), these are app-owned: editable, in config
  backup/export, and re-seed re-asserts only `kind`/`isSystem` — never clobbers operator
  edits.
- **One seed file, registry loop.** Seven judges are a _set_ (one panel), so a single
  `018-design-evaluation-judges.ts` iterating the dimension registry beats the F4
  one-agent-per-file convention — mirroring the platform's `016-evaluation-judges.ts`.
- **`processesPii = false`.** Goal/audience/questions are admin-authored content, not
  respondent personal data — simpler than the F4.2–F4.5 answer capabilities (no
  `redactProvenance` override).
- **Sub-flag 404s the whole route when off.** Unlike F4.5 (whose deterministic assessment
  stays free under the master flag), the entire evaluate route is paid LLM work — there is no
  free half — so `APP_QUESTIONNAIRES_DESIGN_EVALUATION_ENABLED` off → 404, the F4.2 posture.
- **Findings address targets by `targetKey`** — a slot `key`, `section:<title>`, or `goal` /
  `audience` — a free string reconciled fail-cleanly by F5.3 at apply time (the pure core has
  no live graph), the F2.3 revert-planner posture. `dimension` is stamped by the capability,
  never the LLM.
- **Fail-soft per judge.** A dimension whose judge errors/is unseeded returns a `diagnostic`;
  the other six still return. Only _zero_ judges seeded → 404 (`run db:seed`).
- **No migration, no persistence.** F5.1 adds no schema — the run + suggestion models are
  F5.2. The preview route stores nothing.
- **No CHANGELOG entry.** ConQuest app surface, not the Sunrise platform public surface
  (F4.x added none either).

## Build shape — 2 PRs (one branch, risk-sequenced)

### PR1 — pure core + seeds + sub-flag (no LLM, no HTTP)

Pure core `lib/app/questionnaire/evaluation/`: `types.ts` (`EVALUATION_DIMENSIONS`,
`FINDING_SEVERITIES`, `JudgeFinding`, `JudgeVerdict`, `VersionStructureInput`),
`dimensions.ts` (the dimension→slug/label/summary registry), `judge-schema.ts` (Zod verdict
contract + `validateJudgeVerdict` + JSON-schema + `MAX_FINDINGS_PER_JUDGE`),
`structure-schema.ts` (the shared `VersionStructureInput` Zod + `parseAudienceShape`),
`judge-prompt.ts` (per-dimension rubrics + structure serialiser + retry), `index.ts` barrel.
Constants (`APP_QUESTIONNAIRES_DESIGN_EVALUATION_FLAG`, capability slug/handler/function-def)
in `constants.ts`; `isDesignEvaluationEnabled()` in `feature-flag.ts`. Seeds
`018-design-evaluation-judges.ts` (7 judges) + `019-design-evaluation-flag.ts`. Unit tests.

### PR2 — dispatch capability + preview route + integration + docs

Capability `capabilities/evaluate-structure.ts` (one judge per call, `reasoning` tier,
`runStructuredCompletion`, `processesPii = false`), registered in `lib/app/capabilities.ts`;
`AiCapability` seed `020-design-evaluation-capability.ts` (unbound — dispatched per-judge).
Route `…/versions/[vid]/evaluate-preview/route.ts` (gate stack → load structure → load judge
agents → concurrent fan-out, fail-soft per judge); route-local loader
`_lib/evaluation-structure.ts`; per-admin `designEvaluationLimiter` (20/min). Capability +
route integration tests. Docs: `design-evaluation.md` + README index. This tracker +
dev-plan status flip.

## Tests

- **Unit** (`tests/unit/lib/app/questionnaire/evaluation/`): `dimension-parity` (7 distinct
  dimensions ↔ registry ↔ slugs, round-trip, every dimension builds a prompt); `judge-schema`
  (well-formed verdict, empty findings, optional `sourceQuote`, score bounds 0/1, score
  out-of-range / non-numeric, bad severity, empty required fields, the findings cap, issue
  paths, JSON-schema shape); `judge-prompt` (system+user pair, dimension-specific rubric
  splice, structure serialisation incl. keys/types/required, continuous numbering across
  sections, goal/audience placeholders, empty sections, determinism, the `targetKey`
  convention, retry message). 28 cases.
- **Integration — capability** (`evaluate-structure-capability.test.ts`): happy path (score +
  findings), dimension stamp, clean pass (empty findings), `reasoning` tier + system-default
  binding, cost logged as CHAT with the dimension in metadata, malformed→retry repair,
  no-silent-failure (`evaluation_failed`), provider fail-closed, unknown-dimension schema
  reject.
- **Integration — route** (`evaluate-preview-routes.test.ts`): gate order (404 master-off
  before auth; 404 sub-flag-off after auth, no dispatch), 401/403, scope-404,
  not-configured-404 (no judges), 429, full-panel fan-out (one dispatch per dimension),
  dimension subset + dedupe, unknown-dimension 400, finding aggregation, fail-soft per judge
  (one errors → diagnostic, rest return), missing-judge → `judge_not_configured`. 51 tests
  total across the feature.

## Sequencing

- **PR1** — pure core + seeds + flag + unit tests. The judge contract + rubrics are
  exhaustively unit-testable with no LLM/HTTP.
- **PR2** — the LLM-touching surface: capability + preview route + integration tests + docs.

Gates each PR: `/pre-pr`, `/test-review`; `/code-review` + `/security-review` on PR2 (a new
authed admin route that dispatches paid LLM work).

## Operator note

Design-time evaluation is opt-in: enable `APP_QUESTIONNAIRES_DESIGN_EVALUATION_ENABLED` on
top of `APP_QUESTIONNAIRES_ENABLED` and run `db:seed` (judges + capability + flag). Then
`POST …/versions/:vid/evaluate-preview` (admin-only) runs the panel — all seven judges by
default, or a `dimensions[]` subset — and returns each verdict (score + findings) with no
persistence. The run takes a per-admin 20/min sub-cap; a flaky judge degrades to a
`diagnostic` rather than failing the panel. Disabled → 404.

## Not in F5.1

Run + suggestion persistence and run history, the `AppQuestionnaireEvaluation*` models, the
link to a run (F5.2); the suggestion review queue with accept/decline/edit/apply and
stale-suggestion derivation (F5.3); any admin UI (P5 UI); empirical calibration of judge
cost/timeouts against real runs.

# F2.5.1 — Demo-client foundation

> Committable tracker for **F2.5.1**, now the **standalone entirety of P2.5**.
> The 2026-06-03 decomposition (see the [development-plan decisions log] and
> [PR #20]) collapsed the old three-feature P2.5 to this foundation slice and
> distributed the rest (theming, invitation branding, session reset, content
> seed) into their host phases. This doc plans the foundation. Reconciled against
> Sunrise's actual seams and the F1.1/F2.1 patterns already in the tree. Flip
> statuses as commits land.
>
> [development-plan decisions log]: ../development-plan.md#decisions-log
> [PR #20]: https://github.com/human-centric-engineering/conquest/pull/20

## Intent

Stand up the demo-client **identity**: a `AppDemoClient` table, a nullable
`demoClientId` FK on `AppQuestionnaire`, and the admin surface to **create demo
clients and attribute questionnaires to them**. This is the one slice of demo
tenancy that must **lead** — with the FK in place from here on, P3 (invitations),
P4/P6 (sessions), and P7 (user UI) build tenant-aware in a single pass instead of
retrofitting.

Deliberately lightweight and app-owned: application-layer scoping on
single-tenant Sunrise, **not** a security isolation boundary. Demo clients aren't
adversarial, so hard isolation is out of scope (a real multi-customer fork
activates Sunrise's RLS seam — see the schema warning below). The whole surface
is gated by `APP_QUESTIONNAIRES_ENABLED` (404 when off), API-first.

**Definition of feature complete** (the plan's F2.5.1 DoD): an admin can create
demo clients and attribute questionnaires to them; the F2.1 list/detail surface
shows attribution; the FK is in place for downstream phases.

## PR shape (gate-suite-tax calibration) — **one PR**

The plan sizes F2.5.1 at `~1–2 PRs`; the honest cut is **one**. Applying the
[gate-suite-tax doctrine](../development-plan.md#two-feature-shapes-pr-sizing): a
risk-type split is something to **justify**, not a default, and F2.5.1 has **one
dominant risk type** — additive admin CRUD over a non-sensitive identity table
plus a nullable scalar FK. There is no live destructive endpoint, no file upload,
no LLM call, no PII surface. The whole diff resolves to one reviewable claim —
_"an admin can manage demo clients and attribute questionnaires, gated and
audited"_ — that one `/security-review` + `/code-review` pass covers cleanly.
Splitting (e.g. schema-PR vs UI-PR) would pay the entire gate suite twice for a
contained feature — the F1.1 "PR1 was below the floor" mistake.

The only mild correctness concern — DELETE refusing while a questionnaire still
references the client — is a single guard, easily held in one pass.

> Commits ≠ PRs: the task table below is **commit-sized work inside one PR**.

## Architecture decisions

### AD1 — Identity only; theming is a separate module that lands later

F2.5.1 ships `AppDemoClient` with **identity** fields only (`slug`, `name`,
`description`, `isActive`). The ~14 theme columns, the `resolveTheme()` /
`themeToCssVariables()` module, and logo/favicon upload are **not** here — they
land with their first renderer (F3.4 if themed invites are adopted, else F7.1).
This is the decision that keeps F2.5.1 one PR with no file-handling/CSS-injection
risk. Theme columns are additive nullable columns later — a cheap migration, no
backfill.

### AD2 — `demoClientId` FK shape: proper relation, `onDelete: SetNull`, app-layer 409 guard

The deep spec proposed a plain `String` FK "to avoid a reverse relation field."
**Diverge:** use a proper Prisma `@relation` with a reverse `questionnaires
AppDemoClient[]` field and `onDelete: SetNull` (FK nullable). Rationale:

- The reverse relation is **useful** — the list endpoint counts attributed
  questionnaires per client (`_count`), and the DELETE guard reads it directly.
- DB-level referential integrity for free; fully modelled, drift-clean (a scalar
  FK + relation is not a Prisma-unmodelled object — unlike the deferred pgvector).
- `onDelete: SetNull` matches CLAUDE.md's "retained config, FK nullable" policy
  (a questionnaire survives its demo client being deleted; attribution just
  clears). The primary UX is still the **app-layer 409 guard** (DELETE refuses
  while `_count.questionnaires > 0`, telling the admin to detach first); SetNull
  is the schema-honest backstop, not the happy path.

`null` = "Generic Sunrise demo" (Sunrise defaults end-to-end). Pre-F2.5.1
questionnaires keep working — **no backfill**.

### AD3 — Attribution = a questionnaire-level PATCH (the one shared-file touch)

There is **no** `PATCH /questionnaires/:id` today (F1.1 made `POST` on root +
`GET` on `byId`; F2.1 put meta-edit on the _version_). Attribution introduces
`PATCH /api/v1/app/questionnaires/:id` accepting `{ demoClientId: string | null }`
(404 if the client id is unknown; nullable to detach). This is the **only** place
F2.5.1 edits files in John's P2 territory — an additive handler in the existing
`app/api/v1/app/questionnaires/[id]/route.ts` (GET-only today) plus a small
enrichment of the list/detail read models to surface the attached client. See
[Collision watch](#collision-watch-john).

### AD4 — Not a security boundary (carried verbatim from the plan)

> This table is a branding/content partition, not the foundation a real
> customer-isolation layer plugs into. A multi-customer fork activates Sunrise's
> RLS tenancy seam (`TENANCY_MODE=multi`, `Org`/`orgId` at `lib/db/client.ts`) —
> it does **not** harden this demo table into an isolation mechanism. Every demo
> module carries the `// DEMO-ONLY:` header pointing at `forking.md`.

### AD5 — Keep demo code in a marked, grep-isolated module

Per the decomposition principle (_decompose the schedule, preserve the module
boundary_), all F2.5.1 domain code lives in a new **`lib/app/questionnaire/demo-clients/`**
sub-module (types + client-safe view contracts), routes under
`app/api/v1/app/demo-clients/`, UI under `components/admin/demo-clients/` +
`app/admin/demo-clients/`, each headed `// DEMO-ONLY:`. A fork strips demo
tenancy with `grep -rl DEMO-ONLY`.

## Schema design

In `prisma/schema/app-questionnaire.prisma`, extending the existing graph. Follow
[`schema.md`](../../questionnaire/schema.md) and the **migrate-dev schema-fold
footgun** recipe exactly (review generated SQL; strip every phantom platform
DROP/ALTER/RENAME; `npm run db:drift-check` green). Banked lesson: Prisma 7
`migrate dev` is **not** transactional — see [[prisma7-migrate-dev-partial-apply]].

```prisma
/// DEMO-ONLY: demo-tenancy attribution + branding partition, NOT a security boundary.
/// FORK-GUIDANCE: see .context/app/questionnaire/forking.md § "Replacing demo tenancy"
/// (delete entirely · rename to AppTenant + RLS · keep as AppBrand without the marker).
/// Identity only in F2.5.1 — theme columns land with their first renderer (F3.4 / F7.1).
model AppDemoClient {
  id          String   @id @default(cuid())
  slug        String   @unique // URL-safe; admin URLs (and later invite links)
  name        String   // display name ("Acme Bank Demo")
  description String?  // internal admin note
  isActive    Boolean  @default(true) // soft-disable without delete
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  questionnaires AppQuestionnaire[] // reverse relation (AD2): count + delete guard

  @@map("app_demo_client")
}

// Add to the EXISTING AppQuestionnaire model:
model AppQuestionnaire {
  // … existing fields …
  demoClientId String? // DEMO-ONLY: null = generic Sunrise demo; no backfill
  demoClient   AppDemoClient? @relation(fields: [demoClientId], references: [id], onDelete: SetNull)

  @@index([demoClientId])
  // … existing @@index([status]) / @@map …
}
```

## Skills & agents

The project's command/skill workflow drives this feature end-to-end:

- **Build:** `/api-builder` for the demo-client CRUD + attribution routes (Zod,
  `withAdminAuth`, audit, standard envelopes — the exact recipe the skill emits).
  `/page-builder` for the admin pages (flag-gate, `serverFetch` SSR, metadata).
  `/component-builder` + `/form-builder` for the table, the create/edit form
  (`mode: 'onTouched'`, `apiClient`, `<FormError>`, `<FieldHelp>` on every
  non-trivial field), and the attribution control.
- **Test (the four-command chain, not looped reflexively):** `/test-plan` over
  the branch diff → `/test-write plan` (spawns the **test-engineer** subagent,
  which reads `.context/testing/` and validates lint + type-check before
  finishing) → `/test-review` → `/test-fix --all`. Anti-green-bar: assert what the
  handlers _do_ (the 409 guard actually refuses, attribution actually persists),
  not what mocks return.
- **Gate suite before merge (the fixed tax this one PR pays once):** `/pre-pr`
  (type-check, lint, coverage, anti-pattern scan, public-surface/CHANGELOG check)
  → `/security-review` (admin-auth coverage, IDOR on `:id`, the DELETE guard,
  input validation) → `/code-review` (correctness + reuse/simplification). These
  invoke the `pr-review-toolkit` agents (code-reviewer, silent-failure-hunter,
  type-design-analyzer, etc.) — let the gate commands spawn them; don't invoke the
  agents directly.
- **Not used here:** orchestration skills (no agent/capability/workflow),
  `/email-designer`, storage/upload skills — all belong to deferred pieces
  (F3.4/F7.1).

## Open questions for the owner (resolve at promotion)

1. **FK delete policy (AD2):** `onDelete: SetNull` + app-layer 409 guard
   (recommended) vs. `Restrict` (DB refuses, app surfaces it) vs. plain `String`
   FK per the deep spec. All defensible; the recommendation gives a friendly 409
   UX _and_ DB integrity _and_ the reverse-count.
2. **Attribution placement (AD3):** questionnaire-level `PATCH /:id` (recommended,
   REST-natural) vs. a `demo-clients`-side assign endpoint (avoids touching John's
   `[id]/route.ts` at the cost of a less obvious shape).
3. **Slug input:** admin-typed slug (validated unique) vs. auto-derived from name
   with manual override. Recommend derive-with-override (fewer 409s).
4. **`isActive` semantics:** does an inactive client hide from the attribution
   picker only, or also from the list default filter? Recommend: filterable in the
   list, excluded from the picker by default.

# F0.1 — Foundation scaffolding

> The committable working tracker for **F0.1**, the first feature of **P0
> (Foundations)**. Mirrors the promoted-task model in
> [`development-plan.md`](../development-plan.md#how-features-and-tasks-work):
> indicative scope is broken into PR-sized **tasks** with stable IDs, files,
> dependencies, and status. Work through this doc; flip statuses as PRs land.

## Intent

Stand up the app's territory inside the Sunrise fork — module structure,
app-owned Prisma schema, seed namespace, feature flag, doc namespace, and the
gating/test scaffolding the rest of the build hangs off. **No user-visible
feature**; everything after it depends on this being right.

P0 is deliberately **light**: Sunrise v0.0.1 already closed the pre-fork seams,
so F0.1 is mostly _adding new files in app-owned space_ and _verifying_ existing
seams — not building platform plumbing.

`lib/app/questionnaire/**` module beside them.

## Decisions taken

- **Schema scope = minimal anchor.** F0.1 lays down only `AppQuestionnaire` +
  `AppQuestionnaireVersion`; each later phase adds its own models via its own
  migration. Keeps F0.1 PR-sized and honours the "exercisable before the next
  abstraction" principle.
- **Working-doc location = `.context/app/planning/features/f0.1.md`** (a `features/` namespace under `planning/`).
- **Module skeleton = light top-level** (`index.ts` + `types.ts` + a server-only
  flag helper); sub-modules are created by their owning phase when first needed.

## Two corrections to the plan's indicative tasks (intent over prescription)

1. **Healthcheck path drops the underscore.** The indicative
   `app/api/v1/app/_healthcheck/route.ts` would 404 unconditionally — Next.js App
   Router treats `_`-prefixed folders as **private** (excluded from routing). Use
   `app/api/v1/app/healthcheck/route.ts`.
2. **Defer the plain-String User FK.** The CUSTOMIZATION §5 recipe (scalar
   `createdBy String?` + hand-written migration FK with `onDelete`) creates a
   **Prisma-unmodelled DB object**. Per `scripts/db/check-drift.ts`, such objects
   are silently `DROP`-ped by future `prisma migrate dev` runs and are not covered
   by the platform-owned drift inventory. Nothing in F0.1 needs ownership yet, so
   the anchor models use only an app-internal `@relation` (`onDelete: Cascade`),
   and the User-FK recipe + drift-guard interaction is recorded as the first
   `upstream-gaps.md` entry. First real use lands in **F2.1** (authoring). (`User`
   maps to table `"user"`, lowercase, for that day.)

### T0.1.2 — Module skeleton + feature-flag wrapper + gate helper

Light top-level module; inherits the `lib/app/**` ESLint boundary automatically.

- `lib/app/questionnaire/types.ts` — foundational type aliases (e.g. the
  questionnaire/version `status` string-literal unions used by the anchor models).
  Non-empty, real.
- `lib/app/questionnaire/feature-flag.ts` (server-only at runtime):
  - `export const APP_QUESTIONNAIRES_FLAG = 'APP_QUESTIONNAIRES_ENABLED'`
  - `isQuestionnairesEnabled(): Promise<boolean>` — wraps
    `isFeatureEnabled(APP_QUESTIONNAIRES_FLAG)` from `@/lib/feature-flags`.
  - `ensureQuestionnairesEnabled(): Promise<Response | null>` — the **gating
    template**: returns a 404 `errorResponse` when the flag is off, else `null`.
    Every later app route calls this first.
  - Boundary note: importing `@/lib/feature-flags` is allowed (a `@/` alias, not a
    banned `prisma`/`next`/node specifier). Helper is server-only because
    `isFeatureEnabled` transitively hits Prisma — only called from route handlers.
- `lib/app/questionnaire/index.ts` — barrel re-exporting the flag helpers + types.
- Test: `tests/unit/lib/app/questionnaire/feature-flag.test.ts` — mock
  `@/lib/feature-flags`; assert `isQuestionnairesEnabled` true/false and that
  `ensureQuestionnairesEnabled` returns a 404 Response when off / `null` when on.

Reuse: `isFeatureEnabled` (`lib/feature-flags/index.ts`), `errorResponse`
(`lib/api/responses.ts`).

### T0.1.3 — App Prisma schema + initial migration + schema test

- Add to a **new** `prisma/schema/app-questionnaire.prisma` (a dedicated file —
  `app.prisma` already holds _platform_ models, so keep app models separate):
  - `AppQuestionnaire` — `id`, `title`, `status` (`@default("draft")`), timestamps,
    `versions AppQuestionnaireVersion[]`, `@@map("app_questionnaire")`. **No User FK
    yet** (Correction #2).
  - `AppQuestionnaireVersion` — `id`, `questionnaireId`, `versionNumber`
    (`@default(1)`), `status`, timestamps, a `@relation(... onDelete: Cascade)` back
    to `AppQuestionnaire`, `@@unique([questionnaireId, versionNumber])`,
    `@@index([questionnaireId])`, `@@map("app_questionnaire_version")`.
  - Everything is Prisma-modeled ⇒ no new unmodelled objects, clean drift.
- Generate: `npm run db:migrate:dev -- --name app_questionnaire_init`.
- **Schema-fold footgun (critical):** review the generated `migration.sql`. It must
  contain only `CREATE TABLE "app_questionnaire*"` (+ the relation FK + indexes). If
  Prisma emitted any `DROP` of platform unmodelled objects (vector/HNSW indexes,
  `searchVector` GENERATED column, CHECK constraints, partial unique indexes — see
  `scripts/db/check-drift.ts`), **delete those `DROP` lines** before committing.
- Verify: `npm run db:drift-check` → all 9 platform probes still pass.
- Fill `.context/app/questionnaire/schema.md`: the two models, the migrate-dev
  footgun procedure, the deferred-User-FK note.
- Test: `tests/unit/prisma/app-questionnaire-schema.test.ts` — assert via
  `Prisma.dmmf` (no DB needed) that both models exist with expected
  fields/`@@index`/`@@unique`/relation `onDelete: Cascade`/`@@map` names. (Live FK
  integrity is covered by `db:drift-check` + CI's migrate-deploy job, so no
  `information_schema` test is needed at P0.)

### T0.1.5 — Gated healthcheck route + integration test

- `app/api/v1/app/healthcheck/route.ts` — `GET`:
  ```ts
  const blocked = await ensureQuestionnairesEnabled();
  if (blocked) return blocked; // 404 when flag off
  return successResponse({ status: 'ok' }); // 200 when on
  ```
  Unauthenticated (liveness probe). Inherits the 100/min `api` rate-limit cap
  automatically. Sets the **gating template** every later app route follows
  (flag-gate first; later routes then add `withAuth`/`withAdminAuth`).
- Test: `tests/integration/api/v1/app/healthcheck/route.test.ts` — mock the flag
  both ways → 404 (off) and 200 (on). (Repo "integration" tests mock Prisma; this
  matches the existing pattern.)

Reuse: `successResponse`/`errorResponse` (`lib/api/responses.ts`),
`ensureQuestionnairesEnabled` (T0.1.2).

## Verification (end-to-end)

1. **Static:** `npm run validate` — type-check, ESLint (proves the new
   `lib/app/questionnaire/**` files respect the boundary), Prettier.
2. **Unit/integration:** `npm test` — flag wrapper + gate, schema-shape (DMMF),
   seed unit, healthcheck 404/200 integration.
3. **Database (dev Postgres):**
   - `npm run db:migrate:dev` (or `:deploy`) applies `app_questionnaire_init`
     cleanly against a Sunrise v0.0.1 DB.
   - `npm run db:drift-check` → all platform probes pass (no unmodelled object
     dropped by the schema fold).
   - `npm run db:seed` → the `APP_QUESTIONNAIRES_ENABLED` row exists, `enabled=false`;
     re-running is a skip.
4. **Manual route behaviour:** `curl -i localhost:3000/api/v1/app/healthcheck` →
   `404` while the flag is off; toggle the flag on (admin UI or `toggleFlag`) →
   `200` `{ success: true, data: { status: "ok" } }`.
5. **Route registration sanity (optional):** `mcp__next-devtools__init`, then inspect
   routes to confirm `/api/v1/app/healthcheck` is registered (and that the underscore
   variant would not be).

# F10.8 — Re-run a respondent report against a real session

> Committable tracker for **F10.8**. An admin can look up a real completed session and **re-run** its
> respondent report against the respondent's actual captured answers, optionally with **new
> instructions/settings**. Every re-run is a retained, async-generated **revision**; any completed
> revision can be **promoted** to replace the delivered report the respondent sees.

## Intent

- The only ways to (re)generate a respondent report were: the submit-time enqueue (version config, once),
  the respondent-facing `retry` (regenerates from the version config, no-ops a `ready` report, no
  override), and the admin **config preview** (F10.7 — real generation core but AI-**synthesised** sample
  answers, never a real session). None let an admin say "the delivered report for _this_ respondent reads
  wrong — re-run it with different instructions", nor kept a history of attempts.
- The building blocks already existed (ref → session resolver, the settings-parameterised generation
  core via F10.7's `generateReportFromInputs`, the admin session viewer); F10.8 wires them into a real
  re-run.

## What shipped

**Data model (migration `20260716152300_respondent_report_revision`):**

- New `AppRespondentReportRevision` (`prisma/schema/app-questionnaire.prisma`) — append-only re-run log,
  FK to `AppRespondentReport` (`onDelete: Cascade` → follows the session + GDPR erasure). Carries
  `settingsSnapshot` (the exact config that re-run used), `instructions` (admin note), its own
  `status` + lease (`lockedBy`/`lockedAt`) for async generation, `content`/`formatted`/`completionPct`/
  `costUsd`, `authoredBy` (`admin`), `createdBy` (plain String admin id, UG-1). `@@unique([reportId,
revisionNumber])` — monotonic per report; `@@index([status, lockedAt])` for the worker claim.
- `AppRespondentReport.deliveredRevisionId` — the revision last promoted into the delivered report (null
  = original submit-time generation).
- New `RESPONDENT_REPORT_AUTHORS` (`admin | ai`) + `RESPONDENT_REPORT_RERUN_NOTE_MAX_LENGTH` in
  `types.ts`.

**Generation seam:**

- `generateRespondentReport(sessionId)` is now a thin wrapper that resolves the version config, then
  delegates to the new exported `generateRespondentReportWithSettings(sessionId, settings)` — the
  settings-parameterised core the re-run calls with an overridden config. Zero behaviour change to the
  delivered-report path.

**Re-run orchestration (`lib/app/questionnaire/report/revision.ts`):**

- `enqueueRespondentReportRevision` — ensures the 1:1 header exists (created **`ready` with null
  content**, never `queued`, so the delivered-report worker never claims a header that only hosts
  re-runs), then appends the next `queued` revision with its settings snapshot + note.
- `promoteRespondentReportRevision` — copies a `ready` revision's content onto the delivered report and
  sets `deliveredRevisionId` (clears any pending notify email + lease); no-op when the revision isn't
  ready.
- `getRespondentReportRevisionsView` / `getRespondentReportRevisionDetail` — the admin history (delivered
  summary + revisions, marking which is delivered) and one revision's full content for the viewer.

**Worker:** `processQueuedReportRevisions()` (`report/worker.ts`) — a second drain in the maintenance
tick (task `respondentReportRevisions`, `run-tick.ts`) with the same lease/batch/budget discipline as the
delivered-report worker; generates each revision via `generateRespondentReportWithSettings` with its
snapshot settings and writes the result onto the **revision**.

**Admin API (all admin-only, master + respondent-report flag gated via `withRespondentReportEnabled`):**

- `POST/GET /api/v1/app/questionnaire-sessions/:id/report/revisions` — enqueue (`{ config?,
instructions? }`, 202, per-admin `reportRerunLimiter` 10/min; falls back to the version config; rejects
  `raw`) / list history.
- `GET …/report/revisions/:rev` — one revision's content.
- `POST …/report/revisions/:rev/promote` — promote to delivered (409 when not ready).

**Admin UI:** `SessionReportRerun`
(`components/admin/questionnaires/sessions/session-report-rerun.tsx`) — a dialog opened from the session
viewer's action bar (rendered only when `flags.respondentReport`). Seeds from the version config, edits
the key generation fields + a note, re-runs, polls status, and lets the admin view any completed re-run
and promote it. Reuses the shared `ReportBody` / `ReportPaperHeader` paper renderer.

## Decisions

- **Additive revisions, not a refactor of the delivered report** — the delivered `AppRespondentReport`
  read-path (respondent GET, worker, view, PDF) is untouched; re-runs live in a sibling table. Lowest
  blast radius, and the respondent never sees an experimental re-run until an admin promotes it.
- **Async via the worker, not synchronous** — a re-run is the same costly/slow sub-flow as a delivered
  report (one reasoning call + optional web-search rounds + KB grounding), so it queues a revision the
  maintenance worker drains, mirroring how delivered reports generate. A separate drain so a delivered
  backlog and a re-run backlog never starve each other within a tick.
- **Header for a never-reported session is created `ready`, not `queued`** — so the delivered-report
  worker never picks up a header that exists only to host admin re-runs (which would burn an LLM call
  generating an unwanted delivered report from the version config). Safe because the respondent view only
  reads the header when the version config enables an AI-mode report, in which case the submit-time
  enqueue already created the header.
- **Promote is explicit** — an odd re-run must not silently replace the respondent's delivered report;
  the admin opts in per revision.
- **Re-run is AI-mode only** — a `raw` config has nothing to generate (rejected at the route), matching
  the config-preview constraint.

## Tests

- `revision.test.ts` — inert `ready` header creation; monotonic revision numbering + note trim/null;
  promote copies a `ready` revision (+ `deliveredRevisionId`) and no-ops otherwise; view marks the
  delivered revision; detail read.
- `worker-revisions.test.ts` — claim/lease, generate with the **snapshot** settings, ready/failed writes,
  claim-race loss.
- `report-revisions-route.test.ts` — gate order (both flags → auth → rate-limit → validation → mode →
  session), 202 enqueue, version-config fallback, raw rejection, not-found, 429, GET history.
- `report-revision-promote-route.test.ts` — auth, promote success, 409 not-ready, invalid-rev 400.
- Existing delivered-report suite (worker/generate/enqueue/retry) unchanged and green — the seam split is
  behaviour-preserving.

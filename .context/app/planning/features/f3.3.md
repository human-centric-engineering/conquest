# F3.3 ΓÇõ Pre-launch cost estimation

> Committable tracker for **F3.3**, the **third feature of P3 (Configuration,
> invitations, cost estimation)**. Surfaces a heuristic estimate of the LLM spend of
> running respondents through a questionnaire version ΓÇõ per-session and
> per-questionnaire ΓÇõ in the config editor and on the invitations page, so an admin
> can size the cost budget before launch. Read-only, API-first, gated by
> `APP_QUESTIONNAIRES_ENABLED` (404 when off).

## Intent

F3.1 lets an admin set a per-session `costBudgetUsd` cap but **blind** ΓÇõ nothing
showed expected spend (the cost-budget `FieldHelp` literally said "pre-launch
estimation is a later feature"). F3.3 is that feature: a read-only projection of the
cost of one respondent's conversational session, scaled to an expected respondent
count, shown where the admin configures and where they invite. It reads the F3.1
config and prices through Sunrise's provider-agnostic model registry.

## Decisions (at planning ΓÇõ confirmed)

- **Heuristic-only ΓÇõ there is no empirical mode.** The session/turn engine (P4/P6)
  does not exist yet, so there are **zero real session runs** to calibrate against.
  `basedOn` is always `'heuristic'`; the copy says so and the range is wide. An
  empirical mode keyed on `AppQuestionnaireTurn` actuals is a future PR once P6 logs
  them (the pattern Sunrise's `workflow-cost.ts` already uses).
- **History-aware token model.** A session is modelled as one turn per asked
  question; the dominant cost driver is the **conversation history** replayed each
  turn, so input tokens grow **quadratically** in `Q` (`HISTORY ├ù Q(QΓêÈ1)/2`). A flat
  per-question model would under-count long questionnaires. Constants
  (`SYSTEM_PROMPT_TOKENS=1500`, `HISTORY_TOKENS_PER_PRIOR_TURN=250`,
  `OUTPUT_TOKENS_PER_TURN=400`, range `├ù0.5/├ù1.7`) are conservative and tunable, not
  measured.
- **`Q` honours the F3.1 caps.** Effective questions per session =
  `min(questionCount, maxQuestionsPerSession ?? questionCount)`, floored by
  `min(minQuestionsAnswered, questionCount)`.
- **Provider-agnostic pricing; unknown-price ΓÊá free.** Model resolved via
  `getDefaultModelForTaskOrNull('chat')` (org default), priced via `getModel`. A
  model with no registry price (or a `0` rate) returns `pricingKnown:false` with USD
  withheld and a note ΓÇõ never a misleading `$0.00`. Token volume is still estimated.
- **GET-only, against persisted config.** No POST/unsaved-config variant ΓÇõ the
  estimate reads the saved version + config; the UI re-fetches after a save. Pure
  math, no LLM call, no audit, no rate-limit sub-cap (inherits the section 100/min).
- **Two UI surfaces, one fetch each.** Config editor (full card + respondents input
  - over-budget note) and the invitations page (compact banner). Both fetch once at
    `respondents=1` and scale `perQuestionnaire = perSession ├ù respondents`
    client-side, so changing the respondent count never refetches.

## Build shape

### 1. Pure core (`lib/app/questionnaire/cost-estimation/`, Prisma-free)

New sub-module (barrel `index.ts`, `types.ts`, `estimate.ts`) re-exported from
`lib/app/questionnaire/index.ts`. `types.ts`: the heuristic constants + `CostRange`,
`CostEstimateAssumptions`, `SessionCostEstimate`. `estimate.ts`:
`effectiveQuestionsPerSession(...)` and `estimateSessionCost(input)` ΓÇõ numeric
inputs only (the route does all I/O), returns the full estimate.

### 2. Route ΓÇõ `GET ΓÇª/versions/:vid/cost-estimate`

`cost-estimate/route.ts`, GET only. Order: flag-gate ΓÁÈ `withAdminAuth` ΓÁÈ
`loadScopedVersion` (scope-404) ΓÁÈ validate `respondents` query ΓÁÈ `Promise.all`(load
slots' prompts, config caps, default chat model) ΓÁÈ `getModel` pricing +
`╬ú estimateTokens(slot.prompt)` ΓÁÈ `estimateSessionCost` ΓÁÈ `successResponse`.
Registered as `versionCostEstimate(id, vid)` in `lib/api/endpoints.ts`.

### 3. UI

Shared `components/admin/questionnaires/cost-estimate-card.tsx` (`variant`
`'card' | 'banner'`). Wired into `config-editor.tsx` (under Budget & caps; the stale
"later feature" `FieldHelp` clause replaced; `reloadKey` from the saved cap/floor;
live `costBudgetUsd` for the over-budget note) and the invitations page
(`variant="banner"` above the invite form, against the launched version).

## Tests

- **Unit** (`tests/unit/lib/app/questionnaire/cost-estimation/estimate.test.ts`):
  `effectiveQuestionsPerSession` cap/floor/bounds; ordered low<mid<high; linear
  respondent scaling; super-linear (quadratic) input-token growth vs question count;
  cap applied to asked count; pricing-unknown (null rate **and** `0` rate) withholds
  USD; zero-question all-zero + note; assumptions echo.
- **Integration**
  (`tests/integration/api/v1/app/questionnaires/cost-estimate-routes.test.ts`,
  mirroring `invitations-routes.test.ts`): gate order (404 flag-off before auth),
  401/403, scope-404 (before estimation work), `respondents` validation (reject
  non-int / `0`, default `1`), happy path (asserts `getModel` + per-slot
  `estimateTokens` consulted, response shape, `perQuestionnaire = perSession ├ù
respondents`), default-model fallback, pricing-unknown (undefined model **and** `0`
  rate).

## Not in F3.3

`AppQuestionnaireTurn` costs). Budget **enforcement** ΓÇõ the hard per-session cap is
F6.3; F3.3 only _displays_ the estimate vs the cap. The document-extraction (F1.1)
cost (incurred at ingest, already logged via `logCost`). Demo-client / themed
surfaces (F3.4).

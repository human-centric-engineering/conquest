# F2.3 — Extraction-change review

> Committable tracker for **F2.3**, the third feature of **P2 (Admin CRUD over
> questionnaires)**. The admin-facing **consumer of P1's extraction-change log**:
> list every `AppQuestionnaireExtractionChange` the extractor recorded during
> ingestion (F1.1) — with source quote, before/after, rationale — and **revert**
> any of them. It's the trust surface for an opinionated extractor: the admin sees
> exactly what the model changed and can undo it. Gated by
> `APP_QUESTIONNAIRES_ENABLED` (404 when off), API-first, riding the F2.1
> mutation pipeline (including version-fork-on-launched).

## Intent

The extractor is **opinionated, not literal** — it prunes boilerplate, fixes
typos, rewrites terse prompts, infers types/goal/audience, merges duplicates,
splits compound questions. F1.1 records each decision as one change row;
F2.3 surfaces that log and lets the admin revert. A **verbatim question produces
no record** — the log is a diff against the source, not a transcript. No
migration: the model and all revert columns (`status`, `revertedAt`,
`revertedByUserId`) already exist from F1.1.

## Decisions (at planning)

- **Full revert across all change types** — not just the tractable ones. The
  user chose the ambitious path over a list-only surface or a tractable-types-only
  revert.
- **UI = a dedicated sub-route** `app/admin/questionnaires/[id]/extraction-changes?v=<versionId>`,
  not a tab on the detail page. Keeps the busy detail page clean; the existing
  per-version `changeCount` becomes the link in.
- **Fail cleanly, never guess-and-corrupt.** `beforeJson`/`afterJson` are
  `z.unknown()` free-form LLM output and `targetEntityId` is **null by design**
  for every section/question change (only `infer_*` carry a target — the version).
  So revert of an editorial edit must _reconcile_ the target against the live
  graph; any ambiguity returns a **typed `RevertImpossible` reason → 422**, never a
  silent wrong mutation. This is the posture `extraction-changes.md` already
  commits to.
- **Block, don't cascade** (dependency ordering). The reconciliation _is_ the
  guard: if a later edit changed the entity, `afterJson` no longer matches and the
  planner returns `target_not_found`/`graph_drift`. Admin reverts newest-first
  (UI orders that way); no automatic cascade-revert of dependents.
- **Fork-on-launched marks the source row.** Forks deliberately don't copy change
  records (`_lib/fork.ts` — "a fork starts a clean editorial lineage"). So a revert
  on a launched version applies the inverse mutation to the forked **draft** and
  marks the **source** version's change row `reverted`; the UI redirects to the
  draft (empty log) with a lineage banner. The common path — reverting on the
  just-ingested `draft` — needs no fork and flips the row in place. The planner is
  **dry-run before forking** so a doomed revert leaves no orphan draft.

## Revert semantics by confidence

| Confidence                   | Change types                                                                              | Revert behavior                                                                                                                                                                                                                                               |
| ---------------------------- | ----------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Faithful**                 | `infer_goal`, `infer_audience`                                                            | Clear the version field(s) the inference set (`infer_audience` clears only the inferred key subset). Drift guard: refuse if the admin re-supplied the field since (provenance no longer `inferred`, or value changed).                                        |
| **Faithful**                 | `prune_section`, `prune_question`                                                         | Re-create from `beforeJson` (the only surviving copy; `afterJson` null). Append at end (ordinal max+1); fresh keys via `nextAvailableKey`. Question → resolve parent section by `sectionTitle`/`sectionOrdinal`, else first section, else `target_not_found`. |
| **Best-effort, can fail**    | `rewrite_prompt`, `correct_spelling`, `correct_grammar`, `infer_type`, `augment_question` | Reconcile the (null-id) target by matching `afterJson` against current entities, tiebreak on `sourceQuote`. Exactly-one → restore touched fields from `beforeJson`. Zero → `target_not_found`; ≥2 → `ambiguous_target`.                                       |
| **Structural, default fail** | `merge_questions`, `split_question`, `add_section`                                        | No faithful inverse from free-form JSON. Narrow best-effort only when `beforeJson` carries full question object(s) AND all products reconcile uniquely; else `structural_inverse_unavailable` / `graph_drift`. Never a half-restored graph.                   |

Idempotency: only `status='applied'` is revertible — re-revert → **409**.

## Build shape

- **Pure planner** (`lib/app/questionnaire/extraction-review/planner.ts`,
  Prisma-free): `planRevert(change, graphSnapshot) → RevertPlan | RevertImpossible`
  - `reconcileTarget()`. `RevertImpossibleReason` =
    `target_not_found | ambiguous_target | missing_before_json | structural_inverse_unavailable | graph_drift`.
    The unit-test centerpiece.
- **DB executor** (`app/api/v1/app/questionnaires/_lib/extraction-review-routes.ts`):
  `loadScopedChange`, `buildGraphSnapshot`, `listVersionChanges` (enriches each row
  with `revertable`/`revertBlockedReason` via a dry `planRevert`), `executeRevert`.
- **Routes**: `GET …/versions/:vid/changes` (filters: status, changeType,
  targetEntityType) + `POST …/versions/:vid/changes/:changeId/revert`. Wrapper
  order `withQuestionnairesEnabled(withAdminAuth(...))`. Audited as
  `questionnaire_change.revert`. Registered under `API.APP.QUESTIONNAIRES`
  (`versionChanges`, `versionChangeRevert`).
- **UI**: sub-route page (server component, flag-gate + `serverFetch`) +
  `extraction-changes-table.tsx` (grouped by type, client-side filters, before/after
  JSON, per-applied-row revert with disabled+tooltip when not revertable, confirm
  dialog, fork-redirect). Detail page wraps the per-version `changeCount` in a link.

## Sequencing — 2 PRs

Full revert is a high-stakes graph mutation with structural-inverse hazards, so it
gets its own review pass + `/security-review` (per the gate-suite-tax calibration;
collapsible to 1 PR at the cost of a larger single review).

- **PR-A — Read surface**: lib views/schemas + `buildGraphSnapshot`/`listVersionChanges`,
  `GET …/changes`, `endpoints.ts`, sub-route page + read-only table + detail link.
  Tests: schema unit + integration (gate, scope-404, list, filters). Docs + this tracker.
- **PR-B — Revert engine**: `planner.ts` (+ wire `revertable` enrichment),
  `executeRevert`, `POST …/revert` (fork interaction, 409, 422-before-fork), revert UI.
  Tests: full planner unit suite (every type incl. impossible/ambiguous/drift) +
  integration (each revert path, re-revert 409, fork remap, audit). Flip
  `extraction-changes.md` "Revert (F2.3, preview)" → shipped.

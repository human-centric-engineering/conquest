# F5.3 — Suggestion review

> Committable tracker for **F5.3**, the final feature of P5 (Design-time evaluation). Turns
> F5.2's persisted findings into an interactive **review queue**: the admin works through each
> judge suggestion and accepts, declines, edits, or **applies** it to the draft version (forking a
> launched version first, like every authoring edit). Obsolete suggestions are derived **stale** at
> read time. Gated by `APP_QUESTIONNAIRES_ENABLED` + the
> `APP_QUESTIONNAIRES_DESIGN_EVALUATION_ENABLED` sub-flag.

## Intent

F5.2 made a run a durable artefact; F5.3 makes the findings _actionable_. The phase is complete
when an admin can request an evaluation, see suggestions per judge, review/edit/apply them, and
watch the version evolve.

The load-bearing design choice: the quality ceiling is whether a suggestion arrives **already
actionable**. A queue bolted onto prose-only findings could only ever deep-link an editor to retype
the fix. So F5.3 went _upstream_ into the F5.1 findings contract and gave each finding an optional
**structured, executable edit** — enabling genuine one-click apply through the existing
fork-if-launched seam, with a human always confirming.

## Decisions (confirmed with the user)

- **Build the best end state, not the smallest PR.** The user explicitly chose quality over PR size;
  hence the upstream change to F5.1's contract rather than a UI-only layer.
- **`proposedEdit` is an accelerator, never a trust boundary.** It is prompt-guided, _not_
  provider-enforced — `runStructuredCompletion` sends plain prompt + Zod parse, never the JSON
  schema (verified). So it is optional (prose-only stays valid), soft-degraded to `null` on malform
  at persist (`coerceProposedEdit`), and **re-validated at apply time exactly like a hand authoring
  edit** (`validateTypeConfig`, target resolution, ordinal bounds).
- **Reuse the authoring leaf helpers, not the F2.1 HTTP handlers** (the zero-blast-radius path from
  the plan's R2). The apply engine calls `validateTypeConfig` / `forkVersionIfLaunched` / the
  provenance stamps directly; the shipped routes are untouched. Only the `targetKey`→entity
  resolution is apply-specific (a finding addresses by key/title, not a URL id).
- **`accept` and `apply` stay separate.** Accept is triage (agree, not yet written); apply mutates
  and may fork. Keeping them distinct lets an admin agree across a run, then apply against one
  consistent fork lineage.
- **`stale` is derived, never stored.** A stored flag would rot the instant the structure changed.
  The schema comment in F5.2 that listed `stale` as a status value is corrected to
  `pending | accepted | declined | applied`.
- **Precise staleness via a per-run structure snapshot.** `AppQuestionnaireEvaluationRun.structureSnapshot`
  captures the `VersionStructureInput` the judges read (free — the POST already builds it); the
  deriver diffs only the **targeted slice** vs live, so an unrelated edit never falsely stales a
  finding. The cheap `updatedAt` heuristic (over-flags) was rejected.
- **Per-run queue**, not a cross-run aggregate. A cross-run "open findings for this version" rollup
  is a clean follow-up once per-run review is proven.
- **Extend the finding row**, not a separate `AppQuestionnaireSuggestionReview` table. The dev-plan's
  older "row per finding" line is superseded by F5.2's newer decision to add review state to the
  finding row (additive, nullable columns — no second migration's worth of churn).

  told to attach an op **only** when it maps cleanly, else prose-only.

`20260605213442_app_evaluation_finding_review` (create-only, hand-trimmed of the phantom platform /

- The run-detail admin component is now the interactive queue: accept/decline/edit/apply per derived
  `applicable` + `stale`, a typed edit-override mini-form, a status filter, and a fork banner.

## Known limitations (designed-for)

- ~~**`add_question` and prose-only findings are not auto-applied**~~ — **resolved 2026-06-14**, see
  [§ Follow-up](#follow-up--add_question-apply--editor-pre-fill-2026-06-14). `add_question` is now a
  one-click "Add to questionnaire" (apply engine) **and** a pre-filled editor refine path. Prose-only
  (`manual`) findings still route to the editor (nothing to pre-fill — author by hand).
- **Cross-fork staleness is best-effort.** Read-time staleness for a launched version is computed
  against the frozen original (always not-stale); the apply-time guard catches real `target_gone`
  conflicts on the draft, but content-staleness on the evolving fork isn't re-derived (the plan's R3
  limitation). The fork-lineage convergence rule keeps repeated applies on one draft.
- **No bulk apply** (conflicting findings / cascading forks can't honour the per-write staleness
  re-check). Bulk decline was scoped out too; per-finding decline covers it.
- **`reorder` sets a single ordinal** (mirrors the question route's move semantics — tolerates ordinal
  gaps/collisions, best-effort ordering). It is the weakest op; the prompt asks judges to use it
  sparingly.

## Follow-up — `add_question` apply + editor pre-fill (2026-06-14)

The original F5.3 shipped the `add_question` deep-link as a bare link to the structure editor with
**no finding context** (`editorHref` carried no params, the editor accepted no prefill), so pressing
"Open in editor" landed on a blank editor and the suggested question never appeared — and there was
no apply path at all (the engine returned `needs_authoring`). "Accept" (triage-only) read like the
do-it button, compounding the confusion. This follow-up makes the suggested-question flow
functionally complete, the way the docstrings always described it.

**Decision (confirmed with the user):** one-click add **+** a refine path, with Accept/Dismiss
demoted to quiet secondary triage so the work-action is unambiguous.

- **Apply engine** (`_lib/evaluation-apply.ts`) — `applyAddQuestion`: `add_question` is now a real
  apply. Resolves the target **section by title** (`op.sectionKey` → finding's `section:<title>` →
  last section; `target_gone`/`op_invalid`/`needs_authoring` reasons), validates+**defaults** the
  drafted `typeConfig` (a bare choice/scale type lands placeholder options to refine), derives the
  `key`, and creates the slot + stamps the finding applied in one transaction. Same fork-lineage
  convergence + validate-before-fork discipline as the in-place ops.
- **Staleness** (`_lib/evaluation-staleness.ts`) — `add_question` is stale only when a **named
  section target** is gone/ambiguous; a goal-targeted add (or one whose section still resolves) stays
  applyable (goal-text edits no longer falsely stale it). `applicable` stays `deep-link`.
- **`mark_applied` review action** (`review-schema.ts` + finding `PATCH` route) — records a finding
  applied against a validated draft version **without** mutating structure, for the editor refine
  path (the editor already authored the question).
- **Card** (`evaluation-finding-review.tsx`) — primary "Add to questionnaire" (apply) + secondary
  "Open in editor" (seeded deep-link) for `add_question`; "Apply" (+Edit) for other ops; "Open in
  editor" for prose-only; **Accept / Dismiss** demoted to quiet ghost triage. Adds a drafted-question
  preview block.
- **Draft quality (keys + type)** — follow-on from first review: drafted questions were getting
  whole-sentence keys (`how_would_you_describe_your_current_morale_at_work`) and a wrong type
  (likert for an open-ended question). Fixes: `slugifyKey` now drops grammatical stopwords and keeps
  the first ~4 content words (concise keys across **all** paths — extraction, hand-authoring, data
  slots, apply); the judge's `add_question` op gained an optional concise **`key`** (schema + type +
  apply, slugified + collision-suffixed); and the coverage judge prompt now instructs concise
  `snake_case` keys + fitting the `type` to the answer (free_text for open-ended, likert only for
  fixed scales) rather than defaulting to likert.
- **Editor pre-fill** — `EvaluationSeed` view type; `getEvaluationAddQuestionSeed` resolves the
  `?seedFinding=<runId>:<findingId>` deep-link (flag-gated); the structure page forces edit mode and
  renders `EvaluationSeedComposer` (pre-filled prompt/type/section/guidelines) which creates via the
  authoring route, marks the finding applied (against the possibly-forked draft), and navigates with
  the seed cleared.
- **Data-slot assignment (orphaned questions)** — follow-on: a question added after data slots were
  generated is covered by no slot. New `app_assign_data_slots` capability (reuses the data-slots
  generator agent) + `POST …/data-slots/assign` route: places each orphan into an existing slot or a
  new one (LLM returns _placements_; deterministic `mergeAssignments` writes live via
  `replaceDataSlots`, fork-if-launched, existing slots preserved). Surfaced as a pre-ticked checkbox
  on both add paths (finding card one-click + seed composer, shown only when the version has slots)
  and a catch-all "Assign with AI" button on the Data Slots review surface. Documented in
  `.context/app/questionnaire/data-slots.md` (§ 3b). See seed `032`.
- **Tests** — apply-engine `add_question` (create / typeConfig default / `target_gone` /
  `op_invalid` / no-sections `needs_authoring`); staleness `add_question` branch; run-detail card
  (one-click add, seeded link, stale-disabled, Dismiss); `getEvaluationAddQuestionSeed`
  (override-wins / malformed-ref / terminal / non-add / fetch-fail); `EvaluationSeedComposer`
  (pre-fill, create+mark-applied, fork target, error, discard, empty-prompt disable).

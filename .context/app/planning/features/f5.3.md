# F5.3 — Suggestion review

> Committable tracker for **F5.3**, the final feature of P5 (Design-time evaluation). Turns
> F5.2's persisted findings into an interactive **review queue**: the admin works through each
> judge suggestion and accepts, declines, edits, or **applies** it to the draft version (forking a
> launched version first, like every authoring edit). Obsolete suggestions are derived **stale** at
> read time. Gated by `APP_QUESTIONNAIRES_ENABLED` + the
> `APP_QUESTIONNAIRES_DESIGN_EVALUATION_ENABLED` sub-flag.

## Intent

F5.2 made a run a durable artefact; F5.3 makes the findings _actionable_. The phase is complete
when an admin can request an evaluation, see suggestions per judge, review/edit/apply them, and
watch the version evolve.

The load-bearing design choice: the quality ceiling is whether a suggestion arrives **already
actionable**. A queue bolted onto prose-only findings could only ever deep-link an editor to retype
the fix. So F5.3 went _upstream_ into the F5.1 findings contract and gave each finding an optional
**structured, executable edit** — enabling genuine one-click apply through the existing
fork-if-launched seam, with a human always confirming.

## Decisions (confirmed with the user)

- **Build the best end state, not the smallest PR.** The user explicitly chose quality over PR size;
  hence the upstream change to F5.1's contract rather than a UI-only layer.
- **`proposedEdit` is an accelerator, never a trust boundary.** It is prompt-guided, _not_
  provider-enforced — `runStructuredCompletion` sends plain prompt + Zod parse, never the JSON
  schema (verified). So it is optional (prose-only stays valid), soft-degraded to `null` on malform
  at persist (`coerceProposedEdit`), and **re-validated at apply time exactly like a hand authoring
  edit** (`validateTypeConfig`, target resolution, ordinal bounds).
- **Reuse the authoring leaf helpers, not the F2.1 HTTP handlers** (the zero-blast-radius path from
  the plan's R2). The apply engine calls `validateTypeConfig` / `forkVersionIfLaunched` / the
  provenance stamps directly; the shipped routes are untouched. Only the `targetKey`→entity
  resolution is apply-specific (a finding addresses by key/title, not a URL id).
- **`accept` and `apply` stay separate.** Accept is triage (agree, not yet written); apply mutates
  and may fork. Keeping them distinct lets an admin agree across a run, then apply against one
  consistent fork lineage.
- **`stale` is derived, never stored.** A stored flag would rot the instant the structure changed.
  The schema comment in F5.2 that listed `stale` as a status value is corrected to
  `pending | accepted | declined | applied`.
- **Precise staleness via a per-run structure snapshot.** `AppQuestionnaireEvaluationRun.structureSnapshot`
  captures the `VersionStructureInput` the judges read (free — the POST already builds it); the
  deriver diffs only the **targeted slice** vs live, so an unrelated edit never falsely stales a
  finding. The cheap `updatedAt` heuristic (over-flags) was rejected.
- **Per-run queue**, not a cross-run aggregate. A cross-run "open findings for this version" rollup
  is a clean follow-up once per-run review is proven.
- **Extend the finding row**, not a separate `AppQuestionnaireSuggestionReview` table. The dev-plan's
  older "row per finding" line is superseded by F5.2's newer decision to add review state to the
  finding row (additive, nullable columns — no second migration's worth of churn).

## What shipped

### Pure core (`lib/app/questionnaire/evaluation/`, Prisma-free)

- `types.ts` — `PROPOSED_EDIT_OPS` + the `ProposedEdit` discriminated union; `FINDING_REVIEW_STATUSES`
  (`pending|accepted|declined|applied`) and `FINDING_APPLICABILITIES` (`apply|deep-link|manual`);
  `JudgeFinding.proposedEdit?`.
- `judge-schema.ts` — `proposedEditSchema` (Zod discriminated union, `typeConfig` left `unknown`),
  `coerceProposedEdit`, a compile-time parity guard, the optional field on `judgeFindingSchema`.
- `judge-prompt.ts` — per-dimension `editGuidance` + the OUTPUT exemplar/retry updated; judges are
  told to attach an op **only** when it maps cleanly, else prose-only.
- `review-schema.ts` — `reviewFindingSchema` (accept / decline / edit).

### Schema + migration

`20260605213442_app_evaluation_finding_review` (create-only, hand-trimmed of the phantom platform /
pgvector DROP INDEX statements). Additive nullable columns on the finding (`proposedEdit`,
`editedOverride`, `decidedByUserId`, `decidedAt`, `appliedAt`, `appliedToVersionId`) and the run
(`structureSnapshot`). No backfill.

### Route-local seams (`_lib/`)

- `evaluation-staleness.ts` — `deriveFindingState` / `deriveApplicability` (pure; diffs the targeted
  slice per `targetKey` kind).
- `evaluation-apply.ts` — `applyFinding`: prose/`add_question` → `needs_authoring`; apply-time
  staleness re-check; **fork-lineage convergence** (reuse a prior apply's draft instead of
  re-forking); validate-before-fork; op execution + finding-applied stamp in one transaction.
- `evaluation-run-routes.ts` — persist `structureSnapshot` + per-finding `proposedEdit`; staleness-aware
  `getEvaluationRunDetail`; `loadScopedFinding` / `buildScopedFindingView` (a structure-load failure
  degrades to no-derivation, never 500s a read).

### Routes + UI

- `PATCH …/findings/:findingId` (accept/decline/edit) and `POST …/findings/:findingId/apply`
  (sub-flag gated; apply takes `evaluationApplyLimiter` 60/min; `applied` is terminal → 409;
  unapplicable → 409 with reason). Endpoint helpers in `lib/api/endpoints.ts`.
- The run-detail admin component is now the interactive queue: accept/decline/edit/apply per derived
  `applicable` + `stale`, a typed edit-override mini-form, a status filter, and a fork banner.

### Seeds / docs / tests

- Judge seed `systemInstructions` mirror updated to mention the structured edit.
- `design-evaluation.md` retitled F5.1–F5.3 with a full F5.3 section.
- Tests: `ProposedEdit` schema + `coerceProposedEdit` (judge-schema test); the staleness deriver
  (per `targetKey` kind, including the unrelated-edit-not-stale case); the apply engine (early
  returns, target_gone, in-place draft apply, fork-lineage convergence); the review + apply routes
  (gating, scope, terminal 409, edit override, unapplicable 409, fork meta).

## Known limitations (designed-for)

- **`add_question` and prose-only findings are not auto-applied** — they route to the editor
  (`deep-link` / `manual`). The deep-link is a link to the version editor (`?v=`), not a literal
  field-prefilled create form (the editor doesn't yet accept prefill query params); the suggestion
  text stays visible to copy.
- **Cross-fork staleness is best-effort.** Read-time staleness for a launched version is computed
  against the frozen original (always not-stale); the apply-time guard catches real `target_gone`
  conflicts on the draft, but content-staleness on the evolving fork isn't re-derived (the plan's R3
  limitation). The fork-lineage convergence rule keeps repeated applies on one draft.
- **No bulk apply** (conflicting findings / cascading forks can't honour the per-write staleness
  re-check). Bulk decline was scoped out too; per-finding decline covers it.
- **`reorder` sets a single ordinal** (mirrors the question route's move semantics — tolerates ordinal
  gaps/collisions, best-effort ordering). It is the weakest op; the prompt asks judges to use it
  sparingly.

# F4.3 — Contradiction detection

> Committable tracker for **F4.3**, the consistency-check primitive of P4
> (Conversational engine, non-streaming). Given a respondent's captured answers
> across a version's slots, detect genuine cross-slot logical contradictions and
> **surface them to the agent for confirmation — never auto-overwrite**. Built as a
> pure core + a capability + a no-persistence preview route, mirroring F4.1/F4.2.
> Gated by `APP_QUESTIONNAIRES_ENABLED` + a per-feature sub-flag (404 when off).

## Intent

F4.1 shipped "which question next?"; F4.2 shipped "what did they answer?"; F4.3 is
the consistency check over the accumulating answers — catch the respondent who says
"no children" then "my daughter's at college", so the final dataset is coherent.
Detection only: it produces surfacing findings and stops. Resolution is F4.4, the
session loop + persistence is F4.6. Like F4.2 it persists nothing — a caller
assembles the in-memory context (a test harness now, the engine later), and the
preview route returns findings and writes nothing.

## Decisions (at planning / build — confirmed)

- **Align to the committed schema vocab, not the plan prose.** F3.1 already locked
  `AppQuestionnaireConfig.contradictionMode = off | flag | probe` +
  `contradictionWindowN` (a window _size_). The dev-plan prose listed a different
  axis (`off/every_turn/every_n_turns/sweep_only` — cadence). We keep the committed
  **behaviour** axis (`flag` = surface, `probe` = follow up) and model **cadence** as
  pure logic, no migration. The plan prose is updated to match.
- **Cadence is a pure scheduler now; the cadence config column is deferred to F4.6.**
  `shouldRunDetection(mode, windowN, phase)` covers `phase:'turn'` (= every_turn,
  windowed by `windowN`) and `phase:'completion-sweep'` (= sweep_only) from day one.
  `every_n_turns` (a pure cost knob) waits for F4.6's real turn loop, where it's an
  additive field + a scheduler branch — cheap to add, expensive to guess now.
  **Landed 2026-06-07** (deferred-gaps audit, as predicted): the additive
  `contradictionEveryNTurns` config column + an optional `cadence` arg on
  `shouldRunDetection`, consumed by the F6.1 orchestrator (skips off-boundary turns).
- **Surface, never overwrite.** The capability returns `ContradictionFinding[]`
  (which slots conflict, why, a severity, a `confidence`, and — under `probe` — a
  `suggestedProbe`). It writes nothing. F4.4 acts on a confirmed conflict; the
  `suggestedProbe` is the handoff.
- **Detector-local severity tuple.** `CONTRADICTION_SEVERITIES = [low, medium, high]`
  lives in the core (not the shared `types.ts`), the analogue of
  `EXTRACTOR_EMITTED_PROVENANCES` — detector-internal vocabulary the Zod enum and the
  normaliser clamp derive from.
- **Resilient schema, normaliser enforces semantics.** `slotKeys` is only a non-empty
  array in Zod; the ≥2-distinct-_answered_-slots rule and symmetric-pair dedupe are
  the normaliser's, which drops one odd finding rather than failing the pass (the F4.2
  doctrine). Diverges from the plan's `min(2)` schema note for exactly this resilience.
- **New contradiction-detector agent**, distinct from the answer extractor: its own
  cadence, persona, and monthly budget. Resolves the `chat` tier (snappy).
- **Own opt-in sub-flag** (`APP_QUESTIONNAIRES_CONTRADICTION_DETECTION_ENABLED`, off by
  default) on top of the master flag — detection spends an LLM call per pass, the same
  reasoning as answer extraction's sub-flag.
- **≥2 answers required** — detection is meaningless below two. Enforced at the schema
  (`.min(2)`), the context builder (`insufficient_answers` → 400), and the normaliser.
- **Fail-soft route**: a capability error → 200 with empty findings + a `diagnostic`,
  never a 5xx.

## Build shape

### 1. Pure core (`lib/app/questionnaire/contradiction/`, Prisma-free) — PR1

`types.ts` (`ContradictionContext`, `ContradictionSlotView`, `AnsweredSlotView`,
`ContradictionFinding`, `CONTRADICTION_SEVERITIES`, `DetectionPhase`/`DetectionDecision`),
`detection-schema.ts` (Zod LLM contract + `z.toJSONSchema` + `validateContradictionDetection`),
`detection-prompt.ts` (provider-agnostic prompt — conditional `probe` request — + retry),
`detection-logic.ts` (`normalizeContradictionFindings` — drop/dedupe/mode-shape — and the
pure `shouldRunDetection` scheduler), `index.ts` barrel re-exporting `CONTRADICTION_MODES`.

### 2. Capability + seeds (PR2)

`capabilities/detect-contradictions.ts` (`AppDetectContradictionsCapability`,
`processesPii`, counts-only `redactProvenance`); constants (slug/handler/agent/
function definition); registration in `lib/app/capabilities.ts` + the barrel;
`isContradictionDetectionEnabled()` + flag constant; seeds `009-contradiction-detector-agent`,
`010-contradiction-detection-capability`, `011-contradiction-detection-flag` (disabled).

### 3. Sub-flag + preview route (PR2, bundled)

`contradictionDetectionLimiter` (60/min); `_lib/contradiction-context.ts` (DB seam —
loads slots + the version's `contradictionMode`/`contradictionWindowN`, accepts
hand-supplied answers); `POST …/versions/:vid/detect-contradictions` (flag → auth →
sub-flag → validate → sub-cap → context → dispatch → fail-soft).

## Tests

- **Unit** (`tests/unit/lib/app/questionnaire/contradiction/`, 36): schema round-trip +
  named-path issues + `z.toJSONSchema` shape; normaliser (unknown-slot drop,
  unanswered-slot drop, <2-distinct drop, symmetric dedupe keeps higher confidence,
  severity clamp, flag-strips-probe, probe-without-probe downgrade-keep, three-way,
  empty); scheduler (off → no-run, completion-sweep → run-all, turn+windowN=0 → all,

  16): real dispatcher + `runStructuredCompletion` with a mocked provider — happy path,

  no-silent-failure, provider fail-closed, cost-log + isolation, counts-only redaction.
- **Integration — route** (`tests/integration/api/v1/app/questionnaires/detect-contradictions-routes.test.ts`,

  401/403, scope-404, insufficient-answers 400, body validation, rate-limit 429,
  dispatch wiring (config-defaulted + body-overridden mode/window), fail-soft,
  missing-agent 404.

## Sequencing — 2 PRs (one branch, risk-sequenced)

- **PR1** — pure core + scheduler + unit tests. Zero infra/DB/route risk; the novel
  logic (symmetric dedupe, scheduler, flag/probe shaping) reviewed in isolation.
- **PR2** — capability + agent/capability/flag seeds + registration + sub-flag + preview
  route + context builder + rate-limit + both integration suites + docs. Bundles the
  wiring that only makes sense (and reviews for security) together.

Gates each PR: `/pre-pr`, `/code-review`, `/test-review`; `/security-review` on PR2
(LLM I/O + PII redaction + new authed input surface + LLM cost + respondent PII through
the route).

## Operator note

Contradiction detection is off until **both** `APP_QUESTIONNAIRES_ENABLED` and
`APP_QUESTIONNAIRES_CONTRADICTION_DETECTION_ENABLED` are on; run `db:seed` so the
detector agent + capability exist. With the sub-flag off, the detect-contradictions
route 404s. The preview route defaults `mode`/`windowN` from the version's saved config
but accepts body overrides, so an admin can compare `flag` vs `probe` before launch.

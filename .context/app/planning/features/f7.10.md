# F7.10 — Likert scale labels

> Committable tracker for **F7.10**. In the downloadable report, likert answers rendered as bare
> numbers ("How satisfied are you?" → **1**) — meaningless to a reader. Likert scale-point labels were
> optional in the model and never rendered. This makes every likert answer read as a word, enforces
> labelling at creation/launch, and backfills existing questionnaires.

## Intent

A likert answer in the report (and analytics) shows its **per-point label** ("Neutral"), not a number.
To make _every_ value — not just the endpoints — a word, the scale stores one label per point. A
purely numeric rating with no qualitative meaning is the explicit exception: it uses the `numeric`
type, which needs no labels.

## Decisions

- **Per-point labels, not endpoint anchors.** `likert` `typeConfig` gains `labels: string[]` — one
  non-empty entry per scale point (`labels[i]` ⇒ value `min+i`). Endpoint anchors can't be computed
  into a mid-scale word, so per-point is the only way `3 → "Neutral"` deterministically.
- **Numeric ratings use the `numeric` type.** No "unlabelled likert" — the exception is a type
  choice, keeping `likert` always-labelled.
- **Lenient read, strict write.** The shared `likertConfigSchema` (used by answer validation,
  scoring, distributions, the form reader) stays lenient on `labels` so a pre-backfill/legacy row
  never loses its bounds. A separate `likertWriteConfigSchema` requires complete labels and is used
  only by `validateTypeConfig` (admin edits) and the launch gate. `hasCompleteLikertLabels()` is the
  shared predicate.
- **Enforced at creation and launch.** Extraction + compose prompts now require `labels` for likert
  (and steer numeric ratings to `numeric`). The launch-readiness `scaleLabels` check blocks go-live
  on any unlabelled scale, surfaced in the launch checklist UI and the status-route error.
- **LLM contextual backfill.** `npm run db:backfill:likert-labels` visits each unlabelled likert and
  asks the model to either name every point or declare it numeric (→ reclassify), with a deterministic
  generic word ramp as fallback. Idempotent; `--dry-run` / `--generic` flags.

## Code map

- Schema: `lib/app/questionnaire/authoring/type-config-schema.ts` (read vs write likert schema,
  `hasCompleteLikertLabels`, labelled `defaultTypeConfig`); barrel re-export in `authoring/index.ts`.
- Readers: `lib/app/questionnaire/form/type-config.ts` (`FormLikertConfig.labels`, lenient
  `readLikertConfig` with derived endpoints).
- Render: `lib/app/questionnaire/panel/format-slot-answer.ts` (likert branch — feeds PDF, panel, and
  report transcript); `lib/app/questionnaire/analytics/distributions.ts` (per-point bucket labels).
- Authoring UI: `components/admin/questionnaires/question-editor.tsx` (`BoundsEditor` per-point label
  inputs + numeric-exception FieldHelp).
- Enforcement: `lib/app/questionnaire/ingestion/extraction-prompt.ts`, `ingestion/compose-prompt.ts`;
  `lib/app/questionnaire/launch/readiness.ts` (`scaleLabels` check); `_lib/launchability.ts`;
  status route message; `components/admin/questionnaires/launch-checklist.tsx` + overview page props.
- Backfill: `lib/app/questionnaire/ingestion/likert-labels.ts` (pure: prompt, parse, generic ramp);
  `scripts/migrations/2026-06-25-backfill-likert-labels.ts`; `db:backfill:likert-labels` npm script.
- Seed: `prisma/seeds/app-questionnaire/025-demo-content.ts` likert config → per-point labels.

## Tests

- Unit: `type-config-schema` (write requires/validates labels; read stays lenient;
  `hasCompleteLikertLabels`), `form/type-config` (per-point + legacy + malformed fallback),
  `format-slot-answer` (per-point word, non-1 min, out-of-range/unlabelled fallback),
  `launch/readiness` (`scaleLabels` present-only-with-likert, blocks launch),
  `analytics/distributions` (per-point bucket labels), `ingestion/likert-labels` (generic ramp,
  prompt shape, strict decision parse).

## Not here

- No per-button labels in the respondent form (the scale still shows derived endpoint anchors; the
  richer per-point data is used by the report/analytics). No migration — `typeConfig` is JSON.

# F15.4a — Per-step experience reports

> F15.1 deferred "run-level and per-step reports (extending `ReportScope` with an
> `experience_step` kind) — F15.4". This is the per-step half. The run-level respondent report
> (what the `conclude` path promises) is F15.4b.

An admin can now generate a cohort report over the people who answered **one step** of a journey,
reusing the entire existing pipeline — dataset, agent, revisions, publish, PDF — unchanged.

## Why per step and never per experience

This is the load-bearing constraint of the whole phase, and it is not a simplification.

`buildCohortDataset` resolves questions, data slots, profile fields and the scoring schema by a
single `versionId`. `buildDataSlots` joins fills by `dataSlotId` — the row id, not the key. Fills
belonging to a session of a **different** version find no matching bucket and are silently dropped
(`if (!bucket) continue`). `chart-series.ts` makes the same assumption, resolving specs against
question ids stable within one version.

An experience-wide scope spans several versions by definition. It would therefore produce a
confident, well-formatted report over a fraction of the data, with no error and no warning. That
is a worse failure than refusing outright.

A step pins exactly one version, so a step scope still carries one `versionId` and **five modules
needed zero changes**. An experience-wide view has to be a synthesis over ready step reports, which
is separate work.

## Confirmed decisions

- **`AppQuestionnaireSession.experienceStepId` is denormalised from the leg table.**
  `AppExperienceRunLeg.sessionId` is an unmodelled UG-1 pointer, so there is no relation to join
  through and `scopeSessionWhere` — which is pure and returns a plain Prisma `where` — could not
  express "legs of this step". The alternatives were worse: carrying resolved `sessionIds[]` in the
  scope goes stale, breaks the module's purity contract and puts thousands of ids in a query on a
  large cohort; making `scopeSessionWhere` async breaks the client-safe contract it documents.
  Same posture as the `roundId` / `cohortMemberId` / `cohortSubgroupId` pointers beside it, and for
  the same stated reason: read for access and STATS, never as a graph edge.

- **Filtering on `versionId` alone is NOT equivalent** and would be a silent data-correctness bug:
  it sweeps in every ordinary round and walk-up session on the same questionnaire and reports them
  as part of the journey. Pinned by a test.

- **`stepId` is a REQUIRED parameter of `createSessionForExperienceLeg`.** Optional would let a
  caller silently mint a leg no step report can ever see. Making it required turned the two call
  sites into compile errors, which is the point.

- **`resolveStepVersionId` was extracted from `run-advance.ts` into the shared `steps.ts`.** The
  run path asks "which version does the next leg run?" and the report scope asks "which version did
  these legs run?" — if those two ever disagree, the report resolves its data slots against the
  wrong vocabulary and analyses nothing. One definition, not two.

- **`scopeOwnerCreate` uses an exhaustive `switch`, not a ternary chain.** Adding a fourth scope
  kind must fail to compile rather than silently write a row with every owner key null — which
  would collide with the next such row on all three unique indexes.

- **The opt-in gate is the step version's own `config.cohortReport.enabled`.** Deliberately not a
  new experience-level setting: an author who turned reporting off for a questionnaire has not
  consented to it being generated because that questionnaire was reached through a journey.

- **Reports get their own workspace tab, not a section under Runs.** A run is one respondent's
  journey; a report is every respondent who answered one step. Two different questions.

## What shipped

**Schema** (two hand-stripped migrations): `AppQuestionnaireSession.experienceStepId` (nullable,
indexed, no FK) and `AppCohortReport.experienceStepOwnerId` (nullable-unique, no FK) — the third
owner key in the existing polymorphic pattern.

**Lib**: `experienceStepScope()` + `scopeStepId()` and the `experience_step` arm of
`scopeSessionWhere` / `scopeOwnerWhere` / `scopeOwnerCreate`; `CohortReportView.stepId`.

**API seam**: `_lib/step-report.ts` (`loadStepReportScope`, `isStepReportEnabledForVersion`);
`resolveStepVersionId` moved into `_lib/steps.ts`.

**Routes**: the full seven under `/api/v1/app/experiences/:id/steps/:stepId/cohort-report` —
read/patch, dataset, generate, generate/stream, revisions, publish, export.pdf.

**UI**: a Reports tab with a step selector, reusing `CohortReportPanel` verbatim via a new
`stepReportApi` bundle.

**Tests**: 24 scope tests (all three owners, incl. "exactly one owner key" and "always sets
versionId"), 11 step-report resolver tests, and 19 schema tests including migration-SQL guards
that the phantom pgvector `DROP INDEX` and `DROP DEFAULT` statements stayed stripped.

## Design notes

**Both migrations needed hand-stripping, and one revealed a second trap.** Beyond the five pgvector
indexes, `app_questionnaire_session` carries a raw-SQL PARTIAL UNIQUE INDEX that Prisma cannot
model, and `ai_knowledge_chunk.searchVector` is `GENERATED ALWAYS` — which Prisma reads as a
default it wants to drop. All six phantom statements were removed and every index verified present
afterwards. The schema test now asserts this for both migrations so a future regression fails loudly.

**A `column_default IS NOT NULL` check on a generated column is a false alarm.** Postgres reports
`column_default` as NULL for `GENERATED ALWAYS` columns — the expression lives in
`generation_expression`. Worth knowing before concluding a migration dropped something.

**The mirrored routes compiled while carrying wrong copy.** Mechanically deriving the seven step
routes from the version ones left "version scope" headers and "No version report to publish" error
strings behind. Type-checking cannot catch that; each file was reviewed and corrected.

## Deferred

- **The run-level respondent report** — F15.4b. `AppRespondentReport.sessionId @unique` makes it
  strictly 1:1 with one session, so a report spanning legs needs the same polymorphic treatment
  `AppCohortReport` already has. `generateReportFromInputs` takes assembled inputs, so the existing
  pipeline should be reusable by extending only the input seam.
- **An experience-wide synthesis** over ready step reports — genuinely separate work, and must not
  become a cross-version re-aggregation.
- Per-section AI assist (`refineUrl`) on step reports; only the round owner exposes it today.

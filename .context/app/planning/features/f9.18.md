# F9.18 — Paraphrased, living free-text comment fields

> Free-text answers were captured **verbatim** — the extractor copied the respondent's raw chat words
> into `AppAnswerSlot.value` and the panel rendered them as-is (chat ref 53ZF-A0AG: a "comments to
> support your scores" field stuffed with a rambling message). This makes free-text comment fields
> **paraphrased** (the respondent's account, with notable wording in quotes) and **living** (updated as
> the conversation builds), reusing the data-slot fill machinery rather than inventing a new one.

## Confirmed decisions

- **Scope:** all `free_text` answers are paraphrased by default (not verbatim).
- **Quotes:** the paraphrase embeds only **significant/impactful** verbatim in quotes ("we don't use
  it", "ad hoc") — not whole sentences.
- **Classification at extraction:** each free-text question is classified `isolated` vs `section` by the
  LLM during document extraction (admin-overridable later); a `section` field also synthesises from the
  section's data slots. Either way the paraphrase is living and fed by tangential chat mentions.
- The raw words are not lost — `value` keeps the direct answer and the full message stays in the turn
  transcript; the **panel shows the paraphrase**.

## What shipped

- **Schema:** `AppAnswerSlot.paraphrase String?` (migration `…_app_answer_slot_paraphrase`, stripped of
  the phantom pgvector DDL; 5 vector indexes verified intact).
- **Classification:** `FREE_TEXT_COMMENT_AGGREGATIONS` + `readCommentAggregation` (`types.ts`), stored
  in free-text `typeConfig.commentAggregation` (default `isolated`, no column). The document-extraction
  prompt (`ingestion/extraction-prompt.ts`) sets it per free-text question.
- **Extraction:** a free-text **paraphrase rule** in the answer-extractor `SYSTEM_RULES`
  (`extraction/extraction-prompt.ts`) — reported account, significant verbatim in quotes, RE-SCAN/accumulate,
  section-synthesis; `paraphrase` added to `extractedAnswerSchema`, `AnswerSlotIntent`,
  `NewAnswerIntent` and threaded through `normalizeAnswerIntents` (free-text only). `describeSlot` shows
  each free-text candidate's aggregation mode + current summary.
- **Living context:** `currentParaphrase` flows DB → `turn-context` (joined from session answers) →
  `CapabilitySlotView` → capability args → `ExtractionSlotView` → prompt, so the extractor builds on the
  prior paraphrase across turns.
- **Persistence:** `upsertAnswerSlot` writes `paraphrase` (conditionally, so refinement/re-seed writes
  never null a prior one); `turn-run` passes `intent.paraphrase`.
- **Render:** `PanelSlotView.paraphrase` carried through the pure builder + DB loader; the panel item
  shows `paraphrase ?? value` for free-text (`answer-slot-item.tsx`).

## Design notes

- **Reuse, not reinvent.** The data-slot fill system already paraphrases + re-scans every turn; this
  extends that shape to answer slots. No second LLM call — the existing per-turn extractor emits the
  paraphrase alongside the value.
- **Non-destructive.** `value` stays the raw answer; the paraphrase is additive. Respondent-typed form
  answers keep `respondentEdited` and render `value` verbatim.

## Not in scope (follow-ups)

- **Admin override select** for `commentAggregation` in the Structure editor (auto-classification +
  default cover the stated need; admins can't yet flip it in the UI).
- **Re-scanning an already-answered comment field every turn:** the candidate set still excludes answered
  slots, so accumulation runs while a comment field is active/volunteered and via `currentParaphrase`
  when re-extracted — not yet a guaranteed every-turn re-scan of answered comment slots.

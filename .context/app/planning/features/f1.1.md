# F1.1 — Document → questionnaire ingestion

> The committable working tracker for **F1.1**, the keystone feature of **P1
> (Questionnaire ingestion)**. Mirrors the promoted-task model in
> [`development-plan.md`](../development-plan.md#how-features-and-tasks-work):
> indicative scope, designed against Sunrise's _actual_ v0.0.1 surface, broken
> into right-sized **PRs** of commit-sized **tasks** with stable IDs, files, deps,
> and the review gate each PR is shaped to pass. Reconciled against the original
> deep spec ([Phase 1](../Conversational%20Questionnaire%20Phases.md)) — taken as
> indicative, with its now-closed Sunrise assumptions corrected (see
> [Reconciliation](#reconciliation-with-the-deep-spec)). Work through this doc;
> flip statuses as PRs land. It prescribes structure, not prompt wording or the
> final field list, which firm up at the moment of work.

## Intent

Let an admin upload a questionnaire document (PDF / DOCX / MD / TXT) and have an
LLM extract its **sections, questions, types, goal, and audience** into the app's
schema. The extractor is **opinionated, not literal** — it prunes boilerplate,
corrects typos, rewrites terse prompts, infers types, merges duplicates, splits
compound questions, and infers goal/audience — and **every editorial decision is
recorded as a revertible `AppQuestionnaireExtractionChange`** with before/after
JSON, a source quote, and a one-line rationale. **API-only — no UI** (P2 builds
the review/edit surface that consumes the change log).

**Definition of feature complete** (the plan's P1 DoD): uploading a representative
PDF produces a populated `AppQuestionnaire(Version|Section|Slot)` graph plus a
complete change-record log; integration tests cover the happy path and the three
big failure modes (scanned PDF, oversized doc, unparseable type); audit + cost
logs populated.

## Architecture decision 1: orchestration **primitives**, not a workflow

The development plan's headline question for this feature: _"if a task is best done
by configuring an agent workflow, that's what we should do."_ Considered seriously,
**rejected** for the core ingestion path.

**What F1.1 actually is:** a fixed, admin-triggered, _synchronous_ transformation
— bytes → parsed text → opinionated structured extraction (LLM) → a validated
graph written **transactionally** into bespoke app tables, with a per-decision
change-record audit trail. The value and the risk live in the **typed persistence
and change records**, not in LLM orchestration.

**Why not the workflow engine** (`AiWorkflow` + the 19 step types — see
[functional-specification §5](../../orchestration/meta/functional-specification.md#5-workflow-engine)):

- **No step type persists a bespoke app graph.** A workflow's terminal output is
  JSON in an execution trace. Writing `AppQuestionnaireSection` /
  `AppQuestionSlot` / `AppQuestionnaireExtractionChange` transactionally is
  app-specific code we write _regardless_; wrapping it in a `tool_call` only
  relocates it into the dispatcher and bolts on async execution leasing, the
  reaper, and poll-for-completion around a single blocking admin request.
- **Configurability is a liability here.** Workflows exist to be _recomposed_ by
  admins; the persist step is load-bearing app code that must not be reordered.
- **The parse step isn't a workflow input.** Bytes → text is route-level
  multipart handling before any LLM step.

**When this would flip to a workflow** (for a future reviewer): if ingestion grows
human-approval gates, parallel multi-judge fan-out, or admin-recomposable stages.
That shape is **P5 design-time evaluation**, which the plan already routes through
agents-as-judges / `AiEvaluationRun` — not P1.

## Architecture decision 2: extraction **is a capability** (dispatched programmatically)

The deep spec models extraction as a Sunrise **capability**
(`app_extract_questionnaire_structure`, a `BaseCapability` subclass), dispatched
**directly from the route** via `capabilityDispatcher.dispatch(slug, args, ctx)` —
not by an LLM in a tool loop. **Adopt this.** It aligns with F0.1's reserved
`initAppCapabilities()` hook ("populated from P1"), the plan's
primitives-consumed table, and keeps the extractor logic in `lib/app/questionnaire/`
(the plan's stated home for questionnaire logic — a `BaseCapability` subclass
importing `@/lib/orchestration/*` is lint-clean and storage-agnostic-by-transitivity,
the same precedent as F0.1's `feature-flag.ts` importing `@/lib/feature-flags`).

The dispatch pipeline ([functional-spec §4.2](../../orchestration/meta/functional-specification.md#42-dispatch-pipeline-7-stages))
gives us, for free: Zod argument validation, per-capability rate limiting, and the
**`processesPii` + `redactProvenance()` discipline** — load-bearing here, since
questionnaire documents carry PII (names, contact details in answers/examples).

**Two honest caveats (where the deep spec is inaccurate or thin):**

- **No built-in capability actually calls an LLM inside `execute()`.** The deep
  spec cites `estimate-cost` / `model-auditor` as precedents — they are not (a grep
  of `lib/orchestration/capabilities/built-in/**` finds no provider call in any
  `execute`). This is a **novel pattern** in the codebase; flag it for the
  reviewer. The clean primitive to use _is_ `runStructuredCompletion<T>`
  (`lib/orchestration/evaluations/parse-structured.ts`, verified): caller resolves
  the model via `resolveAgentProviderAndModel(extractorAgent, 'reasoning')` +
  `getProvider()`, passes `messages`, `parse: (raw) => tryParseJson(raw, zodValidate)`,
  a `retryUserMessage`, and `onFinalFailure`. It owns the call → parse →
  retry-once-at-temp-0 → cost-sum loop and returns `{ value, tokenUsage, costUsd }`.
- **The capability does NOT persist.** It stays storage-agnostic: it returns the
  structured result (`sections`, `questions`, `inferredGoal?`, `inferredAudience?`,
  `changes[]`) and the route writes it. This keeps the `lib/app/**` boundary clean
  (no `prisma` import in the capability) and is the right test seam — the
  capability is unit-tested by dispatch with a mocked provider; persistence is
  tested at the route.

Cost: the capability calls `logCost({ agentId, operation: CostOperation.CHAT, … })`
with the `costUsd` `runStructuredCompletion` returns (do not assume the dispatcher
auto-logs LLM cost — it doesn't, because no built-in makes such a call).

**Net shape** — a clean pure/IO split that satisfies both the deep spec and the
`lib/app` boundary:

| Layer                      | Lives in                                     | Does                                                            |
| -------------------------- | -------------------------------------------- | --------------------------------------------------------------- |
| Pure domain core           | `lib/app/questionnaire/ingestion/**`         | Zod extraction schema, prompt builder, change-record validation |
| Extractor capability (LLM) | `lib/app/questionnaire/capabilities/**`      | `runStructuredCompletion` + cost log; returns structure (no DB) |
| Persistence + HTTP         | `app/api/v1/app/questionnaires/{route,_lib}` | multipart, parse, dispatch, transactional persist, audit        |

## A sharper way to size PRs (the lever this feature pulls)

The development plan asks for a better articulation of right-sizing. Proposed
formulation — lift it back into the plan's sizing section if it lands:

> **A PR is a single reviewable claim.** Every gate (`/security-review`,
> `/test-review`, `/code-review`, type-check, coverage) evaluates one thesis:
> _"this change correctly and safely does X."_
>
> - **Too small** → the claim is trivial, so the _fixed_ cost of the gate ceremony
>   (CI runs, review round-trips, tracker/CHANGELOG updates) dominates the marginal
>   code. You pay the overhead N times for one coherent capability.
> - **Too large** → the claim is plural ("schema + parsing + LLM + API +
>   change-records"). No reviewer — human or agent — holds all the invariants at
>   once; a finding in one area masks a gap in another; the security review of the
>   upload path drowns in unrelated schema diff. The gate runs but its signal is
>   diluted.
>
> **Right size = the largest scope that still resolves to one claim with a
> self-contained test surface.** A right-sized PR: (1) leaves the build green and
> the feature-flag behaviour coherent at merge; (2) has a test surface a reviewer
> can exercise in isolation; (3) carries **one dominant risk type** so the matching
> gate has a focused target; (4) is revertable without stranding the others.
>
> The mechanical lever stays **commits ≠ PRs**: cluster commit-sized tasks up to
> the one-claim boundary; split the moment a second dominant risk type appears.

F1.1 is a `multi-PR` keystone. Applying the rule yields **four PRs**, each with a
different dominant risk type, in dependency order. The flag ships **off**
throughout, so landing API surface ahead of the P2 UI is safe.

## Schema design (new models)

All in `prisma/schema/app-questionnaire.prisma`, extending the F0.1 anchor.
Conventions + the **schema-fold migrate-dev footgun** are in
[`schema.md`](../../questionnaire/schema.md) — follow it exactly (review the
generated SQL; strip every phantom platform DROP/ALTER/RENAME; `npm run db:drift-check`).
`App…` prefix, snake_case `@@map`, app-internal relations with explicit `onDelete`,
**no `User` FK** (UG-1 — uploader/reverter identity is a plain `String` set from
the session, captured via `logAdminAction`, with no `@relation`).

Field choices are reconciled with the deep spec; divergences are flagged inline.

```prisma
// Goal/audience land on the VERSION (divergence from the deep spec, which put
// them on AppQuestionnaire — see Reconciliation). Rationale: the versioning model
// pins them on the version an admin launches; F3.1's launch gate and F5's judges
// both operate on a version. Add to the existing model:
model AppQuestionnaireVersion {
  // … existing F0.1 fields (id, questionnaireId, versionNumber, status, timestamps) …
  goal     String? // inferred (infer_goal change) or admin-set
  audience Json?   // structured AudienceShape (see types.ts); inferred or admin-set

  sections        AppQuestionnaireSection[]
  changeRecords   AppQuestionnaireExtractionChange[]
  sourceDocuments AppQuestionnaireSourceDocument[]
}

/// A section/group within a version. Plain entity — extraction provenance
/// (sourceQuote, rationale, confidence) lives on the CHANGE RECORDS, not here.
model AppQuestionnaireSection {
  id          String   @id @default(cuid())
  versionId   String
  ordinal     Int
  title       String
  description String?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  version   AppQuestionnaireVersion @relation(fields: [versionId], references: [id], onDelete: Cascade)
  questions AppQuestionSlot[]

  @@index([versionId])
  @@map("app_questionnaire_section")
}

/// A single question. `type` vocabulary is the project-canonical set from the
/// deep spec. `key` is a stable per-version slug (answers/re-ingest reference it,
/// not the cuid). `weight` feeds F4.1 weighted selection. NO embedding column yet
/// (deferred to F4.1 — see decision).
model AppQuestionSlot {
  id                   String   @id @default(cuid())
  versionId            String   // denormalised — F2.2 tagging validates tag+slot share a version
  sectionId            String
  ordinal              Int
  key                  String   // slug, unique per version
  prompt               String
  guidelines           String?
  rationale            String?
  type                 String   @default("free_text") // free_text|single_choice|multi_choice|likert|numeric|date|boolean
  typeConfig           Json?    // choices, likert bounds, etc.
  required             Boolean  @default(false)
  weight               Float    @default(1.0)
  extractionConfidence Float?   // 0–1, set by the extractor
  createdAt            DateTime @default(now())
  updatedAt            DateTime @updatedAt

  section AppQuestionnaireSection @relation(fields: [sectionId], references: [id], onDelete: Cascade)

  @@unique([versionId, key])
  @@index([versionId])
  @@index([sectionId])
  @@map("app_question_slot")
}

/// The revertible audit trail of every editorial decision the extractor made,
/// LLM-self-reported at extraction time. F2.3 lists these and restores `beforeJson`
/// on revert. Write path supports the full vocabulary; F1.1 EMITS the editorial +
/// inference set (see vocabulary note).
model AppQuestionnaireExtractionChange {
  id               String    @id @default(cuid())
  versionId        String
  changeType       String    // see vocabulary note
  targetEntityType String    // section | question | version
  targetEntityId   String?   // null for pruned-never-persisted / version-level
  sourceQuote      String?   // the span in the source document
  beforeJson       Json?     // pre-change state; restored on revert (null for creations)
  afterJson        Json?     // post-change state (null for prune_*)
  rationale        String?
  confidence       Float?
  status           String    @default("applied") // applied | reverted
  revertedAt       DateTime? // set in F2.3
  revertedByUserId String?   // plain String, no FK (UG-1); set in F2.3
  createdAt        DateTime  @default(now())

  version AppQuestionnaireVersion @relation(fields: [versionId], references: [id], onDelete: Cascade)

  @@index([versionId, status])
  @@index([changeType])
  @@map("app_questionnaire_extraction_change")
}

/// The uploaded source doc — supports SHA-256 dedup (F2.4 re-ingest) and parse
/// provenance / source-quote verification. NO User FK (UG-1).
model AppQuestionnaireSourceDocument {
  id            String   @id @default(cuid())
  versionId     String
  fileName      String
  fileHash      String   // sha256 of the raw bytes
  byteSize      Int
  mimeType      String?
  pageCount     Int?
  warnings      Json?    // parsed.warnings[]
  extractedText String   @db.Text // what extraction consumed; F2.3 verifies source quotes against it
  bytes         Bytes?   // OPTIONAL raw upload, for F2.4 diff-against-source/re-parse with new opts
  createdAt     DateTime @default(now())

  version AppQuestionnaireVersion @relation(fields: [versionId], references: [id], onDelete: Cascade)

  @@index([versionId])
  @@index([fileHash])
  @@map("app_questionnaire_source_document")
}
```

**`AudienceShape`** (export from `lib/app/questionnaire/types.ts` so every later
consumer — F5 judges, F6 agent, P2 UI — imports one shape; adopted verbatim from
the deep spec):

```ts
export type AudienceShape = {
  description?: string;
  role?: string;
  expertiseLevel?: 'novice' | 'intermediate' | 'expert';
  estimatedDurationMinutes?: number;
  locale?: string; // BCP-47, default 'en'
  sensitivity?: 'low' | 'moderate' | 'high';
  notes?: string;
};
```

**`changeType` vocabulary.** The model + write helper support the full deep-spec
set. Because the extractor is **opinionated**, F1.1 _emits most of it at ingestion_
(not just creations — a correction to the v1 of this plan):
`prune_section`, `prune_question`, `correct_spelling`, `correct_grammar`,
`rewrite_prompt`, `infer_type`, `merge_questions`, `split_question`, `add_section`,
`augment_question`, `infer_goal`, `infer_audience`. A change record is produced
**per editorial decision** (a verbatim, unedited question yields _no_ record);
`beforeJson` = the source-faithful state (for `prune_*` it is the only place the
data survives), `afterJson` = the final state (null for `prune_*`). `infer_goal` /
`infer_audience` target `version`; their `afterJson` holds the inferred value so
F2.3 revert clears exactly those fields. Conservative default: **when unsure
whether a span is content or boilerplate, keep it** (pruning is reversible; a
missing question the admin remembers is the worse failure).

> **<a name="deferred-embeddings"></a>Decision — defer `AppQuestionSlot.embedding`
> to F4.1.** The deep spec front-loads a `vector(1536)` column + HNSW index +
> `embedBatch()` call in Phase 1 "because Phase 4's adaptive strategy depends on
> it." Defer it: a pgvector column is a **Prisma-unmodelled object** (the
> schema-fold drift footgun + an app drift-probe registration via the now-shipped
> `registerAppDriftProbes()` in `lib/app/db-drift.ts`, UG-1 resolved) and has
> **zero consumer until F4.1**. Consistent with the plan's minimal-anchor stance
> (F0.1 deferred even the User FK). Adding it later is a cheap migration + a
> one-off `embedBatch` backfill over existing slots. Flagged so F4.1 owns it
> deliberately. _If the owner wants adaptive selection demo-ready sooner, this is
> the one decision to revisit._

## Inference & admin-supplied metadata (merge semantics)

Adopted from the deep spec — the upload accepts optional admin-supplied `goal` and
`audience.*` fields, and **admin wins per field**:

- The upload form accepts `goal` (text) and `audience.{description,role,expertiseLevel,estimatedDurationMinutes,locale,sensitivity,notes}`.
  Parse with a Zod schema; the `audience.*` fields nest into one `AudienceShape`.
- The capability receives the admin-supplied values so it knows **which fields not
  to infer**. Suppression is **per field**: admin supplies `role` only → extractor
  may still infer `expertiseLevel`, `sensitivity`, etc., but not `role`.
- Persist with merge: for each field, use the admin value if supplied; else the
  inferred value if non-null; else leave the existing DB value untouched (a second
  upload never blanks a previously-inferred field).
- Inference that actually ran is recorded as `infer_goal` / `infer_audience` change
  records (revertible per field). Inference suppressed by an admin value produces
  **no** change record.
- The response tags each resolved field with provenance: `admin-supplied | inferred | pre-existing` (UI-ready for P2).

  `getProvider(slug)` → `runStructuredCompletion({ provider, model, messages, parse: (raw) => tryParseJson(raw, (p) => extractionSchema.safeParse(p).success ? … : null), retryUserMessage, onFinalFailure })`. Then `logCost({ agentId, operation: CostOperation.CHAT, model, provider, inputTokens, outputTokens })` from the returned `tokenUsage` — fire-and-forget; **no silent fallback** on final parse failure (the silent-failure gate looks for this).
- **Seeds**: agent per `006-quiz-master.ts` (`upsert` by slug `app-questionnaire-extractor`, `isSystem: true`, empty `model`/`provider`, a `monthlyBudgetUsd` cap, `createdBy` via `serviceAccountWhere`); capability row per `011-call-external-api.ts`. App seed keys are the relative paths (`app-questionnaire/00N-…`) — no collision with core seeds.
- Tests mock `getProvider` to return canned `LlmResponse` JSON; assert the parsed structure, the repair path, the redaction round-trip, and the cost call.

### PR 4 — Ingestion HTTP endpoint + persistence

**Claim:** `POST /api/v1/app/questionnaires` ingests end-to-end — flag-gated,
admin-authed, multipart-guarded, rate-limited, parsed, dispatched, persisted in one
transaction, audited. Dominant risk: HTTP/auth/input/transaction/security → the PR
`/security-review` is shaped for.

| T1.4.6 | Integration tests: 404 flag-off · 403 non-admin · 200 happy · 3 failure modes · dedup · merge-semantics · audit | `tests/integration/.../questionnaires/route.test.ts` | T1.4.1–5 | backlog |

- **Gate order**: `ensureQuestionnairesEnabled()` first (404 when off — F0.1
  template), then `withAdminAuth`, then `enforceContentLengthCap`
  (`@/lib/api/multipart-guard`). Extension allowlist (`.pdf .docx .md .txt`) is the
  source of truth; caller MIME advisory (mirror knowledge `documents/route.ts`).
- **Parse**: `parseDocument(buffer, fileName, { extractTables })` directly — _not_
  `previewDocument`/`confirmPreview` (those chunk+embed into the RAG KB; see
  Reconciliation). Scanned-PDF detection from `parsed.pageInfo[].hasText` +
  `parsed.warnings`.
- **Dispatch**: `capabilityDispatcher.dispatch('app_extract_questionnaire_structure', { documentText: parsed.fullText, fileName, mediaType, adminProvidedGoal, adminProvidedAudience }, { userId, agentId: <extractor id> })`.
- **Persist** (`_lib/persist.ts`): one `executeTransaction` (`@/lib/db/utils`) —
  questionnaire + version + sections → slots + change records + source document,
  all-or-nothing — applying the goal/audience merge semantics; return per-field
  provenance tags.
- **Failure modes → typed envelopes** (`errorResponse`): scanned PDF →
  `SCANNED_DOCUMENT`; oversize → `FILE_TOO_LARGE`; unsupported ext →
  `UNSUPPORTED_FORMAT`. No silent success on a bad parse.
- **Rate limit**: route inherits the 100/min `api` section cap automatically (do
  **not** call section limiters directly). Add a tighter per-flow sub-cap (each
  ingest is ≥1 LLM call): recommended a dedicated app limiter (`createRateLimiter`,
  ~10/min keyed on the admin user id) checked in-handler — the `uploadLimiter`
  precedent. Alternative: register via `registerAppRateLimits()`. Pick one, note it.
- **Response**: `successResponse({ questionnaireId, versionId, sectionCount, questionCount, changes, goal, audience, fieldProvenance })`.

## Reconciliation with the deep spec

The original [Phase 1 deep spec](../Conversational%20Questionnaire%20Phases.md) is
indicative and predates Sunrise v0.0.1. What this plan **adopts**, **diverges on**,
and treats as **outdated**:

**Adopted** — extraction as a `BaseCapability` dispatched programmatically;
opinionated extractor emitting per-decision change records (prune/correct/rewrite/
merge/split/augment/infer); structured `AudienceShape`; slot `key`/`type`/
`typeConfig`/`guidelines`/`rationale`/`extractionConfidence`/`weight`;
admin-wins-per-field merge semantics with per-field provenance tags;
`runStructuredCompletion` as the structured-LLM primitive (the one deep-spec code
reference that is still accurate); the 006-quiz-master agent-seed and
011-call-external-api capability-seed shapes; `processesPii` + `redactProvenance`.

**Diverged** (with rationale):

- **Goal/audience on the _version_, not `AppQuestionnaire`** — the versioning model
  pins them where launch/evaluation happen (F3.1 gate + F5 judges are version-scoped).
- **Embeddings deferred to F4.1** — no consumer until then; avoids a pgvector
  unmodelled-object + drift-probe in P1. [Decision above](#deferred-embeddings).
- **Single ingestion endpoint** (`POST /api/v1/app/questionnaires`, multipart,
  returns version id) per the current development plan — not the deep spec's
  two-step create-draft + `/:id/upload`. Full CRUD (list/detail/PATCH/DELETE) and
  the extraction-change **review/revert routes** stay in their development-plan
  homes (**F2.1** authoring, **F2.3** review) — F1.1 only _writes_ the change log.
- **Source document as its own table** (`AppQuestionnaireSourceDocument`) rather
  than `sourceDocumentBytes` columns on the version — cleaner for F2.4 re-ingest
  history + dedup; keeps optional raw `bytes` for diff-against-source.

**Outdated in the deep spec** (Sunrise v0.0.1 closed these — do **not** follow):

- Single `prisma/schema.prisma` + "zero-touch breach" → multi-file schema is live;
  app models are in `app-questionnaire.prisma` (F0.1).
- `lib/app/questionnaire/seeds/` run via `tsx …/seeds/run.ts` → recursive
  `prisma/seeds/app-questionnaire/NNN-*.ts` via `npm run db:seed` (F0.1).
- App-owned `instrumentation.ts` workaround for capability registration → Sunrise
  ships the `lib/app/capabilities.ts` `initAppCapabilities()` hook, auto-wired.
- Plain-`String` User FK with `@db.VarChar(30)` + hand-written cascade → **deferred
  entirely** (UG-1); no User FK in F1.1. The app-extensible drift-probe seam
  (`registerAppDriftProbes()`) now exists for the day one is added.
- "Look at how `estimate-cost` / `model-auditor` call an LLM inside a capability" →
  **inaccurate**: no built-in capability calls an LLM in `execute()`. Use
  `runStructuredCompletion` (this plan, PR3).
- `streamChat()` to run the extractor as a chat/tool loop → not used; a single
  structured call via the capability is the right shape.

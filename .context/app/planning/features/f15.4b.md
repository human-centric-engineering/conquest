# F15.4b — Run-level respondent report

> **Everything still open across P15 is consolidated in [`f15-followups.md`](./f15-followups.md).**

> The other half of F15.4. F15.2's `conclude` path has promised a summary since it shipped; until
> now "See your summary" pointed at the last leg's own report, which described one questionnaire
> rather than the journey the respondent actually took.

## Confirmed decisions

- **The pipeline is REUSED, not forked.** `generateReportFromInputs` already takes pre-assembled
  transcript / data-slot / coverage material, so `run-report.ts` assembles inputs from N legs and
  everything downstream — KB grounding, web-search rounds, the report agent, the formatter, the
  appendix pass, the method record — applies unchanged. A parallel generation pipeline would have
  doubled the surface and drifted on exactly the details that are invisible when wrong.

- **A leg generates NO per-session report.** The run report covers every leg; generating both would
  bill the journey twice and hand the respondent n+1 reports where they were promised one summary.
  The submit route now skips the enqueue for a session that is a leg. The lookup fails soft to
  "not a leg" — a redundant per-leg report is a far better failure than none at all.

- **The run report is enqueued from `concludeRun`** — the single choke point where a journey is
  known to be over. Every dead end funnels through it (selector, exhausted budget, no candidates,
  unrunnable step), so a respondent gets a report regardless of WHY their journey ended. Fail-soft:
  the run IS concluded, and a failed enqueue must not turn that into a failed advance.

- **Settings come from the ENTRY leg's version, not the last.** A run spans several versions, so
  something must arbitrate, and the entry leg is the only leg every run has. Anchoring on the last
  leg would give two respondents on the same experience differently-styled reports purely because
  the selector routed them differently — a difference the author never asked for.

- **The KB scope comes from the EXPERIENCE's `demoClientId`** — a real relation, and the right
  scope for a journey the client owns. Falling back to a leg's questionnaire would pick an
  arbitrary one.

- **Per-leg transcript headings are load-bearing.** Without `## Part N — <title>` the writer reads
  one flat wall of Q&A and cannot tell the respondent was asked about a topic TWICE, in two
  different questionnaires — exactly the progression a journey report should notice and a
  single-questionnaire report never can.

- **Coverage sums across the whole journey.** Someone who answered one leg fully and abandoned the
  next mid-way is at partial coverage overall; reporting the final leg's completion alone would
  overstate how much of the journey was answered and suppress the partial-report caveat.

- **`runId` is a REAL relation with `onDelete: Cascade`**, matching how `AppExperienceRunLeg`
  attaches to its run. This is not a UG-1 violation: UG-1 forbids CONFIG→answer edges, and a run is
  the answer side. Deleting a run must take its report with it.

- **Erasure posture follows the existing pattern deliberately** (no `User` FK). Confirmed against
  `.context/privacy/data-erasure.md`: `AppRespondentProfileSnapshot` is the ONE app table with a
  modelled `User` FK; sessions, answers, transcripts and reports are all retained. Giving the run
  report a cascading FK would have made it inconsistent with the per-session report beside it,
  which is worse than either uniform choice. See "Known gap" below.

- **A run report suppresses the method panel.** The base view resolves the ENTRY LEG's method
  record, which no longer exists now that legs do not generate reports. Showing the wrong run's
  method would be worse than showing none.

## What shipped

**Schema** (hand-stripped migration): `AppRespondentReport.subjectKind` (`session` |
`experience_run`), `sessionId` relaxed to nullable-unique, `runId` nullable-unique with a cascading
FK, plus the reverse `report` relation on `AppExperienceRun`.

**Lib**: `run-report.ts` (`generateRunReport` — multi-leg input assembly), `run-view.ts`
(`buildRunReportClientView` — composes the entry leg's chrome with the run's generation state),
`enqueueRunReport` + `isExperienceLeg`, `RESPONDENT_REPORT_SUBJECTS`.

**Worker**: one queue, one lease, one tick — `ClaimedReport` carries both owner keys and
`driveReport` branches. A row with neither owner fails terminally rather than being re-claimed
forever. Run-scoped revisions (admin re-run) fail terminally too: not supported yet, but they
cannot currently be created either.

**Routes**: `GET /api/v1/app/experiences/runs/:runId/report`, gated by the shared `canReadRun` with
the admin bypass excluded — a report is a narrative about a specific person.

**UI**: `useRespondentReport` takes an optional `runId` and polls the run endpoint; `SessionComplete`
takes `runId`; the workspace passes it from the session status view.

## Design notes

**Making `sessionId` nullable surfaced every assumption at once.** The type error cascade through
`worker.ts` was the feature working as intended: each site that assumed a session had to be
examined, including the report-ready email (session-scoped, now guarded) and the revision claim
path (run reports cannot be re-run yet, and now say so instead of crashing the tick).

**`sessionId: run:<runId>` follows the existing `preview:<vid>` sentinel convention.** It is used
only for research logging and KB warnings, never as a lookup key — passing a leg's real id would
attribute the whole run's research to one arbitrary session.

## Known gap (pre-existing, deliberately not widened)

`eraseUser()` removes the profile snapshot but retains sessions, answers, **transcripts** and
reports — their `respondentUserId` is a plain String with no FK, and no erasure hook is registered.
For structured answers that is a defensible "retain de-identified research data" posture; free-text
transcripts and generated report prose are not de-identified by removing the snapshot.

F15.4b follows the existing pattern rather than half-fixing it. Closing the gap is its own scoped
work with a real product decision inside it (delete the answers, or keep them and redact the free
text).

## Deferred

- An experience-wide synthesis over ready step reports.

## Closed later

- **Re-running a run report.** `POST /runs/:runId/report/retry` — "Check again" now re-queues and
  kicks the worker rather than only re-polling. Refuses to clobber a report with content or one in
  flight.
- **A run-scoped notify opt-in.** `POST /runs/:runId/report/notify`, accepted only while the report
  is still `queued` or `processing` — an address attached to a finished or failed report promises
  something that is not coming.
- **The run's own method record**, gated on the same `explainMethod` delivery setting the base view
  applies, so a run report cannot become a way around an author's opt-out.

# F9.15 — Config Advisor

> Committable tracker for **F9.15**. A questionnaire's behaviour is spread across many settings —
> structure, goal/audience, the run-time `AppQuestionnaireConfig`, data slots, scoring — and they
> interact in non-obvious ways. This adds an admin-triggered AI panel on the version Settings tab
> that reads the whole picture, streams a narrative of the respondent experience the current config
> produces (plus the lifecycle state), and proposes one-click config tweaks. Ephemeral and
> re-runnable; it never runs on its own.

## Intent

Give the admin a single "is this configured well?" review before launch (and after tweaks):
flag conflicts (e.g. adaptive selection under a 1-question cap; public access with required PII;
contradiction probing that eats a tight per-session budget; conversational features under a `form`
presentation), describe in plain language what a respondent will actually experience, and offer
concrete, applyable fixes — without the admin having to reason across every settings panel.

## What shipped

- **Flag** `APP_QUESTIONNAIRES_ADVISOR_ENABLED` (DB row, seeded disabled) + resolver/gate
  (`isAdvisorEnabled` / `withAdvisorEnabled`) in `lib/app/questionnaire/feature-flag.ts`; surfaced as
  `flags.advisor` in `resolveQuestionnaireWorkspaceFlags`. Seed `056-advisor-flag.ts`.
- **Agent** `app-questionnaire-advisor` (`QUESTIONNAIRE_ADVISOR_AGENT_SLUG`), runtime-resolved
  binding, internal visibility, budget-capped. Seed `057-advisor-agent.ts`.
- **Context assembler** `loadAdvisorContext` (`_lib/advisor-context.ts`) — reuses `getVersionGraph`
  and adds session count, data slots, scoring; bounded snapshot (counts + sample prompts).
- **Prompt builders** (`advisor-prompt.ts`) — two phases: streamed narrative, then structured
  analysis constrained to the applyable-field allowlist.
- **Orchestrator** `streamAdvisor` (`stream-advisor.ts`) — async generator, never throws: Phase 1
  `provider.chatStream` (narrative deltas), Phase 2 `runStructuredCompletion` (conflicts +
  suggestions), `logCost` once. Pure event types in `advisor-events.ts`; analysis schema +
  normalisation in `advisor-schema.ts`.
- **Route** `POST …/versions/:vid/advisor/stream` — admin-only SSE; flag gate, per-admin sub-cap
  (`advisorLimiter`, 20/min), audit action `questionnaire.advisor`. No GET, no auto-run.
- **Client** `AdvisorPanel` — admin-triggered; streams the Markdown narrative, renders conflicts +
  one-click suggestions. Apply reuses `PATCH …/config` via `authoringMutate` (full validation +
  fork-on-launch); panel goes stale after apply → re-run. Rendered on the Settings tab behind
  `flags.advisor`.

## Design notes

- **No new write path.** One-click apply posts a partial config `patch` to the existing version
  config endpoint, inheriting `updateConfigSchema`'s cross-field validation and the fork-on-launch
  redirect. Suggestions only target scalar/enum fields (`ADVISOR_APPLYABLE_CONFIG_FIELDS`); the
  structured JSON blocks (tone, reports, intro, profile/invitee fields) surface as prose conflicts.
- **Ephemeral.** No Prisma model / migration — the run is on-demand and re-runnable; revisiting the
  tab resets to the idle state.
- **Admin-triggered only.** Nothing fires on render, tab visit, apply, or settings change.

## Tests

Unit: advisor-schema (normalisation), stream-advisor (lifecycle + fatal paths + cost-once + never
throws), advisor-prompt (shape + allowlist), advisor-flag / advisor-agent seeds. All green under
`npm run validate` + `npm run test`.

# F9.14 — Re-hydrate the Turn Inspector drawer on resume

> Committable tracker for **F9.14**. The Preview Turn Inspector drawer is fed by live `inspector` SSE
> frames into client-only state, so a page reload emptied it until the next turn — even though F9.13
> already persists every turn's traces. This closes that gap: the transcript replay now also returns
> the persisted traces (preview-gated) and the chat hook seeds the drawer from them, so the drawer
> survives a reload.

## Intent

An admin previewing as a respondent reloads the page (or returns to an in-progress preview) and finds
the inspector drawer **gone**, despite the version's "Turn inspector" toggle being on. Reported as a
regression; it was never wired — F9.13 persisted the traces but only added an admin **lookup** read
path, not a **resume** one. The transcript already rehydrates the conversation on resume; the
inspector should rehydrate the same way.

## Root cause

`useQuestionnaireSessionStream` seeds `turns` from `initialTurns` (rehydrated from the DB on resume)
but initialized `inspectorTurns` to `[]`, populated **only** by live frames. The drawer mounts on
`inspectorTurns.length > 0`, so after a reload it stayed hidden until the next answer. Capture +
persistence (F9.13) were healthy; only the resume read side was missing.

## What shipped

- **Loader** `loadInspectorTurns(sessionId)` (in `_lib/transcript.ts`, beside `loadTranscript`):
  reads each turn's persisted `inspectorCalls`, validates with the shared `inspectorTurnSchema`
  (drops empty/malformed — never throws), and maps the 1-based `ordinal` to the live 0-based
  `turnIndex` (`ordinal − 1` = the `selectionRound` the live frame used), so a hydrated turn maps to
  the same transcript message the drawer derives its evaluation context from.
- **Transcript route** now selects `isPreview` + `version.config.previewInspectorEnabled` and returns
  `inspectorTurns` **only** for a preview session with the toggle on — the same gate the live frame
  enforces. The field is omitted entirely otherwise (no empty-array intent leakage).
- **Client thread:** `transcriptResponseSchema` (in `anonymous-session-boot.tsx`) validates the new
  `inspectorTurns`; `fetchTranscript` returns `{ turns, inspectorTurns }`; boot state carries
  `initialInspectorTurns`; `SessionWorkspace` passes it to the hook; the hook seeds
  `inspectorTurns` state from `initialInspectorTurns ?? []`.

## Tests

- `transcript.test.ts` — `loadInspectorTurns`: ordinal-asc select shape, `ordinal − 1` → `turnIndex`,
  empty-calls turn skipped, malformed `inspectorCalls` JSON dropped (fails soft).
- `transcript-route.test.ts` — preview + toggle on returns `inspectorTurns` (loader called); toggle
  off and non-preview both omit the field and never call the loader.
- `use-questionnaire-session-stream.test.ts` — seeds `inspectorTurns` from `initialInspectorTurns`;
  defaults to `[]` for a real respondent.

## Gotchas

- **Same gate, two read paths:** the resume replay reuses the live-frame gate (preview + toggle). A
  real respondent's transcript read never includes the field — verified by test. This updates the
  F9.13 note that "transcript/resume select only message/warnings/reasoning": still true for
  respondents, but a preview now also gets `inspectorCalls` (admin-only).
- **`turnIndex` must reproduce `selectionRound`**, not the raw ordinal — otherwise the drawer's
  `deriveTurnContext(messages, turnIndex)` would point at the wrong user message. `ordinal − 1` holds
  because `ordinal = count + 1` at write time and `selectionRound = count` at turn start.
- **No new endpoint:** folded into the existing transcript route (one boot fetch), mirroring how
  `initialTurns` already flows — rather than a second round-trip.

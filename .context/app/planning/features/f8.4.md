# F8.4 — Admin session viewer

> Committable tracker for **F8.4**. The "P8 admin session view" the F7.4 `export.pdf` route was
> explicitly built for ("P8's admin session views will wire a button to it"). An admin pastes a
> respondent's **support reference** and reads the conversation; a real respondent conversation
> opens **read-only**, while an admin's own **preview** conversation can be **continued**. No
> migration — reads existing rows. See `.context/app/questionnaire/admin-session-viewer.md`.

## Intent

F7.4 shipped an admin-authenticated session export route with no UI trigger, deliberately, so the
P8 admin session views would be pure front-end. F8.4 builds that surface — but on screen, not just
as a PDF: look up a session by reference, replay its conversation, and (for a preview the admin
started) continue it.

## Decisions (confirmed with the user)

- **Read-only vs. continue keys on `isPreview`.** A real respondent session (`isPreview: false`) is
  read-only; an admin preview session (`isPreview: true`) that is still `active` is continuable.
  This is the authorization the system already enforced — F8.4 surfaces it, it does not invent it.
- **Two-layer gate.** The UI only offers continue for an active preview; independently, the
  `preview-token` route refuses to mint a continue credential for anything but a preview
  (`409 SESSION_NOT_PREVIEW`), and `resolveTurnAccess` `403`s any admin posting into a respondent's
  authenticated session. A real respondent session is therefore structurally un-continuable.
- **Both entry points.** A compact ref input in the workspace header (reachable from every tab) AND
  a dedicated **Sessions** tab (gated on the `liveSessions` flag), built to later grow a session
  list above the lookup.
- **Mirror the export's redaction.** Anonymous-mode sessions show no respondent identity in the
  viewer — `loadAdminSessionView` never queries the name when `anonymousMode`, exactly like the F7.4
  model builder.
- **Admin authz mirrors the export, not the respondent surface.** The reads are nested under the
  questionnaire (`…/questionnaires/:id/sessions/:sessionId/…`) with the F7.4 ownership rule
  (cross-questionnaire → 404), `withAdminAuth`, flag-gated first.

## Surface

- **API** (`app/api/v1/app/questionnaires/[id]/sessions/[sessionId]/…`): `GET transcript`,
  `POST preview-token`; plus `GET …/questionnaires/sessions/by-ref/[ref]` (a static sibling of the
  `[id]` segment) to resolve a reference → viewer location. Read seams in
  `_lib/admin-session-view.ts` (`loadAdminSessionView`, `resolveSessionRefLocation`).
- **Read-only chat:** additive `readOnly` prop on `SessionWorkspace` / `QuestionnaireChat`; the
  panel/lifecycle/form hooks gained an `enabled` flag (mirroring `useFormAnswers`) so read-only mode
  fires no respondent-scoped fetches (the viewing admin holds no respondent credential).
- **UI:** `SessionRefLookup` (compact header field + Sessions-tab panel), the Sessions tab page, and
  the viewer route `…/v/[vid]/sessions/[sessionId]` (branches on `isPreview`; mints the preview
  token server-side via `mintSessionToken`).

## Tests

- Integration (`admin-session-viewer-routes.test.ts`): admin-only access (401/403), ownership
  (cross-questionnaire → 404), the `SESSION_NOT_PREVIEW` / `SESSION_NOT_ACTIVE` continue gates, the
  happy-path token mint, ref-lookup resolution, and anonymous redaction carried through the read.
- Unit (`admin-session-view.test.ts`): identity redaction (no user query in anonymous mode), status
  narrowing, forgiving ref normalisation before lookup.
- Component (`session-workspace.test.tsx`): `readOnly` renders only the read-only transcript, makes
  the three hooks inert, and shows the conversation (not the completion screen) for a completed
  session.

# F9.19 — Respondent-controlled early finish

> Completion was entirely **admin/agent-driven**: a session became submittable only when the
> agent's own thresholds (`coverageThreshold` / `minQuestionsAnswered`) or the hard
> `maxQuestionsPerSession` cap were met, and the submit route refused anything else with
> `SUBMIT_NOT_READY`. There was no way for a **respondent** to say "I've done enough — give me my
> report" before that bar. This adds an opt-in escape hatch: once they cross an admin-set minimum, a
> calm **Continue / Finish up** control lets them end the session and get their report — bypassing
> the required-question gate.

## Confirmed decisions

- **Unlock rule:** min coverage % **OR** min answered-count — crossing _either_ configured bar
  unlocks. A bar of `0` means "not a criterion on that axis" (both `0` ⇒ available from the start).
- **Bypasses the required gate:** the whole point is a true escape hatch — the respondent can finish
  with required questions still open. (The agent's own `offer` still respects required questions.)
- **Presentation:** a _persistent_ control once unlocked (not a one-time banner). "Continue"
  collapses it to a slim, ever-present "Finish up now" link so it never nags.
- **New dedicated settings**, not a reuse of `coverageThreshold` / `minQuestionsAnswered` — "the
  agent proactively offers" and "the respondent may bail early" are distinct bars.
- **Config-only** (no platform flag), like `inlineCorrectionEnabled`.

## What shipped

- **Config / schema:** three fields on `AppQuestionnaireConfig` — `allowEarlyFinish` (bool, `false`),
  `earlyFinishMinCoverage` (0–1, `0.5`), `earlyFinishMinQuestions` (int ≥0, `0`). Added to
  `QuestionnaireConfigShape` + `DEFAULT_QUESTIONNAIRE_CONFIG` (`types.ts`), the read projection
  (`CONFIG_SELECT` / `ConfigRow` / `toConfigView` in `_lib/detail.ts`), and the Zod
  `updateConfigSchema`. Migration `20260630120000_app_early_finish` (additive ALTER TABLE; applied
  via `migrate deploy`, 5 vector indexes verified intact). Settings import/export auto-includes them
  (key list derives from `DEFAULT_QUESTIONNAIRE_CONFIG`).
- **Pure core:** `assessCompletion` computes `earlyFinishAvailable` independently of `kind` via the
  exported `isEarlyFinishAvailable(config, coverage, answered)` (OR of the two bars; `0` = not a
  criterion). `COMPLETION_ACTIONS` gains `'finish_early'`; `resolveCompletion('finish_early', …)` →
  `submit` when available (no sweep, ignores the required/threshold gate), else `continue`. The
  data-slot orchestrator's inline assessments reuse the same helper so they can't drift.
- **Status + hook + route:** `StatusCompletionView.earlyFinishAvailable` + `canFinishEarly(view)`
  (`status-view.ts`); `useSessionLifecycle` exposes `canFinishEarly` + `finishEarly()` (POSTs
  `{ early: true }`); the submit route accepts an optional `{ early?: boolean }` body, resolves via
  `finish_early`, and completes with reason `respondent_early_finish` (vs `respondent_submit`).
- **Respondent UI:** new `EarlyFinishControl` (persistent Continue / Finish-up, collapses to a slim
  link). `SessionWorkspace` renders by precedence: full `CompletionOffer` → else
  `EarlyFinishControl` when `canFinishEarly` → else nothing; shared by the chat and form surfaces.
- **Admin UI:** a toggle + two numeric fields (min coverage, min questions) in the Settings tab's
  "Questions & completion" group, each with `<FieldHelp>`.

## Design notes

- **Orthogonal to `offer`.** `earlyFinishAvailable` is computed and surfaced independently of the
  agent's `offer`/`not_ready`/`blocked_on_required` decision; both can be true at once, and the UI
  prefers the full submit offer when so.
- **Authoritative gate is recomputed.** `GET …/status` and the submit route each re-run
  `assessCompletion`, so a stale/forged client can't early-finish an ineligible session.

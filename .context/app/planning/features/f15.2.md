# F15.2 — Agentic Switcher runtime

> **Everything still open across P15 is consolidated in [`f15-followups.md`](./f15-followups.md).**

> F15.1 gave an admin somewhere to author a journey. F15.2 makes one run: a respondent completes
> the entry questionnaire, a decision is made about what would serve them next, and they are either
> routed into a chosen follow-up or given their report.

Ships `linked` continuity — the respondent sees two distinct conversations with an explicit handoff
between them. `stitched` (the same run presented as one continuous chat) is F15.3 and changes
nothing about the data written here.

## The three-tier decision

Resolution order at every fork, in `_lib/run-advance.ts`:

1. **Budget gate** — a run at hard cap concludes regardless of what anything else wants.
   Continuing is precisely what the budget exists to prevent.
2. **Deterministic rules** (`routing/rules.ts`) — first match by ordinal wins, and the LLM is never
   called.
3. **The LLM selector** (`routing/select.ts`), then the configured **fallback**
   (`routing/fallback.ts`) on error, unknown step key, or sub-threshold confidence.

Rules exist for two reasons, not one: authors want certain cases hard-pinned, **and** the
respondent is standing at the fork waiting — a rule resolves in microseconds where the selector
takes seconds.

## Confirmed decisions

- **Never strand a respondent.** Every dead end resolves to `conclude`, never to an error: an
  exhausted budget, no candidates, a chosen step whose version was deleted, a closed round window.
  Someone who has just finished a questionnaire must always end up with a report — a run left
  mid-flight means they sit on a spinner until it times out and receive nothing. `blocked` is
  reserved for an incoherent _call_ (unknown run, a session that is not one of its legs).
  **This was caught by end-to-end verification, not by design** — the first implementation returned
  `blocked` for a session-create failure, which left the run stuck in `awaiting_handoff`.

- **`@@unique([runId, ordinal])` arbitrates the handoff race, not a read-check.** `after()`, a
  double-tapped submit and a cron retry can all race `advanceExperienceRun`. A read-then-write
  existence check loses that race; the constraint does not. P2002 is treated as success (`noop`).
  Verified with concurrent calls against the real DB.

- **`AppExperienceRunLeg.sessionId` is a plain String, `@unique`, with NO relation.** The
  load-bearing FK decision of the feature. It is an identity↔answer pointer exactly like
  `AppQuestionnaireSession.roundId`; a real relation would make respondent answers reachable by
  walking experience config and let config deletion cascade into respondent data.

- **Run-level budget closes a real hole.** `AppQuestionnaireConfig.costBudgetUsd` is PER SESSION,
  so an n-leg run would silently get n× the intended spend. `AppExperience.costBudgetUsd` is
  enforced at the handoff gate (the highest-value single control) and via `effectiveLegBudget`,
  which takes the tighter of the two caps. `remainingRunBudget` floors at 0 rather than going
  negative — a negative remainder would read as a non-positive cap, i.e. _uncapped_, which would
  hand an overspent run unlimited budget.

- **Carry-over is deterministic-first and frozen at handoff.** Data-slot fills are the spine (the
  semantic answer layer, stable across question rewording); the LLM briefing is strictly additive
  and optional. Frozen onto `AppExperienceRun.carryOver` and never recomputed — an earlier leg may
  later be re-scored, and a finished report must not shift underneath it.

- **Anonymity is read from the SOURCE LEG's version config, never the experience's.** An anonymous
  entry leg has no profile to carry regardless of the experience's `carryProfile` setting. Getting
  this backwards would leak PII the respondent was promised would not be collected.

- **Safeguarding state carries unconditionally.** No setting gates `sensitivityLevel` /
  `sensitivityNotes`, and `createSessionForExperienceLeg` copies them onto the new session. An
  experience that forgets a disclosure between legs makes the next interviewer re-open it.
  Carried as summaries only — never the raw disclosure text, which has no business being re-inlined
  into another questionnaire's prompt.

- **Provisional data-slot fills are NOT carried.** They are best-effort inferences the pipeline
  recorded after giving up on a re-ask, and are shown to the respondent as "may revisit". Carrying
  them into the next questionnaire as settled fact would launder a guess into a premise.

- **Leg B's create bypasses `sessionStartLimiter` and the access-mode gate.** A server-initiated
  handoff is not a respondent-initiated start (counting it would burn a legitimate respondent's
  quota), and a step's questionnaire may legitimately be `invitation_only` — reachable only through
  this experience. Round windows still apply when a step pins one.

- **The poll endpoint never triggers work.** A poll that could cause an advance would let a page
  refresh double-fire the handoff. Two indexed reads, no LLM, no writes — and P15.5's facilitator
  console reuses the same primitive.

- **Every routing decision writes an `AppAiRun`, including deterministic ones.** "Why did this
  respondent get that questionnaire" is a question an admin asks months later, and a rule-based
  answer is as worth defending as an inferred one. Rule/budget decisions record
  `provider: 'deterministic'` — a real filterable value, not a fake provider slug that would
  pollute cost trends.

## Design notes

**`[object Object]` in a prompt.** Lint caught `String(fill.value)` on values that are legitimately
objects — which would have silently replaced a real structured answer with a meaningless token that
the model had no way to know was a substitution. Fixed by a single shared
`carryover/text.ts`: `fillText` (value-first, for rules) and `fillPromptText` (paraphrase-first,
for prompts). The inverse preferences are deliberate — rules want the answer, prompts want the
respondent's own words.

**`useRef(Date.now())` is impure.** Reading the clock during render differs between a server render
and its hydration. Stamped in the effect instead.

**Verification found the stranding bug.** Unit tests all passed; the end-to-end script against the
real DB is what surfaced that a step pinned to a deleted version left the run stuck. Worth
repeating for later phases: the pure logic was right, the orchestration was not.

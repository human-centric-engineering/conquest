# F2.4 — Re-ingest

> Committable tracker for **F2.4**, the **final feature of P2 (Admin CRUD over
> questionnaires)**. An admin uploads a _replacement_ source document against an
> existing **draft** version; an identical re-upload (same SHA-256) short-circuits
> to a no-op; a non-identical upload re-runs the extractor and **replaces that
> draft's extracted graph + change log** with a fresh one. Closes the P2 loop:
> ingest → review → edit → tag → version → **re-ingest**. Gated by
> `APP_QUESTIONNAIRES_ENABLED` (404 when off), API-first, riding the F1.1 ingest
> pipeline and the F2.1 scoped-version helpers.

## Intent

F1.1 ingests a document once into a brand-new questionnaire. F2.4 lets an admin
**swap the source** of a draft they're still authoring — they fixed a typo in the
source doc, got a newer revision from the client, or want to re-extract with
table extraction on. Re-ingest re-runs the same opinionated extractor and lays
down a fresh structure + editorial change log over the **same draft version**.

**No migration.** Every column F2.4 needs already exists from F1.1 — including the
nullable `bytes` on `AppQuestionnaireSourceDocument` (added by
`20260601144112_app_questionnaire_ingestion`, reserved "for F2.4"). F2.4 does
**not** start persisting `bytes` (the admin re-uploads; there's no diff-against-
stored-source consumer yet) — that stays deferred.

## Decisions (at planning — confirmed)

- **Replace the draft graph in place** (not a new version). The fresh extraction
  overwrites the target draft version's sections → slots, change-log, and re-runs
  the goal/audience merge; `versionNumber` and `status` are untouched. Matches the
  plan's wording ("against an existing draft version", "fresh extraction + change
  log"). **Destructive** of manual authoring edits and tag assignments on that
  draft — guarded by an explicit confirm in the UI.
- **Draft-only.** Re-ingest is a draft editorial operation. A `launched` or
  `archived` target returns a typed **409 `REINGEST_NOT_DRAFT`** — _not_ a
  fork-on-launched. (Fork is the right seam for incremental edits; wholesale
  re-extraction onto a just-forked copy is heavier and semantically muddier. An
  admin who wants to re-ingest a launched version forks/creates a draft first via
  the existing F2.1 flows.) This is the one place F2.4 deliberately diverges from
  the F2.1/F2.2/F2.3 fork-on-launched pattern.
- **Version-scoped dedup short-circuit.** Re-ingest's SHA-256 dedup is scoped to
  the **target version's** source documents, not global. Identical bytes already
  on this version → **200 no-op** (`{ deduped: true }`), change log returned
  unchanged, **no writes, no extractor call**. (The F1.1 _new-ingest_ dedup stays
  global/409 — its comment already anticipated that "F2.4 re-ingest will
  legitimately attach the same hash to a new version".)
- **What's wiped vs preserved.** Replace-in-place deletes the version's
  `sections` (cascades `slots` → cascades slot-tag assignments), its
  `changeRecords`, and its **tag vocabulary** (`AppQuestionTag`) — a fresh
  extraction is a clean slate. **Preserved:** the questionnaire `title`, the
  version row (number/status/id), and **prior source documents** (the new one is
  _appended_, keeping provenance history; newest is the active source).
- **Re-merge admin-wins-per-field.** Re-ingest re-resolves goal/audience through
  the existing `mergeGoalAudience` with the re-upload's admin-supplied fields, and
  rewrites `goal`/`audience`/`goalProvenance`/`audienceProvenance`.

## Build shape

### 1. Refactor — single-source the ingest pipeline (safety-netted by F1.1 tests)

The "bytes → validated extraction" stretch of `route.ts::handleIngest` is
identical for both flows. Extract it into `_lib/extract-pipeline.ts`, leaving the
**divergent** steps (dedup scope, persistence) in each handler:

- `parseAndGuardUpload(request) → { ok: true, file, buffer, fileHash, adminMeta, extractTables, parsed } | { ok: false, response }` — body-size guard, multipart parse, `file` field check, post-parse size check, extension allowlist, admin-metadata + extract-tables parse, SHA-256, `parseDocument`, scanned/empty detection. **No dedup, no dispatch** — those differ per flow.
- `dispatchExtraction({ parsed, file, adminMeta }) → { ok: true, extraction } | { ok: false, response }` — extractor-agent load, `capabilityDispatcher.dispatch`, dispatch-error → HTTP mapping (the existing `dispatchErrorStatus`/`dispatchErrorCode`), and `assertPersistable` coherence check.

Also factor the inner graph write shared by new-ingest and re-ingest:
`writeGraph(tx, versionId, extraction, merged)` (sections → slots → change records)
into `_lib/persist.ts`, consumed by both `persistIngestion` and the new
`reingestVersion`. `handleIngest` is rewired onto these helpers — its **22 existing
route integration tests are the regression net** that proves the refactor is behaviour-preserving.

### 2. Re-ingest persistence writer (`_lib/reingest.ts`)

`reingestVersion({ versionId, extraction, admin, source }) → PersistReingestionResult`
in **one `executeTransaction`**:

1. `deleteMany` change records for `versionId`.
2. `deleteMany` tag vocabulary for `versionId`.
3. `deleteMany` sections for `versionId` (cascades slots → slot-tag assignments).
4. `update` the version's `goal`/`audience`/`goalProvenance`/`audienceProvenance`
   from a fresh `mergeGoalAudience` (status/number unchanged).
5. `writeGraph(tx, versionId, extraction, merged)` — recreate sections, slots, change log.
6. `create` the new `AppQuestionnaireSourceDocument` (appended; prior docs kept).

Returns `{ versionId, sectionCount, questionCount, changeCount, goal, audience, fieldProvenance }`.

### 3. Route — `POST …/versions/:vid/reingest`

`app/api/v1/app/questionnaires/[id]/versions/[vid]/reingest/route.ts`. Pipeline
(order load-bearing):

> flag-gate → `withAdminAuth` → `ingestLimiter.check(adminId)` (reuse F1.1's
> per-admin 10/min sub-cap — re-ingest is equally a reasoning LLM call) →
> `loadScopedVersion(id, vid)` (404 on unknown/cross-id mismatch) → **status ===
> 'draft' else 409 `REINGEST_NOT_DRAFT`** → `parseAndGuardUpload` →
> **version-scoped dedup short-circuit** (200 `{ deduped: true }`, no writes) →
> `dispatchExtraction` → `reingestVersion` → `logAdminAction('questionnaire.reingest')`
> → **200** `{ versionId, sectionCount, questionCount, changeCount, goal, audience, fieldProvenance, deduped: false }`.

Flag gate in the `POST` export, `withAdminAuth` inside (matches F2.3). Endpoint
registered as `versionReingest(id, vid)` under `API.APP.QUESTIONNAIRES` in
`lib/api/endpoints.ts`.

### 4. UI — re-ingest control on the detail page

`app/admin/questionnaires/[id]/page.tsx`: a **Re-ingest** action on each `draft`
version (hidden/disabled for `launched`/`archived`). Opens a
`<ReingestDialog>` client component (co-located with the existing admin
questionnaire components) — file picker + optional goal/audience + extract-tables
toggle (FieldHelp ⓘ on each, per CLAUDE.md), behind a **destructive confirm**
("Replaces this version's structure, change log, and tags from the new document —
manual edits will be lost."). Reuses the ingest upload form pattern. On success:
refetch / route to the version graph; surface the `deduped` no-op result
("Identical document — nothing changed").

## Tests

- **Route integration** (`reingest-routes.test.ts`): gate order (404 flag-off),
  401/403, 404 (unknown questionnaire, unknown version, cross-id mismatch), **409
  non-draft** (launched + archived), size/format guards (413/400),
  parse/scanned/empty 422, **dedup short-circuit** (200, asserts _no_ new source
  doc + change log unchanged + extractor not called), **happy path** (old
  sections/slots/changes/tags gone, new graph present, new source doc _appended_,
  goal/audience re-merged with provenance), dispatch-error mapping (429/502/503),
  rate-limit 429, audit `questionnaire.reingest`.
- **Regression**: the existing ingest route + merge/persist unit suites stay green
  after the refactor (the behaviour-preserving guarantee).
- **Unit**: focused tests for the factored `writeGraph` and any pure logic in
  `parseAndGuardUpload`/`dispatchExtraction` not already covered.
- **No schema-shape test / no migration** — F2.4 adds no columns.

## Not in F2.4

Persisting raw upload `bytes` / diff-against-stored-source (no consumer; stays
deferred). Re-ingest of launched/archived versions (draft-only by decision).
Re-ingest as a _new version_ (replace-in-place by decision). P2.5+ demo tenancy
and all of P3.

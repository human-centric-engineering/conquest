# F6.3 — Cost cap enforcement at turn boundary

> Committable tracker for **F6.3**. Enforces the per-session USD budget
> (`AppQuestionnaireConfig.costBudgetUsd`) at the live `/messages` turn boundary: a soft
> wrap-up nudge at ≥90%, a hard 402 + auto-pause at ≥100%. Fires the `cost_cap_reached`
> hook F4.6 wired but never fired. Gated by `APP_QUESTIONNAIRES_ENABLED` +
> `APP_QUESTIONNAIRES_LIVE_SESSIONS_ENABLED` + a new `APP_QUESTIONNAIRES_COST_CAP_ENABLED`
> dark-launch sub-flag.

## Intent

F6.1 made the engine live but unmetered — a respondent (incl. the no-login anonymous path)
can drive turns until the questions run out, each one spending LLM cost. F6.3 puts a ceiling
on it: the session's accumulated spend is graded against its budget at each turn boundary, the
agent is nudged to wrap up as it nears the budget, and the turn is refused once it hits it.
It's the first feature to fire `recordCostCapReached` (reserved in F4.6 for exactly this).

## Decisions (confirmed with the user)

- **Include the cost-accuracy wiring.** The recorded per-turn `costUsd` only captured the
  offer-stream (+ adaptive selection) cost — extract/detect/refine invokers stubbed `0`, so a
  cap would rarely fire. F6.3 closes the F6.1-deferred per-turn aggregation: the three
  capabilities surface their real `costUsd` on their app-owned `data` (computed from the same
  `runStructuredCompletion` result already logged to `AiCostLog`), the invokers read it, the
  orchestrator sums it. **No platform `CapabilityResult` change** — the cost rides inside the
  app-owned data payload.
- **Soft cap biases toward offering early, not just a prose hint.** At ≥90% the orchestrator
  responds to a `not_ready` assessment (thresholds merely unmet, `answeredCount > 0`) as an
  offer instead of asking the next question — so the session winds down before the hard wall —
  **and** tags the offer prose with a wrap-up instruction. It never bypasses the
  required-questions gate (`blocked_on_required` stays authoritative) and never offers on an
  empty session.
- **Dedicated dark-launch sub-flag.** `APP_QUESTIONNAIRES_COST_CAP_ENABLED` (seed 023,
  disabled by default), ANDed with the master + live-sessions flags. Enforcement toggles
  independently of the live surface.
- **Reuse the existing field — no schema change.** The cap is the existing
  `AppQuestionnaireConfig.costBudgetUsd` (`Float?`, null = uncapped). No migration.
- **Turn-boundary semantics, summed turn cost.** "Spend so far" = `_sum` of prior
  `AppQuestionnaireTurn.costUsd` (the spend before this turn runs); the current turn's cost is
  recorded afterward and caught at the next boundary. No pre-run estimation.
- **Hard cap is pre-stream.** 402 + auto-pause before any per-turn work — the status gate then
  rejects every later turn. Soft fires its event once (deduped by an existing soft-tier event).

## Build shape — 1 PR

- **Part A — cost accuracy.** `extract-answer-slots.ts` / `detect-contradictions.ts` /
  `refine-answer.ts` add `costUsd` to their `*Data` and populate from `completion.costUsd`;
  `turn-invokers.ts` reads `dispatch.data.costUsd ?? 0`.
- **Part B — classifier.** `lib/app/questionnaire/session/cost-cap.ts` — `SOFT_CAP_RATIO`,
  `classifyCostCap(spentUsd, capUsd) → none|soft|hard` (re-exported from `session/index.ts`).
- **Part B — seams.** `sumSessionTurnCost(sessionId)` in `_lib/turns.ts`; `recordCostCapReached`
  detail gains `tier`; `hasCostCapReachedEvent(sessionId, tier)` dedupe helper in `_lib/sessions.ts`.
- **Part B — flag.** `APP_QUESTIONNAIRES_COST_CAP_FLAG` constant, `isCostCapEnforcementEnabled()`
  resolver, seed `023-cost-cap-flag.ts` (disabled).
- **Part B — orchestrator.** `TurnState.costPressure?: 'soft'` + `OfferComposeInput.costWrapUp?`;
  `orchestrator.ts` biases `not_ready → offer` under soft pressure (with the guards) and threads
  `costWrapUp`; `offer-stream.ts` appends the wrap-up system instruction.
- **Part B — route.** `[id]/messages/route.ts` pre-stream gate: hard → 402 + pause + event;
  soft → set `costPressure` + fire the soft event once.

## Tests

- **Unit** — `classifyCostCap` boundaries (null/zero/neg cap, just-below/at 90%, just-below/at
  100%, over; cent-scale budgets); orchestrator soft-bias (forces offer on `not_ready` + ≥1
  answered, not on empty, not over `blocked_on_required`; `costWrapUp` threaded; absent under no
  pressure).
- **Integration** — capability cost surfacing (extract returns `costUsd` on its data); invoker
  cost surfacing (extract/detect/refine read `data.costUsd`, fallback 0); `sumSessionTurnCost`
  (`_sum`, null→0); `recordCostCapReached` tier metadata + `hasCostCapReachedEvent` JSON-path
  query; offer-stream wrap-up prose; `/messages` matrix (cap null → no enforcement, below-soft →
  normal, hard → 402 + pause + hard event + no turn, soft → runs + soft event once + biased offer
  with `costWrapUp`, soft dedupe, sub-flag off → no enforcement over cap).

## Not in F6.3

- Admin UI for setting `costBudgetUsd` (lives in the existing version-config form, unchanged).
- Cost-cap analytics / alerts; F6.5 budget UX.
- A respondent-facing "you've hit the limit" UI (P7).

## Operator note

Cost enforcement is off until an operator enables `APP_QUESTIONNAIRES_COST_CAP_ENABLED` **and**
sets a `costBudgetUsd` on the version config. A breached session is auto-paused; an admin can
resume it via the F4.6 transition route (which resets nothing — the next turn re-checks the cap
and refuses again unless the budget is raised). The `cost_cap_reached` events (`tier: soft|hard`,
`spentUsd`, `capUsd`) are on the session's audit trail.

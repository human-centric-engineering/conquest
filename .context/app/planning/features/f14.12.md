# F14.12 — Version archiving (soft-archive + archive-on-fork prompt)

> Editing a launched version forks a fresh draft and leaves the old version behind,
> so over time a questionnaire's version list accumulates superseded versions with no
> way to tidy them. The ask: **each time a new version is created, offer to archive the
> current one.** This adds a per-version soft-archive (mirroring the F14.11
> questionnaire pattern) and surfaces the offer as an opt-in checkbox on the existing
> "Create a new draft version?" fork dialog.

## Confirmed decisions

- **Archive "the previous version", via a new `archivedAt` — not `status`.** The user
  chose to archive the version being superseded. The version lifecycle already has an
  `archived` **status**, but it's terminal (no restore) and **blocked while a launched
  version has live sessions/invitations** — exactly the state that triggers a fork — so
  it can't do the job. A nullable `archivedAt` on `AppQuestionnaireVersion`, orthogonal
  to `status` (the same shape F14.11 chose one level up), sidesteps both problems and
  stays restorable.
- **Trigger: the launched-edit fork only.** The offer rides the existing fork-confirm
  dialog; the (whole-questionnaire) Duplicate action was explicitly out of scope.
- **Archived versions are retired from respondents.** An archived version stops serving
  respondents even while its `status` is `launched`: every respondent entry point (all four
  session-create paths, in-flight turns, cross-device resume) refuses with a distinct
  `VERSION_ARCHIVED` (410) code, and both respondent surfaces — the no-login boot and the
  authenticated start page — render an "archived" notice. The admin **preview** path and
  `isPreview` turn sessions stay exempt. (This supersedes the initial
  "respondent behaviour unchanged" scoping — archiving is a real kill switch, not just a
  declutter.)
- **Idempotent, single-sourced writes.** Archive/restore are no-op `200`s when already
  in state; the fork path and the standalone routes share one writer.

## What shipped

- **Schema:** `archivedAt DateTime?` + `@@index([archivedAt])` on
  `AppQuestionnaireVersion`. Migration `20260717191031_app_version_archived_at`
  (additive `ALTER TABLE` + `CREATE INDEX`; generated `--create-only`, phantom pgvector
  `DROP INDEX` / `searchVector DROP DEFAULT` statements stripped by hand, applied via
  `migrate deploy`, all 5 vector indexes verified intact).
- **Server:**
  - New `_lib/version-archive.ts` — `setVersionArchived(vid, archived, audit)`: stamps/
    clears `archivedAt`, idempotent, audited `questionnaire_version.archive` /
    `.restore`. Single source for both the routes and the fork path.
  - `_lib/fork.ts` — reads a new `x-fork-archive-source` header; on the confirmed fork
    path, after the new draft commits, soft-archives the branched-from version. Absent
    for legacy/non-interactive callers, so the fork path is otherwise unchanged.
  - New routes `POST …/versions/:vid/archive` and `…/restore` (admin-only, scoped via
    `loadScopedVersion`). Endpoints `API.APP.QUESTIONNAIRES.versionArchive(id, vid)` /
    `versionRestore(id, vid)`.
  - Read path: `archivedAt` added to the `detail.ts` versions select + mapped into
    `QuestionnaireVersionSummary`.
- **Admin UI:**
  - Fork dialog (`launched-edit-confirm-dialog.tsx`): an opt-in **"Archive the previous
    version (v_n_)"** checkbox (default off) with a `<FieldHelp>` explaining it retires
    v_n_ from respondents (they see an archived notice), is reversible, and — since the new
    draft isn't launched yet — can leave the questionnaire unavailable until launch. The
    choice threads through the
    fork-confirm bridge/provider (`boolean` → `ForkConfirmChoice { confirmed,
archiveSource }`) and `authoringMutate` sends `x-fork-archive-source: true` on the
    confirmed retry.
  - Version picker (`workspace/version-selector.tsx`): archived versions hidden, except
    the one currently being viewed.
  - Overview timeline (`[id]/v/[vid]/page.tsx`): active versions each gain an **Archive**
    action; archived versions collapse into an **"Archived versions (_n_)"** group with a
    **Restore** action per row, via the new `useArchiveVersion` hook +
    `workspace/version-archive-button.tsx`.

## Design notes

- **Two "archived" notions on a version now coexist** — the terminal `status: 'archived'`
  lifecycle state and the orthogonal `archivedAt` marker. The admin version list keys its
  hide/show on `archivedAt`, and the respondent gate keys on `archivedAt` too (returning
  `VERSION_ARCHIVED`) — so `archivedAt` is the reversible, always-available way to take a
  version offline, whereas the `status` transition is terminal and blocked while it has live
  work. See `.context/app/questionnaire/archiving.md`.
- **Restart a running dev server** after the migration — a `dev` process started before it
  holds a stale Prisma client that doesn't know `archivedAt` (Next 16 blocks a second dev
  server). See memory `verify-after-migration-restart-dev`.

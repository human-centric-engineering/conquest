# F5.2 — Evaluation run

> Committable tracker for **F5.2**, the second feature of P5 (Design-time evaluation).
> Turns F5.1's ephemeral judge **preview** into a **persisted, synchronous run**: a POST
> runs the seven-judge panel over a version's structure, writes the run header + one row
> per finding in a single transaction, and returns the completed run. Adds an admin run
> history (list newest-first + read-only run detail). The findings it stores are the rows
> **F5.3's review queue** will mutate (accept / decline / edit / apply). Gated by
> `APP_QUESTIONNAIRES_ENABLED` + the `APP_QUESTIONNAIRES_DESIGN_EVALUATION_ENABLED` sub-flag.

## Intent

F5.1 shipped the judges and a no-persistence preview route — enough to tune the panel by
hand. F5.2 makes a run a durable artefact: the admin runs an evaluation, the result is
stored, and prior runs are browsable. This is the substrate F5.3 reviews. No new judging
logic — F5.2 is persistence + history on the F5.1 dispatch seam.

## Decisions (confirmed with the user at planning)

- **Synchronous run — no worker, no polling.** The POST runs the panel inline (the F5.1
  `Promise.all` fan-out) and writes the result before returning. Async (a queued run + a
  background worker + UI polling) was **explicitly considered and rejected**: the codebase
  has **no background-task registration seam** — the maintenance tick
  (`lib/orchestration/maintenance/run-tick.ts`) is a platform-owned hardcoded task array —
  so an app worker would force either a layering-inverting edit of a platform file (platform
  importing app, a merge-fight every upstream sync) or building a brand-new upstream seam,
  for no gain over the proven synchronous dispatch. The "status polling / live updates"
  indicative task was vestigial from the superseded `AiEvaluationRun` vision. `status` is
  kept a plain String holding a terminal value, so a future worker could add
  `running`/`queued` with **no migration**.
- **App-owned models, NOT a Sunrise `AiEvaluationRun` link.** The dev-plan's original F5.2
  sketch ("persist the link to the resulting `AiEvaluationRun`") is **superseded** — that
  worker grades a subject's output over a dataset and scores 0–1; P5 grades an existing
  artefact and emits suggestions. F5.2 persists app-owned `AppQuestionnaireEvaluationRun` +
  `AppQuestionnaireEvaluationFinding` on the F5.1 app-native seam (see the dev-plan decisions
  log, 2026-06-05).
- **Row-per-finding, not a JSON blob.** F5.3 mutates findings individually
  (accept/decline/edit/apply + stale derivation) and queries/filters/paginates them — that
  access pattern wants rows. The run header keeps a small `dimensionSummary` **JSON** array
  (per-dimension score/diagnostic, ≤7 entries, read wholesale) — that one is never queried
  per-dimension, so no per-dimension table.
- **Minimal finding `status` added now.** Each finding row carries `status` (default
  `pending`) in F5.2's migration so F5.3 **extends** rows rather than running a second
  migration. The heavier F5.3 columns (edited text, appliedAt, staleReason) are deferred
  until their shape settles. In F5.2 `status` only ever holds `pending`.
- **Shared dispatch service.** The F5.1 preview route's fan-out was extracted to
  `runEvaluationPanel(...)` (Prisma-free) so the preview route (ephemeral) and the new run
  route (persisting) share one dispatch path — no duplicated `Promise.all`/fail-soft logic.
- **POST sub-flag-gated; GETs read-only under the master flag.** The POST is paid LLM work,
  so it keeps F5.1's gating verbatim (sub-flag 404, the reused `designEvaluationLimiter`
  429, version-scope 404, not-configured 404). The two GETs serve persisted data, so they're
  master-flag + version-scope only — history stays readable if the sub-feature is later
  switched off (the read-only `changes`-list posture).
- **Reused the `designEvaluationLimiter`** (20/min) — a run is the same seven-call cost as a
  preview, so no new limiter.
- **No CHANGELOG entry.** ConQuest app surface, not the Sunrise platform public surface.

## Build shape — one branch (`feat/f5.2-evaluation-run`)

1. **Models + migration.** `AppQuestionnaireEvaluationRun` + `AppQuestionnaireEvaluationFinding`
   in `prisma/schema/app-questionnaire.prisma` (back-relation on `AppQuestionnaireVersion`).
   Migration via `--create-only`, phantom platform DDL hand-stripped (the four pgvector
   DROP INDEX + the `searchVector` ALTER — the documented app-migration footgun), then
   `db:drift-check` (all 11 probes pass — both tables fully Prisma-modelled, no new probe) +
   `db:generate`.
2. **Shared service.** Extract `lib/app/questionnaire/evaluation/run-panel.ts`
   (`runEvaluationPanel`); refactor `evaluate-preview/route.ts` to call it (net deletion).
3. **DB seam + routes.** `_lib/evaluation-run-routes.ts` (`persistEvaluationRun` —
   status derivation + ordinal-stable finding flatten + one `$transaction`;
   `listEvaluationRuns`; `getEvaluationRunDetail` — version-scoped; `dimensionSummary`
   Zod-validated on read). `…/versions/[vid]/evaluations/route.ts` (POST + GET list) and
   `…/evaluations/[runId]/route.ts` (GET detail). Endpoint constants
   (`versionEvaluations`, `versionEvaluationById`). View types in `views.ts`.
4. **Admin UI.** `evaluation-runs-table.tsx` (client; "Run evaluation" button +
   newest-first table), `evaluation-run-detail.tsx` (read-only findings grouped by
   dimension), `evaluation-status-badge.ts`; pages `…/[id]/evaluations/page.tsx` (runs,
   `?v=` selector mirroring `extraction-changes`) + `…/evaluations/[runId]/page.tsx`
   (detail, version via `?v=`). Sub-flag-gated "Evaluations" entry button on the version
   detail page.
5. **Tests + docs.** Schema-shape test extension, `run-panel` unit test, route integration
   test. `design-evaluation.md` § F5.2; dev-plan status + this tracker.

## Tests

- **Schema-shape** (`tests/unit/prisma/app-questionnaire-schema.test.ts`): both models'
  fields/types/relations via `Prisma.dmmf`; migration SQL asserts both `ON DELETE CASCADE`
  FKs, the six indexes, exactly-two `CREATE TABLE`, and the no-platform-DDL guard. (71 in
  the file.)
- **Unit — shared service** (`tests/unit/lib/app/questionnaire/evaluation/run-panel.test.ts`):
  fan-out (one dispatch per dimension), provider-binding pass-through, summary tallies, and
  the three fail-soft paths (`judge_not_configured`, `{ success:false }` code,
  `dispatch_error` on throw) + the `evaluation_failed` fallback. 6 cases.
- **Integration — routes** (`tests/integration/api/v1/app/questionnaires/evaluation-run-routes.test.ts`):
  POST gate order (master-off 404 before auth; sub-flag-off 404 after auth, no write),
  401/403, scope-404, not-configured-404, 429; persistence (derived `completed`/`partial`/
  `failed` status, per-judge finding rows, the detail response); GET list (scope-404,
  newest-first paged headers); GET detail (scope-404, run-not-found 404, findings payload).
  16 cases. The preview route's own test still passes (the refactor is behaviour-preserving).

## Operator note

Same opt-in as F5.1: `APP_QUESTIONNAIRES_DESIGN_EVALUATION_ENABLED` on top of
`APP_QUESTIONNAIRES_ENABLED`, plus `db:seed` (judges + capability + flag). Then in the admin
a version's **Evaluations** button (visible only when the sub-flag is on) opens the run
history; **Run evaluation** runs the panel synchronously and lands on the new run's detail.
A run takes the per-admin 20/min sub-cap; a flaky judge degrades to a per-dimension
diagnostic and the run is recorded `partial` (or `failed` if every judge fails) rather than
lost. History remains readable under the master flag even if the sub-flag is later disabled.

## Not in F5.2

The suggestion review queue — accept / decline / edit / apply and stale-suggestion
derivation at read time (F5.3). The finding `status` column exists but only holds `pending`.
No async worker / status polling (deliberately — runs are synchronous and terminal). No
per-run cost surfaced yet (`costUsd` is a reserved column).

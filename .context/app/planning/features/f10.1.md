# F10.1 — Respondent Report (report kind `respondent`)

> Committable tracker for **F10.1**. A per-respondent report delivered after a respondent completes a
> questionnaire — either raw answers, or raw answers **plus AI-generated, actionable insights**
> grounded (optionally) in a per-client knowledge base, branded and downloadable as PDF. The first of
> two report kinds; the cross-respondent **Cohort Report** (`cohort`) is later, separate work.

## Intent

After completing a conversational questionnaire, a respondent gets a personalised report. Admins
configure it per version from a dedicated **Respondent report** workspace tab. Insights are written by
a report agent that may draw on an **optional, per-client** knowledge base with strict no-bleed
isolation. Gated by the platform flag `APP_QUESTIONNAIRES_RESPONDENT_REPORT_ENABLED` (seeded disabled).

## Agreed scope (v1)

Modes **1 (raw)** + **2 (raw + insights)**; on-screen + PDF download; **async generation via the
maintenance-tick worker**; **per-client KB via tags** (no platform-schema fork); **flat
background-context blob**. Deferred: mode 3 (full narrative), email delivery, answer-conditioned
guidance, and the AI config-crafting chat / admin-interview assistant (Phase 4b).

## What shipped (by phase, one commit each)

1. **Foundation** — the `respondentReport` config slice on `AppQuestionnaireConfig` (mirrors the
   `tone` precedent: types + Zod + `narrowRespondentReportSettings` + read-view/PATCH/fork-copy);
   `AppRespondentReport` model (1:1 session, `onDelete: Cascade`, lease columns);
   `AppDemoClient.knowledgeTagId`; additive migration (pgvector strip). Also folded in the
   workspace tab + flag scaffolding.
2. **Per-client KB isolation (tag-based)** — `ensureClientKnowledgeTag` /
   `resolveClientKnowledgeDocumentIds` / `getClientKnowledgeViewForQuestionnaire`
   (`lib/app/questionnaire/report/client-knowledge.ts`) + app-side `GET …/:id/report/knowledge`.
   Reuses the platform `KnowledgeTag` machinery; no platform-schema fork.
3. **Agent + generation + async worker** — seeded `app-respondent-report` agent (045);
   `generateRespondentReport` (answers transcript → optional client-KB grounding → structured
   completion); `processQueuedRespondentReports` lease-worker in the maintenance-tick chain;
   `enqueueRespondentReport` submit-time trigger.
4. **Generation-tab config UI** — `RespondentReportEditor` (Content / Generation / Delivery /
   Appearance panels, saved whole via the config PATCH) + embedded `ClientKnowledgePanel`
   (client-scoped upload/list/delete).
5. **Respondent delivery** — `GET …/questionnaire-sessions/:id/report` status endpoint +
   `useRespondentReport` poller; `SessionComplete` renders insights inline (preparing/ready/failed)
   and gates the PDF on `delivery.download`; PDF gains a "Your insights" section (on-screen and
   download match).
6. **(4b) Config-crafting assistant** — `ReportConfigAssistant` chat in the Generation tab →
   `POST …/report/craft` → `craftReportConfig` (seeded `app-respondent-report-assistant` agent, 046);
   interviews the admin and proposes full `instructions`/`structure`/`backgroundContext` text the admin
   applies per-field. Stateless turn (mirrors the authoring refine pattern), per-admin rate-limited.

## Key decisions

- **Tag-based KB isolation, not a `demoClientId` column** — keeps the app from forking the Sunrise
  platform knowledge schema; isolation is enforced at query time via the client tag's documentIds.
- **Tick-worker generation, not `after()`** — crash-safe lease recovery, reuses the evaluations
  worker pattern; 0–60s latency is acceptable for an async report.
- **Generate-once-and-store** (mode 2 only) — `AppRespondentReport` caches content; raw mode renders
  deterministically on demand with no row.
- **`enabled` = config AND platform flag** everywhere a respondent could otherwise wait on a report
  the platform has turned off.

## Demo polish (post-ship)

Refinements to the completion/waiting experience, on `feat/completion-reveal-and-prep-visual`:

- **Completion reveal** — the confirmation card and the report (summary → sections → actions) ease
  in with the same restrained fade-and-rise the intro splash uses (`motion-safe` only), so the close
  of the conversation lands as a moment rather than an abrupt surface swap.
- **Shared-positions cycler** — while the AI report generates, the "preparing" state gently cycles
  the respondent's own captured positions (the data-slot paraphrases from the answer panel, or
  answered question slots when not in data-slot mode) instead of a bare spinner. Cross-fades one at a
  time; under `prefers-reduced-motion` it shows a short static list. Sourced from the last-settled
  `useAnswerPanel` view passed into `SessionComplete` via the new `captured` prop — no extra fetch.
- **Graceful poll timeout** — `useRespondentReport` now exposes `timedOut` (true once `MAX_POLLS`
  is exhausted without settling) and `retry`. The completion screen swaps the endless spinner for a
  calm "taking a little longer than usual" message + a "Check again" button, so a slow worker never
  strands the respondent on a spinner that will never resolve.

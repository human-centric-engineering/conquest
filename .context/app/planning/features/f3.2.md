# F3.2 — Invitation flow

> Committable tracker for **F3.2**, the **second feature of P3 (Configuration,
> invitations, cost estimation)**. Turns a launched questionnaire into something
> real respondents can be invited to and can register for — the end-to-end
> invitation lifecycle plus the first non-zero `countLaunchBlockers()`. Gated by
> `APP_QUESTIONNAIRES_ENABLED` (404 when off), API-first, riding the F2.1
> scoped-version + fork-on-launched seams.

## Intent

F3.1 made a questionnaire **launchable**. F3.2 adds the invitation lifecycle: an
admin invites respondents (single or bulk) to a launched questionnaire, the system
emails a tokenised link, the respondent opens it and **registers a real account**,
and the admin tracks every invitation's state. A sent invitation **pins** the
launched version it targets — the first feature to make `countLaunchBlockers()`
return non-zero, so editing a launched-with-invites version forks a draft.
Downstream: F3.4 themes the invitation per demo client; P6/P7 sessions consume the
registered respondent and drive the `started`/`completed` tail of the lifecycle.

## Decisions (at planning — confirmed)

- **Lifecycle reaches `registered`; `started`/`completed` are seam states.** F3.2
  builds the admin side completely (`pending → sent`, `revoked`) + the respondent
  landing (`opened`) + registration (`registered`). `started`/`completed` are
  enum values only — no transition logic until P6/P7 sessions exist (the same
  seam-now-consumer-later discipline F3.1 used for `countLaunchBlockers`).
- **Dedicated version-scoped model, not Sunrise's `Verification` store.** Sunrise's
  invite system is email-keyed singletons for platform users with roles — wrong
  storage for a per-version, lifecycle-bearing domain object. `AppQuestionnaireInvitation`
  FKs the **version** (`onDelete: Cascade`, like config) — the launched version at
  send time. In-flight invitees stay pinned when the version forks; new invitations
  target the now-launched version. Admin lists per-questionnaire through the version
  relation (`where: { version: { questionnaireId } }`) — no `questionnaireId`
  denormalisation. We reuse Sunrise _primitives_: the token pattern, `sendEmail`,
  `inviteLimiter`/`acceptInviteLimiter`, and the accept-invite signup machinery.
- **Identity = full better-auth account.** Accept reuses `app/api/auth/accept-invite`'s
  pattern (`auth.api.signUpEmail` → set `emailVerified` → `signInEmail` → forward
  Set-Cookie for auto-login) and stamps `AppQuestionnaireInvitation.userId`. An
  already-registered email returns a typed `409 ACCOUNT_EXISTS` (claim-via-login is
  deferred to P7).
- **Token at rest = SHA-256 hash.** Reuse the established pattern
  (`randomBytes(32).toString('hex')` + SHA-256). Store `tokenHash @unique`; plaintext
  only in the email URL. Email is **derived server-side** from the row (token-only link).
- **First app→User references.** `userId` (respondent, nullable) + `invitedByUserId`
  (admin) are plain `String`, no `@relation` (UG-1 — same as `revertedByUserId`). These
  are the first questionnaire rows that point at `User`; the erasure decision (does
  `eraseUser()` null the respondent `userId`?) is recorded in
  `.context/app/questionnaire/invitations.md` and verified against
  `.context/privacy/data-erasure.md`.
- **App-layer dedup, not a DB unique.** On send, an existing non-revoked, non-terminal
  invitation for `(versionId, email)` is skipped (resend regenerates) — mirrors
  Sunrise's `getValidInvitation` check. No `@@unique(versionId, email)` (revoke →
  re-invite must work).
- **Invite requires a launched version.** POST resolves the questionnaire's currently-
  launched version; `409 INVITE_NO_LAUNCHED_VERSION` if none (can't invite to a draft).
- **New app-owned email template.** `emails/questionnaire-invitation.tsx` — copy differs
  from Sunrise's platform-onboarding `emails/invitation.tsx`; F3.4 themes it.

## Build shape

### 1. Schema + migration

`AppQuestionnaireInvitation` in `prisma/schema/app-questionnaire.prisma`: `id`,
`versionId` (FK→version, Cascade), `email`, `name?`, `tokenHash @unique`,
`status @default("pending")`, `userId?` + `invitedByUserId` (plain String, UG-1),
`expiresAt`, lifecycle timestamps (`sentAt? openedAt? registeredAt? revokedAt?`),
`createdAt/updatedAt`; reverse `invitations` on the version. Indexes: `versionId`,
`status`, `email`, `userId`. Migration `<ts>_app_questionnaire_invitation` — the
**schema-fold strip** applied by hand (pgvector DROP INDEX / `searchVector` ALTER
removed; schema-guard test asserts they stay out).

### 2. Pure core (`lib/app/questionnaire/invitations/**`, Prisma-free)

- `types.ts`: `APP_INVITATION_STATUSES` const tuple + `AppInvitationStatus`
  (note `started`/`completed` are seam states), `INVITATION_TOKEN_EXPIRY_DAYS`.
- `token.ts`: `mintInvitationToken()` → `{ token, tokenHash, expiresAt }`,
  `hashInvitationToken(token)`. `crypto` only.
- `schemas.ts`: `createInvitationsSchema` (single email or `emails[]`, optional names;
  email format; in-batch dedup; max batch size).
- `status.ts`: transition table + `assertInvitationTransition()`.
- `views.ts`: `InvitationView` (never exposes `tokenHash`).
- `index.ts`: barrel.

### 3. Launch-blocker wiring (route-local, Prisma)

`app/api/v1/app/questionnaires/_lib/launch-blockers.ts`: Prisma-backed
`countLaunchBlockers(versionId)` → `{ invitations: <non-revoked, non-terminal count>,
sessions: 0 }`, re-exporting `hasLaunchBlockers` + the `LaunchBlockers` type from the
pure `lib/app/questionnaire/authoring/launch-blockers.ts`. The two callers
(`_lib/fork.ts`, `.../status/route.ts`) switch their `countLaunchBlockers` import to
the route-local file; the pure stub keeps the type + `hasLaunchBlockers`.

### 4. Email template

`emails/questionnaire-invitation.tsx` — props `{ inviteeName?, questionnaireTitle,
invitationUrl, expiresAt }`. App-owned (comment-marked); F3.4 adds theming props.

### 5. Routes — admin (`app/api/v1/app/questionnaires/[id]/invitations/`)

Flag-gate → `withAdminAuth` → validate → mutate → audit → response.

- `route.ts` **GET**: enriched list (status filter, pagination) → `InvitationView[]`.
- `route.ts` **POST**: resolve launched version (409 if none) → per email: app-layer
  dedup, `mintInvitationToken`, create row, `sendEmail`, `sent`/`pending` on failure →
  per-recipient result. `inviteLimiter`. Audit `questionnaire_invitation.create`.
- `[invitationId]/route.ts` **PATCH** revoke → `assertInvitationTransition`. Audit.
- `[invitationId]/resend/route.ts` **POST** regenerate token + resend. Audit.
- Endpoints registered in `lib/api/endpoints.ts`.

### 6. Routes — public respondent (`app/api/v1/app/invitations/`, token-gated)

Flag-gated, rate-limited, **no** `withAdminAuth`.

- `metadata/route.ts` **GET** `?token=`: hash→lookup; expired→410, invalid→404;
  hit → `{ questionnaireTitle, inviteeName, status }` + transition `sent→opened`.
- `accept/route.ts` **POST** `{ token, password, name? }`: lookup → email from row →
  `signUpEmail` → `emailVerified` → set `userId`/`registered` → `signInEmail` (forward
  cookies). Email-taken → `409 ACCOUNT_EXISTS`. `acceptInviteLimiter`.

### 7. UI

- **Admin** — sub-route `app/admin/questionnaires/[id]/invitations/page.tsx` (mirrors
  F2.3 extraction-changes). Components: `invitations-table.tsx`, `invite-form.tsx`
  (single + bulk paste), `invitation-status-badge.tsx`; `<FieldHelp>` on non-obvious
  fields; link from the detail page.
- **Respondent** — public token-gated landing + set-password page in the route group
  Sunrise's `/accept-invite` page uses; posts to `/api/v1/app/invitations/accept`.

## Tests

- **Schema guard** (`tests/unit/prisma/app-questionnaire-schema.test.ts`): dmmf block
  (map, fields, `tokenHash @unique`, version relation + reverse) + migration-SQL block
  (FK Cascade, indexes, unique index, no-platform-DDL guard).
- **Unit** (`tests/unit/lib/app/questionnaire/invitations/*.test.ts`): schema single/bulk
  - in-batch dedup + max-batch + email format; status transitions (legal/illegal,
    `started/completed` unreachable); token hash stable & `token !== hash`; `InvitationView`
    excludes `tokenHash`.
- **Integration — admin** (`tests/integration/api/v1/app/questionnaires/invitations*.test.ts`):
  gate order (404 flag-off before auth, 401/403), POST send + bulk per-recipient result,
  409 no-launched-version, dedup skip, rate limit, audit; GET filter/pagination; revoke;
  resend. **Launch-blocker**: status route refuses un-launch with a live invite; fork-on-edit
  forks when an invite exists.
- **Integration — public** (`tests/integration/api/v1/app/invitations/*.test.ts`): metadata
  valid→`opened`, expired→410, invalid→404; accept creates user + sets `userId`/`registered`
  - forwards cookies, email-taken→409.

## Sequencing — 2 PRs (risk-based, one `/security-review` each)

- **PR1 — Admin invitation management + launch-blocker wiring.** Schema + migration +
  pure core + email template + admin routes (GET/POST/revoke/resend) + `countLaunchBlockers`
  wiring + admin UI sub-route + tests. Structured commits: (1) schema+migration; (2) pure
  core; (3) launch-blocker wiring; (4) email + admin routes; (5) admin UI; (6) tracker+tests.
- **PR2 — Respondent registration.** Public metadata + accept routes + landing/registration
  page (reusing accept-invite machinery) + tests.

Gates each PR: `/pre-pr`, `/security-review`, `/code-review`, `/test-review`.

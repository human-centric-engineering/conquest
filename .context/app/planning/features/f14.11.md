# F14.11 — Questionnaire archiving (soft delete + restore)

> There was no way to remove a questionnaire from the admin list. The `[id]` route
> exposed only `GET`/`PATCH` (rename + demo-client attribution) — no delete of any
> kind, hard or soft — so a finished, mistaken, or superseded questionnaire sat in
> the list forever. This adds a **reversible** removal: archive hides it, restore
> brings it back in its exact prior state. Nothing is ever destroyed.

## Confirmed decisions

- **Soft delete, not hard delete.** A dedicated nullable `archivedAt` timestamp on
  `AppQuestionnaire`, **orthogonal to `status`** (`draft | launched | archived`).
  Archiving never touches `status`, versions, sessions, or any other row, so restore
  is just "clear the marker" and the questionnaire returns exactly as it left.
- **Not the `status='archived'` value.** The questionnaire-container `status` field
  is effectively inert (only ever `draft`), and overloading its `archived` value
  would (a) collide with the version-level `archived` lifecycle status and (b) lose
  the "restore to prior state" property. A separate dimension keeps both concepts
  clean. (The user chose the **"Archive / Restore"** wording over "Delete / Trash".)
- **Idempotent endpoints.** Archiving an already-archived (or restoring an
  already-active) questionnaire is a no-op `200` — no second write, no duplicate
  audit — mirroring the rename no-op on the same route.
- **Archived tile made real.** The previously-inert "Archived" stat tile (it counted
  the never-set `status='archived'`) now counts soft-deleted questionnaires.

## What shipped

- **Schema:** `archivedAt DateTime?` + `@@index([archivedAt])` on `AppQuestionnaire`.
  Migration `20260717131222_app_questionnaire_archived_at` (additive `ALTER TABLE` +
  `CREATE INDEX`; generated `--create-only`, phantom pgvector `DROP INDEX`/`DROP
DEFAULT` statements stripped by hand, applied via `migrate deploy`, all 5 vector
  indexes verified intact).
- **API:**
  - `DELETE /api/v1/app/questionnaires/:id` → archive (stamp `archivedAt`), audited
    `questionnaire.archive`. Added to the existing `[id]/route.ts`.
  - `POST /api/v1/app/questionnaires/:id/restore` → restore (clear `archivedAt`),
    audited `questionnaire.restore`. New `[id]/restore/route.ts`. Endpoint added as
    `API.APP.QUESTIONNAIRES.restore(id)`.
  - `_lib/list.ts` gains an `archived` string-enum (`'true' | 'false'`) query param.
    Every list read applies an `archivedAt` gate: default → `null` (active only);
    `true` → `{ not: null }` (archived slice). `archivedAt` added to the row select +
    to `QuestionnaireListItem`; the detail read model + `QuestionnaireDetail` carry
    it too.
- **Admin UI:**
  - New shared `useArchiveQuestionnaire` hook (`archive` / `restore`, sibling to
    `useDuplicateQuestionnaire`).
  - `QuestionnairesTable`: an **Active / Archived** view toggle (independent of the
    status filter; independent pagination), an **Archive** row action guarded by a
    confirmation `AlertDialog`, and a **Restore** row action in the Archived view.
    On success the table refetches the current page and `router.refresh()`es so the
    stat tiles update.
  - List page (`page.tsx`): the "Archived" stat tile now reads a 1-row
    `?archived=true` sweep's meta `total`.

## Design notes

- **Defense in depth via the default list.** Because the default list — and any
  picker built on it — excludes archived rows, an archived questionnaire drops out
  of the surfaces that offer it for new work without touching those call sites.
- **Deeper enforcement is out of scope.** Blocking a respondent from _starting a new
  session_ on a questionnaire that is archived while a launched version still exists
  is a future guard; archiving today is an admin-surface hide + restore.
- **Note for a running dev server:** a `dev` process started before the migration
  holds a stale Prisma client and won't know `archivedAt` — restart it to exercise
  archive/restore live (Next 16 blocks a second dev server). See memory
  `verify-after-migration-restart-dev`.

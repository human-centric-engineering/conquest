# F2.1 — Questionnaire authoring

> Committable tracker for **F2.1**, the keystone feature of **P2 (Admin CRUD over
> questionnaires)**. The first admin UI for the platform: list, view, and (PR2)
> edit questionnaires, with the version-fork-on-launched lifecycle. Reconciled
> against Sunrise's actual admin seams (nav registry, `withAdminAuth`,
> `logAdminAction`, the `serverFetch`→client-table pattern). Flip statuses as PRs
> land.

## Intent

Give an admin a place to **live** in the platform: browse every ingested
questionnaire, drill into its versions and full section/question graph, and edit
its structure — with edits to a launched version forking a new draft (in-flight
sessions stay pinned). The whole surface is gated by `APP_QUESTIONNAIRES_ENABLED`
(404 when off), API-first (every read/write is an endpoint before it is a page).

## PR shape (gate-suite-tax calibration)

F2.1 is a **keystone**, split into **two PRs** — the read surface carries no
high-stakes mutation, so it takes one clean review pass; the write surface
(mutation + version-fork correctness) justifies its own `/security-review`.

| PR      | Title                               | Scope                                                                                               | Status   |
| ------- | ----------------------------------- | --------------------------------------------------------------------------------------------------- | -------- |
| **PR1** | Read surface + admin shell          | nav entry · list/detail/version-graph GET APIs · read-only list + detail UI                         | **done** |
| **PR2** | Structural authoring + version-fork | edit/add/remove/reorder sections & questions · goal/audience edit · `forkVersionIfLaunched` · audit | **done** |

> Editor depth (confirmed): **full structural authoring** in PR2.
> Version fork (confirmed): **build full fork mechanics now**, behind a
> `countLaunchBlockers()` seam returning 0 until P3 (invitations) / P4 (sessions).
>
> **PR2 decisions (at build):**
>
> - **Reorder = drag-and-drop** via `@dnd-kit` (added as an app dependency).
> - **Manual edits are `logAdminAction`-only** — they do _not_ write
>   `AppQuestionnaireExtractionChange` rows (that log stays the extractor's editorial
>   provenance, F2.3's consumer). A **fork starts a clean editorial lineage** — it
>   deep-copies structure but not change records.
> - **Status transitions (launch/archive) pulled into PR2.** Originally parked in P3;
>   bringing the lifecycle flip forward makes the fork seam **live** (editing a
>   `launched` version actually forks) rather than dormant. The _full_
>   config-completeness launch gate (F3.1) and invitations/cost estimation stay P3 —
>   PR2 ships only the status flip with a minimal launch guard (goal + ≥1 section +
>   ≥1 question). `countLaunchBlockers` remains a zero-returning P3/P4 seam.
> - **Delete cascades** (FK is `onDelete: Cascade`) with a UI confirm.

## PR1 — Read surface + admin shell (done)

**APIs** (`app/api/v1/app/questionnaires/`):

- `GET /` — paginated, enriched list. Latest-version + section/question counts
  computed in a **fixed query budget** (1 findMany + 1 count + 2 `groupBy`
  sweeps), never per-row. Query: `page`, `limit`, `q` (title), `status`,
  `sortBy`, `sortOrder`. Read model: `_lib/list.ts`.
- `GET /:id` — questionnaire + newest-first version summaries (counts). Read
  model: `_lib/detail.ts`.
- `GET /:id/versions/:vid` — one version's full section→question graph, scoped by
  **both** ids (mismatched pair 404s). Read model: `_lib/detail.ts`.

All three: flag-gate first (404 off) → `withAdminAuth` (401/403) → 404 on unknown
id. View contracts (client-safe, ISO dates) in `lib/app/questionnaire/views.ts`.

**Provenance is stored per field.** The ingest merge's admin-wins-per-field
outcome is persisted on the version (`goalProvenance` + `audienceProvenance`
columns, added in migration `…_app_questionnaire_version_provenance`); the
version-graph endpoint reads them straight back and the UI marks each inferred
field. (An earlier read-time derivation from `infer_*` change records was replaced
by these columns during PR1 review — storing the real provenance is the correct
data model, and gives per-audience-field granularity the change-log heuristic
couldn't.)

**UI:**

- Nav entry registered in `lib/app/admin-nav.ts` via `registerNavSection()` (seam
  4 — no sidebar edit). Always present; pages 404 when the flag is off.
- `app/admin/questionnaires/page.tsx` (+ `QuestionnairesTable`) — list with
  debounced title search, status filter, pagination, row click-through.
- `app/admin/questionnaires/[id]/page.tsx` (+ `VersionGraph`) — read-only detail:
  SSR `?v=` version selector, goal/audience with `inferred` badges, section→
  question tree. Edit affordances deferred to PR2.

**Endpoints** registered under `API.APP.QUESTIONNAIRES` in `lib/api/endpoints.ts`.

**Schema (PR1):** migration `…_app_questionnaire_version_provenance` adds
`goalProvenance` + `audienceProvenance` to `app_questionnaire_version` (phantom
DDL stripped per the schema-fold recipe; drift-check green).

**Drift fix (PR1):** the status vocabulary is now the single-source
`APP_QUESTIONNAIRE_STATUSES` tuple in `types.ts` — the type, the route's Zod query
enum, and the UI filter options all derive from it (no more hard-coded duplicate
lists). `FieldProvenance` moved from the route `_lib/merge.ts` into the domain
`types.ts` so the client-safe view contract doesn't import route internals.

**Tests:** route integration (gate order, 401/403, 404, envelope, query parsing) +
unit (`listQuestionnaires` fixed-query-budget/no-N+1 + mapping;
`getQuestionnaireDetail`/`getVersionGraph` column-backed provenance, null/404,
version scoping) + persist (provenance written) + schema-shape (new columns +
phantom-strip guard). `npm run validate` + full questionnaire suite (178) green.

**Not in PR1:** any mutation, the fork service, tagging (F2.2), change review
(F2.3), re-ingest (F2.4). No CHANGELOG entry (app surface, not Sunrise platform).

## PR2 — Structural authoring + version-fork (done)

**Pure authoring core** (`lib/app/questionnaire/authoring/`, Prisma-free):

- `launch-blockers.ts` — `countLaunchBlockers(versionId)` (returns
  `{sessions:0, invitations:0}`; P3/P4 seam) + `hasLaunchBlockers`.
- `key.ts` — `slugifyKey` + `nextAvailableKey` (per-version `key` derivation +
  collision suffixing).
- `type-config-schema.ts` — per-type `typeConfig` Zod + `validateTypeConfig`
  (choices ≥2 distinct · likert max>min · numeric bounds · config-less types).
- `schemas.ts` — request-body schemas (`updateVersionMeta`, `updateVersionStatus`,
  section/question create/update, `reorder`). Enums derive from the `types.ts`
  tuples; audience shape reused from ingestion. Provenance is **not** client-sent —
  the route stamps `admin-supplied` server-side.

**Fork seam** (route-local DB seam, `_lib/`):

- `fork.ts` — `forkVersionIfLaunched` deep-copies goal/audience/provenance +
  section→question graph into a new `draft` when the target is launched (or a
  blocker is live), preserving ordinal/key/typeConfig. Returns old→new id maps so
  child mutations retarget the copy. Tags **not** copied (F2.2 seam); change records
  **not** copied (clean lineage). Audited as `questionnaire_version.fork`.
- `authoring-routes.ts` — `loadScopedVersion` (scope-404), `resolveForkedId`,
  `forkMeta`, `asKeyConflict` (P2002→400), `resolveQuestionKey`, `applyReorder`
  (permutation guard), `adminAudienceProvenance`, `jsonInput`.

**Mutation APIs** (all: flag-gate → `withAdminAuth` → scope-404 → fork preamble →
validate → tx → P2002 map → `logAdminAction` → `successResponse(data, {forked,…})`):
`PATCH …/versions/:vid` (meta) · `PATCH …/versions/:vid/status` · section
`POST`/`PATCH`/`DELETE`/`reorder` · question `POST`/`PATCH`(incl. move)/`DELETE`/
`reorder`. Registered under `API.APP.QUESTIONNAIRES` in `lib/api/endpoints.ts`.

**Editor UI** (`?edit=1` over the PR1 detail view): `version-editor.tsx` +
`section-editor.tsx` + `question-editor.tsx` — `@dnd-kit` drag-reorder, inline
edits, per-type config editor, `<FieldHelp>` on goal/audience, status controls,
`router.refresh()` refetch, and the "edited a launched version → new draft (vN)"
fork notice + redirect. Meta-aware client helper `authoring-mutate.ts` (exposes
the fork `meta` that `apiClient` discards).

**Tests:** fork decision matrix (`fork.test.ts`) · key/typeConfig/schemas/
launch-blockers units · route integration (gate order, scope-404, provenance
stamp, status legality + launch guard, typeConfig 400, key-conflict 400, audit).
Questionnaire suite 256 green (PR1 was 178; +78). `npm run validate` clean.

**Not in PR2:** tagging (F2.2), extraction-change review (F2.3), re-ingest (F2.4),
the full config-completeness launch gate + invitations/cost (P3/F3.1). No CHANGELOG
entry — app surface, not the Sunrise platform public surface (consistent with PR1).

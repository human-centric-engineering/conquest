# F3.1 — Questionnaire configuration

> Committable tracker for **F3.1**, the **first feature of P3 (Configuration,
> invitations, cost estimation)**. Adds the per-version run-time configuration an
> admin authors before launch — every knob that controls how a session runs — plus
> a tightened launch gate. Makes a questionnaire **launchable**. Gated by
> `APP_QUESTIONNAIRES_ENABLED` (404 when off), API-first, riding the F2.1
> scoped-version + fork-on-launched seams.

## Intent

F2.1–F2.4 let an admin author a questionnaire's _content_ (goal, audience,
sections, questions, tags) but nothing about _how a session runs_. F3.1 adds the
`AppQuestionnaireConfig` editor — selection strategy, completion thresholds,
budget/caps, voice/contradiction/anonymous modes, and the session-start profile
fields collected from each respondent — and extends the launch gate so a version
can only go live once it is genuinely ready. Downstream phases consume the stored
settings: F3.3 (cost estimation) reads the budget, P4 (sessions) reads selection /
thresholds / profile fields, F6 reads voice / contradiction / cost-cap.

## Decisions (at planning — confirmed)

- **Dedicated 1:1 model, lazy.** `AppQuestionnaireConfig` is its own table, one row
  per version (`versionId @unique`, `onDelete: Cascade`), so it forks with the
  version. **Lazy materialization:** no row at ingest/version-create — the read
  path resolves an absent row to `DEFAULT_QUESTIONNAIRE_CONFIG`, the first PATCH
  upserts it, the fork copies it only when present. This keeps the F1.1 ingest path
  and the no-config fork path byte-for-byte unchanged.
- **Profile fields as JSON, not a relational model.** The session-start profile
  fields are an ordered `ProfileFieldConfig[]` (`{ key, label, type, required,
options? }`) stored in a `profileFields` JSON column — a small, admin-authored,
  version-scoped list edited as one unit and read wholesale at session start, never
  queried field-by-field (the same shape precedent as `audience`/`typeConfig`). A
  separate model would pull in its own CRUD/reorder routes, fork-copy loop, and
  schema-guard block — a second PR. Field-type enum is `text | email | number |
select`; `select` requires a non-empty distinct `options` list.
- **Launch gate = goal + audience + ≥1 section + ≥1 question + saved config.**
  Extends F2.1's minimal gate (goal + section + question) with **audience** and a
  **saved config row**. Because every config setting has a default, "config
  populated" can't mean "has values" (it always does) — it means **a config row
  exists** (the admin opened and saved the editor at least once, even with all
  defaults). That's the deliberate readiness opt-in. An empty-`{}` audience counts
  as not-populated (the editor may persist `{}`).
- **Fork-on-launched compatible.** Editing config on a `launched` version forks a
  fresh draft (the F2.1 seam); the fork preamble copies the existing config row
  into the draft, then the upsert edits the copy. Audited as
  `questionnaire_config.update`.
- **UI inline in the editor.** The config editor is a collapsible **Configuration**
  section inside the existing `VersionEditor` (edit mode) — no new page, route, or
  SSR fetch. The config rides the `VersionGraphView` the detail page already
  fetches. (Confirmed with product: inline over a dedicated sub-route.)

## Build shape

### 1. Schema + migration

`AppQuestionnaireConfig` in `prisma/schema/app-questionnaire.prisma` (1:1 with
`AppQuestionnaireVersion`, cascade): typed scalar columns
(`selectionStrategy`, `minQuestionsAnswered`, `coverageThreshold`,
`costBudgetUsd?`, `maxQuestionsPerSession?`, `voiceEnabled`, `contradictionMode`,
`contradictionWindowN`, `anonymousMode`) + `profileFields Json @default("[]")`, and
a reverse `config AppQuestionnaireConfig?` on the version. Migration
`20260604090526_app_questionnaire_config` — the **schema-fold strip** applied by
hand (re-emitted pgvector DROP INDEX / `searchVector` ALTER removed, same as every
prior app_questionnaire migration; schema-guard test asserts they stay out).

### 2. Pure core (`lib/app/questionnaire/**`, Prisma-free)

- `types.ts`: `SELECTION_STRATEGIES`, `CONTRADICTION_MODES`, `PROFILE_FIELD_TYPES`
  `const` tuples + types, `ProfileFieldConfig`, `QuestionnaireConfigShape`,
  `DEFAULT_QUESTIONNAIRE_CONFIG`.
- `authoring/config-schema.ts`: `profileFieldSchema` (select/options + the
  per-field rules) and `updateConfigSchema` (all-optional partial; at-least-one
  refine; contradiction mode/N coherence + unique profile-field keys via
  `superRefine`). Re-exported from `authoring/index.ts`.

### 3. Read-view wiring

`views.ts`: `ConfigView` (= `QuestionnaireConfigShape` + `saved: boolean`) on
`VersionGraphView`. `_lib/detail.ts`: `toConfigView(row | null)` returns defaults
with `saved: false` when absent, narrows enum strings via the tuples; selected into
`getVersionGraph` so the config rides the existing single fetch.

### 4. Route — `PATCH …/versions/:vid/config`

`config/route.ts`, PATCH only (read is via the version graph). Order:
flag-gate → `withAdminAuth` → `loadScopedVersion` (scope-404) →
`validateRequestBody(updateConfigSchema)` → `forkVersionIfLaunched` preamble →
read `before` → `upsert` (create-with-defaults / update) →
`logAdminAction('questionnaire_config.update')` + `computeChanges` →
`successResponse(toConfigView(row), forkMeta(fork))`. Registered as
`versionConfig(id, vid)` in `lib/api/endpoints.ts`.

### 5. Fork copy

`_lib/fork.ts`: select the config relation on `source`; after the new version is
created, `create` a copied config row when `source.config` exists.

### 6. Launch gate

`assertLaunchable()` in `…/status/route.ts`: add `audience` (via a `hasAudience`
helper) and `appQuestionnaireConfig.count` to the `Promise.all`; extend the
`missing` map with `audience` and `config` keys.

### 7. UI

`components/admin/questionnaires/config-editor.tsx` — plain controlled state + the
`run` runner (matches the surrounding editor tree; not react-hook-form), `Card`/
section layout, `<FieldHelp>` ⓘ on every non-obvious field, `Switch`/`Select`
primitives, a dynamic add/remove profile-fields list with a conditional options
input for `select`. Rendered as a section inside `version-editor.tsx`; fork
notice + redirect handled by the shared `run`.

## Tests

- **Schema guard** (`tests/unit/prisma/app-questionnaire-schema.test.ts`): a
  `Prisma.dmmf` block for `AppQuestionnaireConfig` (map, field types incl.
  `profileFields → Json`, the `version` relation + reverse `config` field) and a
  new `app_questionnaire_config migration SQL` block (one `CREATE TABLE`, the
  `versionId` FK `ON DELETE CASCADE`, the unique index, no-platform-DDL guard).
- **Unit** (`config-schema.test.ts`): valid full + partial PATCH; contradiction
  mode/N coherence (reject `flag` with N=0, accept N>0, reject N>0 when off);
  profile-field unique-key, select-requires-options, non-select-forbids-options;
  bad enum; coverage-threshold bounds.
- **Integration** (`config.test.ts`): gate order (404 flag-off before auth),
  401/403, scope-404, fork preamble + `meta.forked`, upsert create-vs-update,
  audit emission.
- **Launch gate** (extend the status-route integration test): missing audience /
  missing config → 400 with the right `missing` keys; all present → launch
  succeeds.

## Sequencing — ~1 PR

Single PR (the plan sizes F3.1 at ~1–2 PRs; held to 1 by JSON profile fields +
lazy materialization + inline UI). Structured commits: (1) schema + migration;
(2) pure core (types + config schema); (3) read-view + route + fork + launch gate;
(4) UI; (5) tracker + tests. Gates: `/pre-pr`, `/security-review` (a new
authenticated mutation surface), `/code-review`, `/test-review`.

## Not in F3.1

Pre-launch **cost estimation** (stores `costBudgetUsd` only — estimation is
**F3.3**). Any **consumer** of the settings — selection (F4.1), contradiction
(F4.3), voice (F6.2), cost-cap enforcement (F6.3), profile-field collection at
session start (P4). A separate relational profile-field model (JSON by decision).
Invitations (F3.2) and demo-client invitation branding (F3.4).

# F9.6 — Sensitivity awareness / safeguarding

> Committable tracker for **F9.6**. When a respondent discloses something sensitive or contentious
> (abuse, distress, safeguarding), detect it that turn, remember it at the session level, phrase
> every later question more gently, and gently signpost support once on a serious disclosure.
> Mirrors the seriousness gate (F9.5) end-to-end.

## Intent

The interviewer only had a static, author-declared `audience.sensitivity` line — it didn't notice
what was _just said_, didn't remember it, and asked the next question with the same neutral tone.
F9.6 makes the conversation tread carefully after a disclosure, and surfaces a lightweight
safeguarding signal to admins.

## Decisions (confirmed with the user)

- **Detection folds into the answer-extraction call** (no new always-on LLM call).
- **Admin = a count in the existing analytics tab + a session event** (no full session viewer yet).
- **Respondent UX = soften tone AND signpost support** (a configurable, verbatim support message).
- Explicitly **best-effort awareness, not a guaranteed safeguarding net** (detection rides on the
  extractor's structured side-signal).

## Addendum (2026-06-16) — detection reliability: defence-in-depth

**Why:** In a preview session the extractor dropped its optional `sensitivity` field on a busy
data-slot turn, so a textbook disclosure ("i'm being abused by my manager") went unflagged — no
support signpost, no tone memory. Confirmed from the turn log: `app_assess_seriousness` ran (which
only happens when `!extractedSensitivity`), `sensitivityLevel` stayed null, no `support` warning.
The "folded into extraction (no extra call)" decision was the root cause — the same failure mode the
seriousness gate's `suspectedNonGenuine` hint showed. **This reverses the original "Not in scope:
two-stage dedicated judge" decision** — safeguarding is too important to ride on one optional field.

**What changed:** detection is now **three merged signals** (`mergeSensitivitySignals`, strongest
severity wins; LLM beats keyword net on a tie):

1. the answer-extractor field (kept, one cheap signal);
2. a **dedicated detector** — a single-purpose structured LLM call run every answered turn the
   feature is on (`sensitivity/{detect-prompt,detect-schema}.ts`, `invokers.detectSensitivity`,
   tool slug `app_detect_sensitivity`);
3. a **deterministic keyword net** (`sensitivity/keyword-net.ts`) — a non-LLM floor for first-person
   harm / self-harm phrasing, tuned against obvious false positives.

The merge is orchestrator **step 1.4**, before the seriousness gate, so the gate's
`!extractedSensitivity` guard sees the combined signal. Cost: one extra cheap call per answered turn
when the feature is on (roughly neutral on disclosure turns, where the seriousness judge is skipped).

**Follow-up — context-scoping:** the first cut of the dedicated detector was fed recent conversation
and then over-flagged _later_ pure abuse ("oh just fuck off" after a disclosure) as a fresh
disclosure, which silently skipped the seriousness gate again. Fixed by scoping both LLM detectors
(dedicated + extractor block) to the **current message** — recent context is for reading an oblique
message, never for carrying the flag forward. Verified live against the dev server: the disclosure
turn skips the gate + signposts; the two following abuse turns each run the judge and strike
(`abuseStrikes` reaches 2), while `sensitivityLevel` stays `high` so the tone remains gentle.

**Sibling fix (F9.5):** the seriousness judge prompt was also hardened so that pure abuse aimed at
the interviewer after an earlier disclosure ("go fuck yourself" following a real disclosure) is still
struck — the safeguarding rule protects a disclosure of harm, not blanket immunity for later abuse.

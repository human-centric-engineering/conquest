# F14.15 — AI run provenance

> An audit of every place the system runs an LLM to **judge, grade, verify, or generate something
> durable** found ~30 surfaces. The codebase already contains an excellent run-capture pattern —
> `AppQuestionnaireTurn.inspectorCalls` and `AppQuestionnaireTurnEvaluation`, which snapshot prompt,
> model, tokens, cost, rubric version, and the exact evaluated input. **That pattern is applied only
> to the respondent conversation path.** Every authoring, report, and config-advising surface
> persists the _artifact_ and discards the _run_. Two of those surfaces are worse than lossy: one
> silently destroys an existing audit log, and one makes a statistical claim from a comparison it
> never actually ran.

## The decision rule

A run is **preserved** if it satisfies any of:

- **(a) A human later acts on its verdict** — evaluation findings, critic flags, advisor suggestions.
- **(b) It changed durable config** — anything that mutates a questionnaire version's structure.
- **(c) You'd need to defend the output to a client** — reports, scoring, extraction fidelity.
- **(d) It's a calibration signal you want a trend on** — judge scores, coverage, cost per artifact.

Everything else is **ephemeral by design** and stays that way. Preserving interactive previews and
mid-workflow control flow would add cost and noise without adding an answerable question.

### Explicitly ephemeral — not changed by this feature

`evaluate-preview` (design-eval dry run), the `judge_call` / `evaluate` / `guard` / `reflect`
workflow step executors (control flow, already captured in `executionTrace`), the input/output
guards (regex, no LLM call), the golden-set extraction scorer (`scripts/eval/extraction.ts`, a dev
script), and the contradiction / abuse / sensitivity detectors (aggregate state on the session is
the correct granularity — the per-call reasoning is not something anyone acts on).

## Confirmed decisions

- **One `AppAiRun` table, not per-surface capture.** Six surfaces need durable runs. Six bespoke
  columns/tables would drift. A single polymorphic run table with `subjectType` / `subjectId` gives
  one query surface and one retention rule.
- **`AppAiRun` is additive and best-effort at the call site, but never silently lossy.** Capture
  failure must not fail the user's action — but it must log at `error` and set a `status` the UI can
  show, not vanish. This is the mistake `evaluate-turn` currently makes.
- **Platform references stay plain `String`, no FK** — the established UG-1 convention (see
  `AppQuestionnaireTurnEvaluation`). App→platform Prisma relations would fight every upstream sync.
- **Supersede, don't delete, the extraction change log.** Revert is a user-facing feature; a rewrite
  must not silently disable it. Existing rows get marked superseded and stay queryable.
- **Cost accounting unifies on `logCost`.** A second, invisible spend path is worse than no
  tracking, because the budget enforcement docs claim coverage that doesn't exist.
- **Version-pin every evaluation.** A score you can't attribute to a config version is not a
  measurement. `AiWorkflowExecution.versionId` already establishes the pattern.

## Phases

### Phase 1 — Live bugs (correctness, no schema change)

1. **`replaceVersionStructure` destroys the extraction audit log.** `persist.ts:372` runs
   `deleteMany` on `appQuestionnaireExtractionChange` before rewriting the graph. Reached by
   **edit-agent rewrite** (`edit-agent/apply/route.ts:109`) and **compose-refine**
   (`compose/refine/route.ts:162`). On a questionnaire built from an uploaded document, one "make it
   shorter" wipes every `sourceQuote` and `beforeJson` — permanently disabling revert, with no
   warning and no record that a log existed.
   _Scope note:_ for a **composed** (brief-authored) questionnaire the change log is `[]` by design,
   so this is harmless there. The damage is specific to the extract-then-rewrite path — which is the
   common client workflow.

2. **Experiments compare the live agent against itself.** `AiExperimentVariant.agentVersionId` is
   written at `experiments/route.ts:128` and **read nowhere**. The run route creates every variant's
   eval run against `experiment.agentId` — the live agent (`experiments/[id]/run/route.ts:95`). A
   2-arm version comparison therefore evaluates identical config twice, so the Welch/Cohen "winner"
   from `stats/winner.ts:23` is measuring sampling noise and reporting it as a result.

3. **Turn-eval persistence is silently lossy.** `evaluate-turn/route.ts:132` persists inside a
   nested try/catch; on failure it logs and returns `evaluationId: null`. The admin sees the verdict
   on screen, but it is gone. Contrast the design-eval path, which throws
   (`evaluation-run-routes.ts:316`).

4. ~~**Judge scale mismatch.**~~ **Investigated and dismissed — not a bug.** The audit flagged that
   the judge driver enforces 0–1, `evaluate.ts:35` defaults to 1–5, and `evaluations/gate.ts`
   compares against thresholds, concluding a `gate` downstream of an `evaluate` would misfire. It
   does not. `evaluate` and `judge_call` each carry their **own** `threshold` config and compare it
   against their **own** clamped score on their **own** configured scale — self-consistent.
   `lib/orchestration/evaluations/gate.ts` belongs to the separate dataset-driven batch subsystem:
   it reads `AiEvaluationRun.summary.stats` produced by graders, is scale-agnostic (it just compares
   numbers against admin-set thresholds), and never sees a workflow step's output. The two
   subsystems never meet. No change made.

### Phase 2 — `AppAiRun` (new model + migration)

Columns follow `AppQuestionnaireTurnEvaluation`'s proven shape: polymorphic subject, resolved
provider/model, input + output snapshots, token/cost/duration, a multi-dimensional version stamp,
and a status the UI can render.

Migration generated with `--create-only`, phantom pgvector DDL stripped before applying, applied via
`migrate deploy`, with the 5 vector indexes verified intact afterwards.

### Phase 4 — Cost accounting + version pinning

- **Unify on `logCost`.** `report/`, `cohort-report/`, `scoring/`, `persona/`, `contradiction/`, and
  `tagging/` call `getProvider` directly and never call `logCost`. Report spend is therefore
  invisible to `cost-reports.ts` _and_ to per-agent budget enforcement — which
  `.context/orchestration/report-web-search.md` currently claims caps research spend. That doc is
  wrong until this lands.
- **Add `versionId` to cost metadata.** Four of six authoring call sites omit it, so their cost rows
  can't be joined to the questionnaire they produced.
- **Version-pin `AiEvaluationRun`.** It pins the dataset (`datasetContentHash`) and metrics
  (`metricConfigs`) — both with schema comments explaining why — but has no `agentVersionId`.

### Phase 5 — Retention

`lib/orchestration/retention.ts` exports exactly seven prune functions, **none** touching any `App*`
model. `AppQuestionnaireTurnEvaluation` is one row per turn, unbounded, forever.

- Prune `AppQuestionnaireTurnEvaluation`, `AppQuestionnaireEvaluationRun`, `AppAiRun` via the app
  seam (not by editing the platform file — see `CUSTOMIZATION.md`).
- **Ordering hazard:** `pruneCostLogs` is independent of `pruneExecutions`, so a retained execution
  can lose its cost rows while `totalCostUsd` survives — cost drill-down silently empties.
- Two schema comments promise pruning that doesn't exist: `AiWorkflowExecutionLeaseEvent`
  (`orchestration-workflows.prisma:342`) and `AiWorkflowStepDispatch`.

## What shipped

- **Provenance no longer destroyed.** `replaceVersionStructure` supersedes the extraction change
  log instead of deleting it: rows flip to `status: 'superseded'` with a `supersededAt` stamp,
  stay queryable, and render badged + dimmed + non-revertable in the review table. New status
  threaded through the tuple (`schemas.ts`), the read view + counts (`views.ts`,
  `extraction-review-routes.ts`), and the revert route's 409, which now distinguishes "already
  reverted" from "superseded by a rewrite". Re-ingest still deletes, deliberately.
- **Experiments can't fake a comparison.** `AiEvaluationRun.agentVersionId` records the pin; the
  run route copies each variant's pin onto its run and **refuses** a dataset-driven experiment
  whose variants all resolve to the same config; the worker **fails a pinned run loudly**
  (`agent_version_pinning_unsupported`) rather than scoring the live agent under a pinned label.
  The real fix needs a platform seam — raised as **UG-12**, carried patch logged in
  [[development-plan#Carried Sunrise patches]].
- **`AppAiRun`** — new polymorphic run table + `recordAiRun` seam
  (`lib/app/questionnaire/ai-run/{types,store}.ts`). Capped snapshots (`truncateSnapshot`,
  20k chars), never-throws capture, resolved provider/model, multi-dimensional version stamp.
- **Five surfaces now captured** — fidelity critic (with `repairOutcome` distinguishing
  `repaired` / `repair_failed` / `skipped_systemic` / `verifier_unavailable`), Config Advisor, edit-agent precise
  **and** rewrite, learning digests. The digest's wholesale replace stays (k-anonymity requires
  it); history now lives in `AppAiRun` instead of nowhere.
- **Cohort method record** — `methodRecord Json?` on `AppCohortReport` + `AppCohortReportRevision`,
  mirroring the respondent side from F14.14.
- **Cost attribution unified** — `logAppLlmCost` wrapper requires `versionId`, so app spend is
  joinable to the artifact and finally reaches budget enforcement. The web-search doc's cap claim
  is corrected with a note explaining it was previously unfirable.
- **App retention** — `lib/app/questionnaire/retention.ts`, registered as the `appRetention`
  maintenance task, prunes turn evaluations, design-eval runs, and `AppAiRun` on
  `evaluationRetentionDays`. Kept out of the platform module so it survives upstream syncs. The
  `costLogRetentionDays >= executionRetentionDays` hazard is now documented in both the module and
  `.context/orchestration/retention.md`.
- **Turn-eval loss made visible** — both evaluate routes return `persistError`; the inspector
  drawer and ref-lookup panel render it. The best-effort posture is kept (the verdict is expensive
  and the admin is waiting); only the silence is gone.
- **Migration** `20260718204616_app_ai_run_provenance` — additive only; the five phantom pgvector
  `DROP INDEX` statements and the `searchVector DROP DEFAULT` were stripped by hand (a header
  comment in the file says so), applied via `migrate deploy`, all five vector indexes verified
  intact afterwards.

## Design notes

- **Why polymorphic rather than a run FK per subject.** The six surfaces span three subject types
  (version, session, report) and two tiers. A nullable FK per subject would be six columns, five
  always null. `subjectType` + `subjectId` + a compound index is the same query cost with none of
  the schema churn — and it matches how `AppQuestionnaireTurnEvaluation` already denormalises
  `questionnaireVersionId`.
- **Why not reuse `AiCostLog`.** It has only three FK columns (`agentId`, `conversationId`,
  `workflowExecutionId`) and no input/output snapshot. It answers "what did we spend", not "what did
  the model see and say". Both are needed; they are different tables.
- **`inspectorCalls` is not superseded.** It stays the per-turn deep-dive record for the respondent
  path. `AppAiRun` covers the authoring/report paths that run _outside_ the turn pipeline — which is
  also why `AppQuestionnaireError`'s schema comment, which assumes `inspectorCalls` exists for
  deep-dive diagnosis, is currently unsatisfiable for reports.

## Resolved during build

- **Snapshot shape** — interpolated prompt, capped at `AI_RUN_SNAPSHOT_MAX_CHARS` (20k) with a
  `truncated` flag. The debugging question is always "what did the model actually see".
- **`AppAiRun` retention window** — reuses `evaluationRetentionDays` rather than adding a fourth
  operator knob. Same class of data, and the platform module already warns about keeping these
  windows coherent; a knob nobody can reason about is how they drift apart.
- **`recordAiRun` lives in `lib/`, not the API tier** — the learning-digest builder is itself a
  `lib/` module, so putting the store under `app/api/**` would have inverted the dependency. It
  imports `prisma` from `@/lib/db/client` but NOT `@prisma/client` directly (ESLint forbids the
  latter under `lib/app/**`), so absent Json values are written by omitting the field rather than
  via `Prisma.DbNull`.

## Deliberately not done

- **Executing a pinned agent version.** Needs a config-override seam in the platform chat handler
  (`loadAgent` resolves by slug only). Raised as UG-12; forking a ~2,000-line upstream file with
  ~20 `agentSlug` references would guarantee sync conflicts.
- **Enforcing the retention-window orderings in code.** Both couplings
  (`evaluationRetentionDays <= executionRetentionDays`, `costLogRetentionDays >=
executionRetentionDays`) remain documented guidance, matching the platform's existing posture.
  Enforcing them means validating in the settings Zod schema — a platform change worth doing
  upstream, not forking here.
- **Backfill.** No historical run is reconstructed. `methodRecord` and `AppAiRun` are null/absent
  for anything generated before this shipped, and the read paths treat that as "don't offer the
  explanation" rather than inventing a run nobody observed.

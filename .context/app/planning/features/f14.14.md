# F14.14 — Report method record

> An optional transparency panel that explains, in plain English, how a respondent's report was
> produced — what was read, retrieved, searched, and checked — so they can see the process was
> substantive rather than take it on trust. The same record, in richer form, is always available to
> admins in the Sessions drawer.

## Intent

Respondents receive a generated report with no visibility into how it was made. The ask was a link
(modal) explaining roughly how it was generated — the agentic steps taken, the information
considered, any RAG material accessed — without jargon, and reassuring about rigour.

## The central problem

An explanation of a process is precisely the kind of prose an LLM writes fluently from nothing, and a
fabricated reassurance ("we cross-checked 12 clinical sources") is strictly worse than no explanation
at all: it converts an honest gap into a false claim about the system's own diligence. Compounding
this, the pipeline **discarded almost everything an explanation would need** — retrieved KB chunks
were flattened to prose with their document ids dropped, `runReportResearch` tracked `searchesUsed`
purely as a budget counter and never returned the queries, and no stage recorded whether it ran.

So the bulk of the work was instrumentation, not rendering.

## Decisions (confirmed with the user)

- **Meta-agent + deterministic facts** (over deterministic-only or agent-prose-only). The agent writes
  the narrative; the verifiable counts and source links render deterministically beneath it.
- **Same record, richer admin view** — respondent sees summary + counts + sources; admin additionally
  sees model, timings, cost, search queries, contributing documents, and every stage with skip reasons.
- **Hide the link when no record** — no backfill for reports predating the feature.
- **Default off** — existing questionnaires disclose nothing new until an admin opts in.

## What shipped

Four-layer honesty design (`observe → narrate → check → fall back`):

1. **Observe** — `MethodRecorder` threaded through the generation core; every field observed, never
   inferred.
2. **Narrate** — `app-report-method-explainer` (chat tier) sees _only_ a counts-and-flags digest —
   never the report, answers, source URLs, or search queries.
3. **Check** — `rejectSummary` refuses any number absent from the record, any URL or `[n]` marker, or
   over-length output. It admits observed values only; it will not accept arithmetic derived from them.
4. **Fall back** — any rejection, error, or unseeded agent renders the deterministic template. A report
   never fails because its explanation did.

Capture is unconditional (free); only the agent narration is gated on the setting. A record stored with
a null summary still renders via the template at read time — so enabling the panel later yields
truthful explanations for older reports with **no backfill and no regeneration**.

## Data model

`AppRespondentReport.methodRecord` and `AppRespondentReportRevision.methodRecord` — nullable `Json`,
migration `20260718161504_app_report_method_record`. Per-revision because a re-run may take different
steps; **promote carries the record across** (clearing it when the revision has none) so the panel can
never describe a run that produced different prose.

## Two bugs caught during build

- **`countListedQuestions` over-counted** — the unanswered block contains `## Section` headings and a
  trailing "(and N further…)" line, so counting non-empty lines inflated the gap count. Now counts
  only `- ` rows. Caught by an integration test.
- **Preview runs were described as real** — live testing showed the agent reading a preview record
  (whose digest flags it) and writing _"based on all of your answers to the 12 questions you
  completed"_. Instruction-following is not a guarantee, so a preview record now never reaches the
  agent at all. Regression-tested.

## Out of scope

- The method record for the **Cohort Report** (the modules are report-kind-agnostic, but nothing is
  wired).
- Surfacing the panel in the **PDF export** — on-screen only for now.
- Claim-level provenance (which finding supports which sentence). The record is run-level.

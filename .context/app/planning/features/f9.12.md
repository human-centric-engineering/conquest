# F9.12 ‚Äî Persist turn evaluations

> Committable tracker for **F9.12**. Turns the F9.11 evaluator from ephemeral to durable: every
> verdict is persisted with a snapshot of the input it judged, can carry a human comment, can be
> flagged through a learning workflow, can be actioned into an eval dataset, and is searchable from a
> new admin surface. Before this, a verdict lived only in the drawer's component state and was lost on
> close.

## Intent

Make turn evaluations a durable, reviewable signal: store each one, let a human comment on it and flag
it for learning, feed flagged ones into the existing dataset-driven eval/learning infra, and give
admins a way to surface, view, and search the stored verdicts ‚Äî recording the versions (model,
rubric, questionnaire, app) that produced each score.

## Decisions (confirmed with the user)

- **Learning flag = status workflow**, not a boolean: `none ‚Üí flagged ‚Üí reviewed ‚Üí actioned | dismissed`,
  with reviewer + timestamp.
- **Snapshot the input, not just the verdict**: inspector data is otherwise live-only, so a stored
  score is uninterpretable (and unusable as a learning case) without the call traces + context it judged.
- **Wire into datasets now**: actioning a flagged verdict appends it to an eval dataset (the existing
  `appendCasesToDataset` seam), not a standalone flag.
- **Record versions multi-dimensionally**: evaluator model/provider, rubric version, questionnaire
  version, app version ‚Äî a score is meaningless without the rubric (and structure) that produced it.
- **Keep history**: no unique on `turnId`; re-runs and different evaluator models accumulate.

## What shipped

**Schema (phase 1)** ‚Äî `AppQuestionnaireTurnEvaluation` (`prisma/schema/app-questionnaire.prisma`):
verdict + `evaluatedInput` JSON snapshot, denormalised `overallScore`/`effectiveness`, evaluator
provenance, `rubricVersion`/`questionnaireVersionId`/`appVersion`, comment + flag-workflow columns,
`datasetId`/`datasetCaseId`. Session FK cascades; all other refs plain String (UG-1). Indexes on
session, `(questionnaireVersionId, createdAt)`, `(flagStatus, createdAt)`, `overallScore`.
`TURN_RUBRIC_VERSION` added to `prompt.ts`. Migration authored `--create-only` + stripped of phantom
pgvector DDL per the app-questionnaire playbook.

**Persist on evaluate (phase 2)** ‚Äî `_lib/turn-evaluation-store.ts` `persistTurnEvaluation`; the
evaluate-turn route writes the row (best-effort, logs + null id on failure) and returns `evaluationId`.
Derives the 1-based ordinal from the inspector's 0-based `turnIndex`, best-effort back-links `turnId`,
rounds the score, stamps versions + admin id.

**Review API (phase 3)** ‚Äî `PATCH ‚Ä¶/evaluations/:evalId`: comment + flag transitions among
`none|flagged|reviewed|dismissed`, reviewer/timestamp stamped per touched facet. `actioned` excluded;
the store refuses to re-flag an actioned row (409). `updateTurnEvaluationReview` with a discriminated
`not_found | locked | ok` result.

**Learning-action API (phase 4)** ‚Äî `POST ‚Ä¶/evaluations/:evalId/action-learning` `{ datasetId }`:
`actionTurnEvaluationForLearning` appends a learning case via `appendCasesToDataset` (input =
respondent message, expectedOutput = judged interviewer reply, metadata = score/effectiveness/
rubric/version/comment + `source: 'flagged_turn'`), records `datasetId`/`datasetCaseId`, flips to
`actioned`. The only path to `actioned`. Reasons ‚Üí 404/409/404(dataset)/422(full)/422(no_content).
No platform edit ‚Äî provenance rides in case metadata (the dataset `source` enum is platform-owned).

**Search surface (phase 5)** ‚Äî `GET ‚Ä¶/turn-evaluations` (filter flagStatus/effectiveness/version/
model/score-range/date-range, sort createdAt|overallScore, paginated) + `GET ‚Ä¶/turn-evaluations/:id`
(full verdict + snapshot + review state). Read model `_lib/turn-evaluation-list.ts` enriches each page
with questionnaire title + version in a fixed query budget (manual batched join ‚Äî version id is UG-1).
Admin page `/admin/questionnaires/turn-evaluations` + client table (filters, pagination, slide-over
detail). Shared `TurnEvaluationVerdict` + `TurnEvaluationReview` components. Nav entry in the
fork-owned `lib/app/admin-nav.ts`. View types in `views.ts`; endpoints in the registry.

**Inspector reuse (phase 6)** ‚Äî the drawer consumes the `evaluationId`, renders the shared verdict
component, and shows the review controls inline once a verdict has persisted. ~125 lines of duplicated
verdict rendering removed from the drawer.

## Gotchas

- **Rubric version is load-bearing**: bump `TURN_RUBRIC_VERSION` whenever `prompt.ts`'s rubric changes
  in a way that could move scores ‚Äî otherwise incomparable verdicts mix under one label.
- **`actioned` is store-owned**: never settable via the review PATCH; it must be backed by a dataset
  case, so the action route is the only producer.
- **Learning case needs a respondent message** in the snapshot context. The inspector drawer derives
  the turn's conversation context (respondent message + interviewer reply + recent history) from the
  live message list ‚Äî by **walking** to the matching messages, not `index*2` math, so a leading agent
  greeting can't misalign it ‚Äî and POSTs it with the dump, so the snapshot carries what the action
  needs. A verdict genuinely lacking a respondent message still actions cleanly as `no_content` (422).
- **Migration**: authored `--create-only` and stripped of phantom pgvector `DROP INDEX` /
  `searchVector DROP DEFAULT` lines before applying (plain `migrate dev` would drop platform indexes).

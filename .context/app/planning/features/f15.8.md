# F15.8 — Experience-wide synthesis

> **Everything still open across P15 is consolidated in [`f15-followups.md`](./f15-followups.md).**

> Closes follow-up item 5. Reports existed per step and per breakout, but nothing answered the
> question the journey as a whole poses: what held throughout, and where did it disagree with
> itself?

## The trap this phase exists to avoid

The obvious implementation — widen `ReportScope` with an `experience` kind and aggregate sessions
across steps — **fails silently**. `buildCohortDataset` resolves questions, data slots and scoring
by a single `versionId`, and `buildDataSlots` joins fills by `dataSlotId`, the row id rather than
the key. A fill from another version finds no bucket and is dropped with no error and no warning.
An experience spans versions by definition, so that version would emit a confident, well-formatted
report over a fraction of the data — the worst possible failure mode for an analytics feature,
because nothing about the output looks wrong.

So: **prose in, prose out.** The synthesis reads FINISHED per-step outputs and never sessions. A
static import guard in `synthesis.test.ts` asserts the module never reaches for the dataset builder
or the session tables, because this is the one mistake no runtime test would catch.

## Confirmed decisions

- **Its own table, not a fourth `ReportScope` kind.** `AppCohortReport.versionId` is NOT NULL on
  every row — each of its three scopes analyses exactly one version — and `scopeOwnerCreate` /
  `scopeOwnerWhere` are deliberately exhaustive switches. An experience has no honest value to put
  in `versionId`. Forcing it in would mean relaxing a NOT NULL that currently documents a real
  invariant.

- **The two kinds read different inputs**, because they produce different outputs. A switcher has
  per-step cohort reports plus a routing distribution; a facilitated meeting has breakout insights
  and usually no step reports at all. A single uniform input would have meant ignoring a meeting's
  actual output or forcing meetings to generate step reports they have no use for.

- **Divergence is a first-class section**, not a paragraph in the narrative. It is the one thing no
  single step report can contain. Without it this feature is concatenated step reports under a new
  heading, and there would be little reason to build it.

- **Coverage and citations are server-computed.** A model asked to describe its own inputs
  confabulates a tidy answer, and coverage is exactly what a reader uses to judge how far to trust
  everything else. Citations are verified against the steps that actually contributed; unknown keys
  are dropped. A hallucinated citation is worse than none — it sends a reader to check a source that
  never said it, and makes an unsupported claim look sourced.

- **The prompt does not mention that citations are checked.** A prompt is not where this is
  enforced, and saying so would only teach an injected instruction what it has to forge. Same
  reasoning as the breakout synthesiser's `supportedBy`.

- **Anonymity carries through by construction.** Meeting insights are re-gated on READ at the
  current `insightMinSupport`, so raising the threshold after a meeting immediately narrows what a
  later synthesis can see. Anything below the floor never enters the material, so the writer cannot
  surface it, paraphrase it, or fold it into a finding.

- **Partial by design.** Missing or unfinished step reports are recorded in coverage and skipped.
  Blocking would let one never-generated step hold the feature hostage; generating with nothing
  ready returns 409 `NOTHING_TO_SYNTHESISE` **with the coverage attached**, so the panel names the
  missing steps rather than saying "something went wrong".

- **No revision chain**, unlike `AppCohortReport`. A synthesis reads a moving target — its inputs
  are regenerated and edited independently — so a history would imply a stability it does not have.
  A failed regeneration preserves the previous content: retrying must never be worse than not
  trying.

- **Synchronous, not queued.** Every input is already-generated prose, so this is one LLM call over
  loaded text. There is no cohort worker to hook into, and standing one up would add a lease, a
  claim loop and orphan recovery to buy nothing an admin waiting on a button press would notice.

- **No separate enable gate.** Every input has already passed its own opt-in — a step report exists
  only because reporting was enabled for that version; an insight exists only because a meeting ran
  and cleared the support floor. A second switch would let an admin disable a view of data they have
  already consented to producing, which protects nobody.

## What shipped

- `AppExperienceSynthesis` (migration `20260720190000_add_experience_synthesis`) — one row per
  experience, `content` Json, stored `coveredSteps` / `eligibleSteps`.
- `lib/app/questionnaire/experiences/synthesis/` — `types.ts` (content contract + validator),
  `material.ts` (per-kind input assembly), `generate.ts` (agent call + citation verification),
  `persist.ts`.
- `app-experience-synthesiser` agent (seed `083`), reasoning tier, prompt built in code.
- `GET|POST /api/v1/app/experiences/:id/synthesis[/generate]`, sharing the per-admin
  `cohortReportGenerateLimiter` sub-cap.
- `ExperienceSynthesisPanel` above the per-step tabs on the Reports tab.
- `experience-synthesis` Behind-the-Scenes diagram (the Experiences category is now four).
- 31 unit tests, weighted toward citation verification and the re-aggregation guard.

## Verified end to end

Against the dev database with two deliberately CONTRADICTORY synthetic step reports (intake: "cost
is the barrier"; finance: "cost was not the issue, time was") and a third step with no report:

- the divergence was correctly identified and cited to both steps;
- coverage read 2 of 3, with the third marked `no_report` — server-derived, not model-claimed;
- caveats independently flagged the 12-vs-40 sample asymmetry and that a branch is not
  representative of intake;
- 409 `NOTHING_TO_SYNTHESISE` returned for an experience with nothing ready;
- ~7.7s, ~$0.005 per synthesis.

**Not verified:** the panel's client-side render was not exercised in a real browser (no browser
tooling available in that session). The API it consumes is verified; the component is not.

## Deliberately not built

- **Cross-run tracking.** The synthesis compares steps, not the same respondent across steps. Doing
  the latter properly is a different feature with its own identity and consent questions.
- **PDF export.** The step reports have one; this does not yet.
- **Per-section AI assist / editing.** The synthesis is regenerate-only. Follow-up item 6 covers the
  equivalent for step reports and should probably be settled there first.

# F4.6 — Session state machine

> Committable tracker for **F4.6**, the closing feature of P4 (Conversational engine,
> non-streaming). Gives `AppQuestionnaireSession` a lifecycle — `active | paused |
completed | abandoned` — with a deterministic transition machine, an append-only audit
> trail (`AppQuestionnaireSessionEvent`, one row per transition), resume logic for paused
> sessions, and a cost-cap event hook wired for F6.3/F6.5 to fire. Gated by
> `APP_QUESTIONNAIRES_ENABLED` only — **no LLM, no sub-flag**.

## Intent

F4.1–F4.5 shipped which-question / what-answer / do-they-conflict / update-an-answer /
done-enough. F4.6 makes the session itself a first-class state machine: every lifecycle
move is legal-by-construction and leaves an audit row, a paused session can be resumed,
and the budget-exhaustion event is wired (but not fired) for the streaming engine. With
this, P4's whole per-turn behaviour set is exercisable by hand (Vitest) before the
streaming surface exists (F6.1).

## Decisions (confirmed with the user at planning)

- **Deterministic, no LLM.** Unlike F4.1–F4.5, F4.6 has no capability / agent / sub-flag /
  prompt / schema. The whole feature is a pure transition table + the seam writes. Gated
  by the master flag only.
- **Vocab: keep `active`, add `paused` (Option B).** The plan writes `in_progress`; the
  code already shipped `active` (F4.4). Rather than rename (a backfill migration + churn
  across the F4.5 seam + tests), `active` stays the in-progress synonym and `paused` is
  added — an additive `SESSION_STATUSES` tuple edit, no data migration (`status` is a plain
  String column, `@default("active")` unchanged).
- **One thin transition route, API-first.** The DoP only requires Vitest, but the repo is
  API-first, so F4.6 ships one admin route (`pause | resume | abandon`). **Completion is
  not an action here** — accept→submit stays the single entrypoint on the F4.5 `/complete`
  route. An events-read route is deferred to P8 (no consumer yet).
- **The route excludes the preview session (`isPreview: false`).** The F4.4/F4.5 preview
  singleton is reused across the refine/complete routes and must stay `active`/`completed`;
  pausing or abandoning it via `/transition` would brick that version's `/complete` submit
  (an abandoned session is terminal; completing a non-active session is illegal → an
  uncaught throw). The lifecycle machine is for real respondent sessions (F6.1), so the
  route 404s every session that exists today and the machine is exercised by hand (Vitest)
  through the seam.
- **Minimal resume state.** `loadSessionResumeState` returns `{ status, answeredSlots }`.
  Coverage / next-question stay in the F4.1/F4.5 context builders the caller already uses;
  per-turn history is F6.1's. Anything richer would duplicate the future engine.
- **Status + event written atomically.** `transitionSession` does the status update AND the
  event insert in one `$transaction`, so a status can never change without its audit row.
- **`markSessionCompleted` generalised, route untouched.** It moves from the F4.4
  `_lib/answer-slots.ts` seam to the new `_lib/sessions.ts`, now routing through
  `transitionSession` (so the F4.5 accept→submit path writes a `completed` event). It's
  re-exported from `answer-slots.ts`, so the `/complete` route's import is unchanged.
- **`cost_cap_reached` wired, not fired.** `recordCostCapReached` writes a non-transition
  event; the turn-boundary fire is F6.3/F6.5. `created` is likewise reserved for real
  respondent sessions (F6.1).
- **New model, no drift probe.** `AppQuestionnaireSessionEvent` is fully Prisma-modelled
  (no raw-SQL object), so unlike the F4.5 partial unique index it needs no drift probe.
  Migration generated `--create-only` and hand-stripped of the phantom pgvector DROPs.
- **No CHANGELOG entry.** The questionnaire models/routes are ConQuest app surface, not the
  Sunrise platform public surface the CHANGELOG/VERSIONING contract tracks (F4.1–F4.5 added
  none either).

## Build shape — 2 PRs (one branch, risk-sequenced)

`answer-slot-persistence.test.ts`. Docs: `session-state-machine.md` + `schema.md` model

## Tests

- **Unit** (`tests/unit/lib/app/questionnaire/session/session-logic.test.ts`): the **full
  4×4 transition matrix** (apply / noop / illegal per the DoP's 100% edge coverage),
  `canTransition`, `isTerminal` (terminal has no outgoing apply edge), `assertTransition`
  (throws on every illegal edge with from/to; the canonical `paused → completed`),
  `eventTypeFor` (resume → `resumed`, else target status; all in `SESSION_EVENT_TYPES`),
  vocab parity.
- **Unit — schema** (`tests/unit/prisma/app-questionnaire-schema.test.ts`, +): the
  `AppQuestionnaireSessionEvent` DMMF shape + the session `events` reverse relation; the
  migration-SQL block (single CREATE TABLE, session FK `ON DELETE CASCADE`, both indexes, no
  leaked platform DDL).
- **Integration — seam** (`session-state-machine.test.ts`): `transitionSession` apply
  (status + exactly one event, in the transaction; reason/metadata passthrough), noop (no
  write), illegal (throws before any write), not-found; the wrappers map to the right
  event; `recordCostCapReached` (non-transition event, no `$transaction`);
  `loadSessionResumeState` (shape + boundary narrowing + not-found).
- **Integration — route** (`session-transition-routes.test.ts`): gate order (flag → auth →
  validation → scope), invalid action 400, scope 404, the action dispatch + resume payload,
  reason passthrough, illegal → 409.

## Sequencing

- **PR2** — seam refactor (moving `markSessionCompleted`) + route + integration + docs. The

## Operator note

Session lifecycle is on under `APP_QUESTIONNAIRES_ENABLED` — no sub-flag. The transition
route (`POST …/sessions/:sessionId/transition`, admin-only) drives `pause | resume |
abandon`; completion stays on the F4.5 `/complete` route. Every legal transition writes an
`AppQuestionnaireSessionEvent` row; an illegal one is a 409, an unknown session a 404. The
route's scope **excludes the preview session** (`isPreview: false`) — pausing/abandoning the
shared F4.4/F4.5 preview singleton would brick that version's `/complete` submit, so the
route is for real respondent sessions only (F6.1) and 404s every session that exists today;
the machine is exercised by hand (Vitest) through the seam. The `cost_cap_reached` hook
exists but is fired only at the turn boundary in F6.3/F6.5.

## Not in F4.6

The `AppQuestionnaireTurn` table + `lastUpdatedTurnId` population (F6.1); real respondent
binding / `respondentUserId` (P6/P7); the streaming surface + the live per-turn loop that
consumes this machine (F6.1); **firing** the cost-cap event (F6.3/F6.5); writing the
`created` event for real sessions (F6.1); any session-lifecycle UI (P7); an events-read /
audit-trail endpoint (P8).

# F4.1 — Selection strategies

> Committable tracker for **F4.1**, the **opening feature of P4 (Conversational
> engine, non-streaming)**. Four pluggable strategies that decide which question to
> ask next — `sequential`, `random`, `weighted`, `adaptive` — built as pure
> functions exercisable by Vitest before any streaming surface exists. API-first
> via a preview route; gated by `APP_QUESTIONNAIRES_ENABLED` (404 when off).

## Intent

F3.1 shipped the `selectionStrategy` config field as a seam with no consumer.
F4.1 is the consumer: the per-turn "which question next?" intelligence. P6 wraps
the streaming layer around it later; F4.1 proves each behaviour with hand-driven
state, which is the P4 definition of done.

## Decisions (at planning / build — confirmed)

- **Four strategies; `random` added to the enum.** The shipped enum was
  `[sequential, weighted, adaptive]`; the plan prose listed `random` too, so it was
  added → `[sequential, random, weighted, adaptive]`. `random` is selectable in the
  config picker and seeded on `sessionId + round` for crash-replay idempotency (no
  `Math.random`).
- **Pure strategies over an in-memory `SelectionContext`.** Session/turn/answer
  tables don't exist yet (F4.6/P6), so a strategy never touches Prisma — a caller
  (test harness now, engine later) assembles the context. Keeps every per-turn
  behaviour exhaustively unit-testable. Registry mirrors the grader registry.
- **Required-before-optional precedence** for `random`/`weighted` (mandatory
  questions exhausted first); `sequential` keeps strict document order.
- **Weighted constants live in code, not config** (`UNDERCOVERED_SECTION_BONUS`,
  `LOW_CONFIDENCE_MULT`, `LOW_CONFIDENCE_THRESHOLD`) — tuned like the
  cost-estimation token constants; per-questionnaire control is a future PR.
- **Adaptive is the full LLM + pgvector flavour, fail-soft.** Embeds the last
  message → pgvector top-K → seeded selection agent picks via `drainStreamChat`.
  Every failure (no deps, no embeddings, LLM/budget error, off-pool/unparseable
  pick) degrades to `weighted`, never throws.
- **Adaptive gated behind its own sub-flag** (`APP_QUESTIONNAIRES_ADAPTIVE_STRATEGY_ENABLED`,
  disabled by default) on top of the master app flag — it spends per turn. Off →
  hidden from the picker and degraded to `weighted` at run time.
- **Impure surface injected, not imported.** `adaptive` takes `StrategyDeps`
  (`embedText`, `rankByVector`, `llmPick`) so `lib/` stays Prisma-free and mockable;
  the real deps are wired at the route seam.
- **Embedding column mirrors the platform.** `AppQuestionSlot.embedding
vector(1536)` — `Unsupported(...)` in Prisma, raw-SQL read/write, HNSW index
  added by raw SQL (the create-only + strip migration discipline; the generated
  diff's phantom knowledge-index DROPs were stripped).

## Build shape

### 1. Pure core (`lib/app/questionnaire/selection/`, Prisma-free)

`types.ts` (`SelectionContext`, `SelectionDecision`, `SelectionStrategyPlugin`,
`StrategyDeps`, weighted constants), `context.ts` (`unanswered`, `coverageRatio`,
`requiredFirstPool`, `terminalDecision`), `registry.ts` (slug → plugin + reset +
`KNOWN_STRATEGY_SLUGS`), `strategies/{sequential,random,weighted,adaptive}.ts`,
`index.ts` barrel (auto-registers). `random` added to `SELECTION_STRATEGIES` in
`../types.ts` (+ editor label).

### 2. Preview route + context builder (PR2)

`_lib/selection-context.ts` maps a version's sections/slots/tags/config →
`SelectionContext` (answered state supplied by the caller, keyed on slot `key`).
`POST …/versions/:vid/next-question` (`withAdminAuth`, Zod body, no persistence) →
`{ strategy, decision, question? }`.

### 3. Adaptive infra (PR3)

Migration `20260604181453_app_question_slot_embedding` (column + HNSW index).
`_lib/slot-embeddings.ts` (`embedVersionSlots` via `embedBatch`; `rankSlotsByVector`
pgvector `<=>`). `_lib/adaptive-deps.ts` (`buildAdaptiveDeps` → `embedText` +
pgvector rank + seeded `app-questionnaire-selector` agent via `drainStreamChat`).
`POST …/versions/:vid/embed-questions` backfill route. Sub-flag helper
`isAdaptiveSelectionEnabled()`; seeds `004-adaptive-selection-flag` +
`005-selection-agent`. Per-flow sub-caps (`adaptiveSelectionLimiter`,
`embedSlotsLimiter`). Config editor hides `adaptive` when the sub-flag is off
(threaded page → version-editor → config-editor).

## Tests

- **Unit** (`tests/unit/lib/app/questionnaire/selection/`): registry parity;
  `context` helpers (coverage math, completion boundaries, tie-break, required-first);
  `sequential`/`random`/`weighted`/`adaptive` edge cases — no remaining questions,
  cap/threshold completion, random determinism + idempotency, weighted scoring math
  (section + low-confidence bonuses), every adaptive fallback path. ~70 cases.
- **Unit** (`tests/unit/app/api/v1/app/questionnaires/_lib/`): `slot-embeddings`
  (only-missing/force/empty/mismatch + cosine-rank delegation); `adaptive-deps`
  (prompt builder, selector-output parser, `llmPick` resolution + every fail-soft
  branch).
- **Integration** (`tests/integration/api/v1/app/questionnaires/`):
  `next-question-routes` (gate order, validation, each strategy's wiring, adaptive
  flag on/off) and `embed-questions-routes` (gate order, scope-404, counts, force).

## Sequencing — 3 PRs (one branch, risk-sequenced)

- **PR1** — strategy core + deterministic strategies (`sequential`/`random`/`weighted`)
  - interim adaptive + enum/label + unit tests.
- **PR2** — context builder + `next-question` preview route + integration tests.
- **PR3** — embedding migration + slot-embedding backfill + adaptive deps (embeddings
  - pgvector + selector agent) + sub-flag/seeds + rate-limit sub-caps + tests.

Gates each PR: `/pre-pr`, `/code-review`, `/test-review`; `/security-review` on PR2
(new authed input surface) and PR3 (LLM I/O, cost, raw SQL).

## Operator note

`adaptive` needs slot embeddings: run `POST …/versions/:vid/embed-questions` for a
version, then enable `APP_QUESTIONNAIRES_ADAPTIVE_STRATEGY_ENABLED`. Without
embeddings (or with the flag off), adaptive transparently runs as `weighted`.

## Not in F4.1

Session/turn/answer persistence + the streaming surface (F4.6/P6); answer
extraction (F4.2); contradiction detection (F4.3); the offer-to-submit flow (F4.5).
On-write embedding generation (today embeddings come from the explicit backfill
route; generating them on slot edits is a later refinement).

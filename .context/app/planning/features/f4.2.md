# F4.2 — Answer extraction into slots

> Committable tracker for **F4.2**, the answer-capture half of P4 (Conversational
> engine, non-streaming). Given a respondent's message + the active question +
> session context, produce `(value, confidence, provenance, rationale, label)` for
> one or more slots — including **side-effects** (one message answering several
> questions). Built as a pure core + a capability + a no-persistence preview route,
> mirroring F4.1/F1.1. Gated by `APP_QUESTIONNAIRES_ENABLED` + a per-feature
> sub-flag (404 when off).

## Intent

F4.1 shipped "which question next?"; F4.2 is its complement — read the reply and
record what was answered. Selection draws a question, extraction turns the reply
into typed slot values. P6 wraps the streaming layer around both later; F4.2 proves
each behaviour with hand-driven state (the P4 definition of done), persisting
nothing because the session/answer tables are F4.6's.

## Decisions (at planning / build — confirmed)

- **Pure + preview, no persistence.** `AppQuestionnaireSession`/`AppAnswerSlot`
  land in F4.6 (F4.2 deps are only F1.1 + F0.1), so extraction produces a
  version-agnostic `AnswerSlotIntent[]` — the F1.1 `ChangeRecordIntent` analogue —
  that F4.6 persists. A caller assembles the in-memory context (test harness now,
  engine later); the preview route returns intents and writes nothing.
- **Generic value + downstream validation, not a discriminated union.** The LLM
  emits `value: z.unknown()`; `answer-value.ts` validates it against the slot's
  real `typeConfig` (reusing F2.1 `typeConfigSchemaFor`), because choice-membership
  / likert bounds need the slot's runtime config a static schema can't see. Same
  split as ingestion's open `suggestedTypeConfig`.
- **Four-label vocabulary now; extractor emits three.** `ANSWER_PROVENANCES =
[direct, inferred, synthesised, refined]` is the single-source tuple;
  `EXTRACTOR_EMITTED_PROVENANCES` (a `satisfies` subset) drops `refined`, which is
  reserved for the F4.4 refinement flow — F4.4 starts emitting it with no tuple
  edit. The Zod contract derives its enum from the subset.
- **New answer-extractor agent**, distinct from the document extractor (002) and
  the selection agent (005): per-turn work, far higher volume, its own monthly
  budget. Resolves the `chat` tier (snappy), not `reasoning`.
- **Own opt-in sub-flag** (`APP_QUESTIONNAIRES_ANSWER_EXTRACTION_ENABLED`, off by
  default) on top of the master flag — extraction spends an LLM call every turn,
  the same reasoning as adaptive selection's sub-flag.
- **`isActiveQuestion` on the intent** distinguishes the active answer from a
  side-effect (set by comparing `slotKey` to the active key) — one homogeneous
  list for F4.6 to map.
- **Candidacy = unanswered + active slot**, hard `.max()` capped. Smarter
  candidacy (section-proximity / required-only) is a later refinement.
- **Unknown `activeQuestionKey` → 400** (the version exists; the key is
  load-bearing), a deliberate divergence from next-question's drop-unknown-keys
  leniency for `answered`.
- **Fail-soft route**: a capability error → 200 with empty intents + a
  `diagnostic`, never a 5xx, so the engine can continue the conversation.

## Build shape

### 1. Pure core (`lib/app/questionnaire/extraction/`, Prisma-free) — PR1

`types.ts` (`ExtractionContext`, `ExtractionSlotView`, `AnswerSlotIntent`,
`AnswerExtractionResult`), `extraction-schema.ts` (Zod LLM contract +
`z.toJSONSchema` + `validateAnswerExtraction`), `answer-value.ts` (per-type value
validation reusing `typeConfigSchemaFor`), `extraction-prompt.ts` (provider-agnostic
prompt + retry), `answer-intents.ts` (`normalizeAnswerIntents` — drop/downgrade/
dedupe), `index.ts` barrel. `ANSWER_PROVENANCES`/`EXTRACTOR_EMITTED_PROVENANCES`
added to `../types.ts`.

### 2. Capability + seeds (PR2)

`capabilities/extract-answer-slots.ts` (`AppExtractAnswerSlotsCapability`,
`processesPii`, counts-only `redactProvenance`); constants (slug/handler/agent/
function definition); registration in `lib/app/capabilities.ts` + the barrel;
seeds `006-answer-extractor-agent`, `007-answer-extraction-capability`.

### 3. Sub-flag + preview route (PR3)

Constant `APP_QUESTIONNAIRES_ANSWER_EXTRACTION_FLAG` + `isAnswerExtractionEnabled()`;
seed `008-answer-extraction-flag` (disabled); `answerExtractionLimiter` (60/min);
`_lib/extraction-context.ts` (DB seam); `POST …/versions/:vid/extract-answer`
(flag → auth → sub-flag → validate → sub-cap → context → dispatch → fail-soft).

## Tests

- **Unit** (`tests/unit/lib/app/questionnaire/extraction/`): schema round-trip +
  named-path issues; `answer-value` per type (choice membership, likert/numeric
  bounds, out-of-range/wrong-shape failures); prompt builder structure (lists
  types + the 3 emittable labels; retry names paths); normaliser (unknown-slot
  drop, value-fails-type drop, `direct`-without-quote downgrade, dedupe, active-vs-
  side-effect, empty); provenance-vocabulary parity. 50 cases.
- **Integration — capability** (`tests/integration/lib/app/questionnaire/answer-capability.test.ts`):
  the real dispatcher + `runStructuredCompletion` with a mocked provider — happy
  path (active + side-effect), value-fails-type drop, empty extraction, retry,
  no-silent-failure, provider fail-closed paths, cost-log isolation, redaction. 15.
- **Integration — route** (`tests/integration/api/v1/app/questionnaires/extract-answer-routes.test.ts`):
  gate order (master-off before auth; sub-flag-off after auth, no dispatch), auth
  401/403, scope-404, unknown-active-key 400, body validation, rate-limit 429,
  dispatch wiring (candidate pool + agent binding), answered-slot exclusion,
  fail-soft, missing-agent 404. 15.

## Operator note

Answer extraction is off until **both** `APP_QUESTIONNAIRES_ENABLED` and
`APP_QUESTIONNAIRES_ANSWER_EXTRACTION_ENABLED` are on; run `db:seed` so the
answer-extractor agent + capability exist. With the sub-flag off, the
extract-answer route 404s.

## Not in F4.2

Persistence + the session/answer tables (F4.6/P6); contradiction detection (F4.3);
answer refinement + the `refined` provenance transition (F4.4); the offer-to-submit
flow (F4.5); the streaming surface (P6). Smarter side-effect candidacy is a later
refinement.

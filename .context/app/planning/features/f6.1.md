# F6.1 ‚Äî Per-turn orchestrator + streaming

> Committable tracker for **F6.1**, the opening feature of P6. Wraps the P4 engine in a
> streaming conversational surface: a respondent creates a session and drives a streamed
> turn loop (extract ‚Üí detect ‚Üí refine ‚Üí assess ‚Üí respond), the first feature to mint real
> respondent sessions and persist turn records. Gated by `APP_QUESTIONNAIRES_ENABLED` +
> a new `APP_QUESTIONNAIRES_LIVE_SESSIONS_ENABLED` dark-launch sub-flag.

## Intent

P4 shipped the per-turn behaviours (which-question / what-answer / conflict / refine /
done-enough) as admin-preview routes over hand-supplied state. F6.1 makes them live: a
real respondent sends a message ‚Üí a deterministic per-turn orchestrator runs the pipeline
‚Üí the agent reply streams back over SSE ‚Üí the answers + turn are persisted. It's the first
writer of real `AppQuestionnaireSession` rows (`respondentUserId`, the reserved `created`
event) and of `AppQuestionnaireTurn` (firing `AppAnswerSlot.lastUpdatedTurnId`).

## Decisions (confirmed with the user)

- **Reuse the plumbing, not `streamChat`.** Sunrise's `streamChat`/`StreamingChatHandler`
  is an LLM-driven tool loop (one agent, the model picks tools). The F6.1 orchestrator is
  the opposite ‚Äî a **deterministic pipeline** that dispatches the P4 capabilities in a
  fixed order, with the LLM called _inside_ each capability. So the route reuses
  `sseResponse`, the `ChatEvent` shape, attachment/magic-byte validators, and cost
  tracking, but **drives the app-native orchestrator** instead of `streamChat`.
- **Pure core + injected invokers.** `lib/app/questionnaire/orchestrator/` is Prisma-free:
  `runTurn(state, invokers)` decides order/branching/response/events; the impure capability
  calls are injected (real at the route seam, stubbed in tests). Per-step feature flags are
  resolved by the route and passed in, so the core stays sync in its branching.
- **Three respondent-access scenarios** (the user's requirement):
  1. **Invitation-bound** ‚Äî `withAuth`, `respondentUserId` set, identity visible.
  2. **Logged-in anonymous** ‚Äî `withAuth`, `respondentUserId` set, the questionnaire's
     `config.anonymousMode` withholds identity from admins (redaction is a later phase).
  3. **No-login anonymous** ‚Äî a true public surface (pop-up/ad-hoc/demo): `respondentUserId`
     null, access proven by a stateless HMAC session token (no better-auth anonymous plugin
     exists; the token models `approval-tokens.ts`, NOT a bare cuid).
- **Real token streaming for offers.** The F4.5 compose-offer capability returns structured
  JSON, so a streaming variant (`offer-stream.ts`) composes plain prose and streams it off
  `provider.chatStream`. Question/terminal turns stay deterministic (chunked content).
- **Dark-launch sub-flag** `APP_QUESTIONNAIRES_LIVE_SESSIONS_ENABLED` ‚Äî the respondent
  surface stays off until an operator opts in, independent of the admin previews.

## Deferred / follow-ups

- **Attachment input** (extraction over an uploaded image/doc) ‚Äî needs extending the F4.2
  extraction capability to accept content parts; tracked for F6.1 PR8 / a follow-up.
- **Full refinementHistory append** on refine persistence (PR4 persists the corrected value
  via the upsert seam; the history append through `persistRefinement` is a follow-up).
- **Aggregate per-turn token usage** on the `done` event (the capabilities log their own
  `AiCostLog` rows; the turn record carries summed offer cost only).
- **Admin-side anonymous redaction** (`anonId`, redacted session views) ‚Äî a later phase.
- **Cost-cap firing** at the turn boundary ‚Äî F6.3 (`recordCostCapReached` is wired).

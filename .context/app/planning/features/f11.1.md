# F11.1 — Cohorts & Rounds

> Committable tracker for **F11.1**. Introduces cohorts (groups of people under a demo client) and
> rounds (time-bound deliveries of questionnaires to a cohort), plus respondent access enforcement.
> Gated by the `APP_QUESTIONNAIRES_COHORTS` flag (seeded disabled).

## Intent

A demo client could brand questionnaires but had no notion of **who** receives one as an organised
group, nor any way to make a questionnaire **time-bound**. F11.1 adds:

- **Cohort** — a named group (e.g. a team) belonging to a demo client, with a roster of members
  (email + name). A person may belong to multiple cohorts under the client.
- **Round** — a time-bound delivery of one or more **distinct** questionnaires to a cohort
  (`draft → open → closed`, adjustable window, manual admin close).

Cardinal rule: **a round is the only way to make a questionnaire time-bound.** A session without a
`roundId` stays open-ended (today's behaviour, untouched).

Cross-respondent analytics/reports over a round are explicitly **out of scope** here (later stage);
this ships basic per-round/per-member completion counts only.

## What shipped

**Schema** (`prisma/schema/app-questionnaire.prisma`, migration `20260620111014_app_cohorts_rounds`,
generated `--create-only` with the phantom pgvector DROPs stripped):
`AppCohort`, `AppCohortMember` (`@@unique([cohortId, email])`, soft-remove via `status`),
`AppQuestionnaireRound`, `AppQuestionnaireRoundItem` (M:N round↔questionnaire, optional version pin),
plus nullable plain-`String` `roundId`/`cohortMemberId` on `AppQuestionnaireSession` (UG-1 posture —
identity↔answer pointers, never a cascading edge).

**Flag**: `APP_QUESTIONNAIRES_COHORTS_ENABLED` (constant + `isCohortsEnabled`/`ensureCohortsEnabled`/
`withCohortsEnabled` in `feature-flag.ts`; seed `047-cohorts-flag.ts`).

**Domain** (`lib/app/questionnaire/rounds/`): status tuples + view types, Zod schemas (+
`defaultRoundName`), the **pure** access guard `evaluateRoundAccess`, and the admin nav helpers.

**API** (`app/api/v1/app/cohorts/**`, `rounds/**`, admin + flag-gated): cohort CRUD + roster
add/edit/soft-remove; round CRUD + manual close + attach/detach questionnaires. Enriched list stats
via batched `groupBy` (`rounds/_lib/stats.ts`) — fixed query budget, no N+1.

**Grant mechanism — per-member round invitations**: `AppQuestionnaireInvitation` gains `roundId` +
`cohortMemberId` (migration `20260620130555_app_round_invitations`); `POST /api/v1/app/rounds/:id/invitations`
(`rounds/_lib/invites.ts`) mints one server-trusted invitation per active member × bundled
questionnaire-version, reusing the frictionless link. The session-create paths read the round
context off the **invitation** (`roundContextOf`), never the request body — so round membership
can't be forged. Walk-up/public starts are never round-bound.

**Enforcement**: DB wrapper `questionnaire-sessions/_lib/round-access.ts` (`assertRoundAccess`) wired
into the two invitation create paths (deny on bad ref), the per-turn continue gate (auto-pause on
round close/window, 403 on removed member), and the resume path. `buildTurnContext` exposes the new
session columns; `findResumableSession*` is round-scoped.

**Admin UI**: flag-gated **Cohorts** + **Rounds** tabs on the demo-client detail, searchable tables
with drill-down detail pages (roster management, round window + close, questionnaire attach/detach).

**Tests**: pure guard matrix; schema/`defaultRoundName`; stats fold; `assertRoundAccess` wrapper
(window/membership/missing-round/cross-cohort); cohort + round route integration (flag gate, CRUD,
close, duplicate-email/already-closed conflicts); create-path round enforcement.

**Docs**: `.context/app/questionnaire/cohorts.md`; flag added to `feature-flags.md`.

## Deferred

Round-wide cross-respondent **analytics/reports** are a separate later phase (this ships per-round
/ per-member completion counts only). A **shared-link** round grant (anyone with the link, checked
at entry) was considered and not built — round delivery is per-member invitations, which the access
guard needs for per-member removal. Round invitation **email delivery** reuses the existing
invitation surfaces; the generation route returns the minted links for the admin to send/copy.
